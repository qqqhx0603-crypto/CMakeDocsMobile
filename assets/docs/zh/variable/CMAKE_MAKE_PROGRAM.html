CMAKE_MAKE_PROGRAM¶

可启动原生构建系统的工具。其值可以是可执行文件的完整路径，如果该工具预期位于 PATH 中，也可以仅是工具名称。

所选工具取决于用于配置项目的 CMAKE_GENERATOR。

  • Makefile（构建脚本） 生成器 将此变量设置为 make、gmake 或特定生成器的工具（例如，针对 NMake Makefile（构建脚本）s 使用 nmake）。

    这些生成器会将 CMAKE_MAKE_PROGRAM 存储在 CMake 缓存中，以便用户进行编辑。

  • Ninja（构建工具） 生成器将其设置为 ninja。

    该生成器会将 CMAKE_MAKE_PROGRAM 存储在 CMake 缓存中，以便用户进行编辑。

  • Xcode 生成器将其设置为 xcodebuild。

    该生成器倾向于在构建时查找构建工具，而不是提前将 CMAKE_MAKE_PROGRAM 存储在 CMake 缓存中。这是因为 xcodebuild 很容易找到。

    为了与 3.2 之前的 CMake 版本保持兼容，如果用户或项目明确将 CMAKE_MAKE_PROGRAM 添加到 CMake 缓存中，则 CMake 将使用指定的值。

  • Visual Studio（集成开发环境） 生成器 将此变量设置为 MSBuild.exe 或 devenv.com 的完整路径。（另请参阅变量 CMAKE_VS_MSBUILD_COMMAND 和 CMAKE_VS_DEVENV_COMMAND）。

    这些生成器倾向于在构建时查找构建工具，而不是提前将 CMAKE_MAKE_PROGRAM 存储在 CMake 缓存中。这是因为这些工具是特定于版本的，可以使用 Visual Studio（集成开发环境） 安装程序进行定位。此外，这也是必要的，因为正确的构建工具可能取决于项目内容（例如，Visual Studio（集成开发环境） 的 Intel Fortran 插件需要 devenv.com 来构建其 .vfproj 项目文件，尽管通常为了支持 CMAKE_GENERATOR_TOOLSET，更推荐使用 MSBuild.exe）。

    为了与 3.0 之前的 CMake 版本保持兼容，如果用户或项目明确将 CMAKE_MAKE_PROGRAM 添加到 CMake 缓存中，则 CMake 会在可能的情况下使用指定的值。

  • Green Hills MULTI 生成器根据所使用的工具集，将其设置为 gbuild.exe(Windows（微软桌面系统）) 或 gbuild(Linux) 的完整路径。

    一旦生成器为此变量初始化了一个特定值，更改该值将产生未定义的行为。

CMAKE_MAKE_PROGRAM 变量是为项目代码使用而设置的。该值也被 cmake --build 和 ctest --build-and-test 工具用于启动原生构建过程。
