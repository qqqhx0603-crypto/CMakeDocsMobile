FindCUDA¶

3.27 版本变更：仅当策略 CMP0146 未设置为 NEW 时，此模块才可用。请将项目迁移至 CMake 原生支持的 CUDA 语言。

3.10 版本弃用：请勿在新代码中使用此模块。

编译 CUDA 代码不再需要使用此模块或调用 find_package(CUDA)。相反，应在顶层 project() 命令中列出的语言中加入 CUDA，或者调用带有 CUDA 参数的 enable_language() 命令。之后，就可以像处理其他语言一样，直接将 CUDA (.cu) 源文件添加到目标中。

3.17 版本新增：若要手动查找并使用 CUDA 工具包库，请改用 FindCUDAToolkit 模块。无论是否启用 CUDA 语言，它均可正常工作。

弃用用法文档¶

用于构建 CUDA C 文件的工具：库和构建依赖项。

该脚本用于定位 NVIDIA CUDA C 工具。它应适用于 Linux、Windows（微软桌面系统） 和 macOS（苹果桌面系统），并应与 CUDA C 发布版本保持合理同步。

3.19 版本新增：支持 QNX。

此脚本利用 find_package() 的标准参数 <VERSION>、REQUIRED 和 QUIET。CUDA_FOUND 将报告是否找到了可接受的 CUDA 版本。

如果无法通过系统路径中的 nvcc 位置确定前缀，且向 find_package() 指定了 REQUIRED，脚本将提示用户指定 CUDA_TOOLKIT_ROOT_DIR。若要使用不同版本的已安装工具包，请在运行 cmake 前设置环境变量 CUDA_BIN_PATH（例如设置为 CUDA_BIN_PATH=/usr/local/cuda1.0，而不是默认的 /usr/local/cuda），或在配置后设置 CUDA_TOOLKIT_ROOT_DIR。如果您更改了 CUDA_TOOLKIT_ROOT_DIR 的值，依赖于该路径的各个组件将被重定向。

在某些平台上，可能需要手动设置 CUDA_TOOLKIT_ROOT_DIR，或者使用未安装在默认位置的 CUDA 运行时。在较新版本的工具包中，CUDA 库包含在显卡驱动程序中——请确保驱动程序版本与 CUDA 运行时版本所需的内容匹配。

输入变量¶

以下变量（按字母顺序排列）会影响脚本中宏的行为。请注意，在调用 cuda_add_executable()、cuda_add_library()、cuda_compile()、cuda_compile_ptx()、cuda_compile_fatbin()、cuda_compile_cubin() 或 cuda_wrap_srcs() 之前，这些标志中的任何一个都可以在同一目录中多次更改。

CUDA_64_BIT_DEVICE_CODE (默认值：主机位宽)

设置为 ON 以编译 64 位设备代码，设置为 OFF 则为 32 位设备代码。请注意，当从 CUDA 代码生成对象文件或 C 文件时，如果将其与主机代码设置为不同，将无法工作，因为生成的源代码中 size_t 是由 nvcc 定义的。如果您编译为 PTX 然后自行加载文件，则可以混合使用设备和主机的位宽。

CUDA_ATTACH_VS_BUILD_RULE_TO_CUDA_FILE (默认值：ON)

如果您希望将自定义构建规则附加到 Visual Studio（集成开发环境） 中的源文件，请设置为 ON。如果您将同一个 cuda 文件添加到多个目标中，请将其设置为 OFF。

这允许用户从 CUDA 文件构建目标；但是，如果 CUDA 源文件被添加到多个目标中，可能会发生不好的事情。在执行并行构建时，自定义构建命令可能会被多次且并行运行，从而导致晦涩的构建错误。VS 会为目标中的每个源文件运行规则，无论它被添加到多少个项目中，一个源文件只能有一个规则。当规则从多个目标运行时，生成的文件可能会出现竞争条件。最终一切都会构建完成，但如果用户不知道这种行为，可能会产生困惑。如果此脚本能够检测到跨多个目标的源文件重用并自动为用户关闭该选项会很好，但目前尚未找到好的解决方案。

CUDA_BUILD_CUBIN (默认值：OFF)

设置为 ON 可在设备模式下启用带有 -cubin 选项的额外编译步骤。输出将被解析，并在构建期间打印寄存器和共享内存的使用情况。

CUDA_BUILD_EMULATION (默认值：设备模式下为 OFF)

设置为 ON 以使用仿真模式。当 CUDA_BUILD_EMULATION 为 TRUE 时，会为 CUDA C 文件定义 -D_DEVICEEMU。

CUDA_LINK_LIBRARIES_KEYWORD (默认值："")

版本 3.9 中添加。

用于内部 target_link_libraries() 调用的 <PRIVATE|PUBLIC|INTERFACE> 关键字。默认是不使用任何关键字，这会使用旧的“普通”形式的 target_link_libraries()。请注意这很重要，因为在 FindCUDA 模块内部使用的任何内容也必须在外部使用——两种形式的 target_link_libraries() 不能混合使用。

CUDA_GENERATED_OUTPUT_DIR (默认值：CMAKE_CURRENT_BINARY_DIR)

设置为您希望放置生成文件的路径。如果为空，输出文件将放置在 CMAKE_CURRENT_BINARY_DIR 中。中间文件将始终放置在 CMAKE_CURRENT_BINARY_DIR/CMakeFiles 中。

CUDA_HOST_COMPILATION_CPP (默认值：ON)

设置为 OFF 以使用 C 编译器编译主机代码。

CUDA_HOST_COMPILER (默认值：CMAKE_C_COMPILER)

设置 nvcc 使用的主机编译器。如果 CUDA_NVCC_FLAGS 或 CUDA_NVCC_FLAGS_<CONFIG> 变量中已存在 -ccbin 或 --compiler-bindir，则忽略此项。对于 Visual Studio（集成开发环境） 目标，主机编译器是通过一个或多个 Visual Studio（集成开发环境） 宏（例如 $(VCInstallDir)）构建的，当在 VS 中运行命令时，这些宏会展开为路径。

3.13 版本新增：如果设置了 CUDAHOSTCXX 环境变量，它将被用作默认值。

CUDA_NVCC_FLAGS, CUDA_NVCC_FLAGS_<CONFIG>

额外的 NVCC 命令行参数。注意：多个参数必须用分号分隔（例如 --compiler-options;-Wall）

3.6 版本新增：这些变量的内容可以使用 生成器表达式。

CUDA_PROPAGATE_HOST_FLAGS (默认值：ON)

设置为 ON，通过 nvcc 的 -Xcompiler 标志，将 CMAKE_{C,CXX}_FLAGS 及其配置相关的对应项（例如 CMAKE_C_FLAGS_DEBUG）自动传播给主机编译器。这有助于使生成的主机代码更好地与系统的其余部分相匹配。有时某些标志会导致 nvcc 问题，此设置可帮助您关闭标志传播。这不会影响通过 CUDA_NVCC_FLAGS 直接提供给 nvcc 的标志，也不会影响通过 cuda_add_library()、cuda_add_executable() 或 cuda_wrap_srcs() 指定的 OPTION 标志。用于共享库编译的标志不受此标志影响。

CUDA_SEPARABLE_COMPILATION (默认值：OFF)

如果设置，将为所有 CUDA 运行时对象文件启用可分离编译。如果是在 cuda_add_executable() 和 cuda_add_library() 之外使用（例如直接调用 cuda_wrap_srcs()），则应调用 cuda_compute_separable_compilation_object_file_name() 和 cuda_link_separable_compilation_objects()。

CUDA_SOURCE_PROPERTY_FORMAT

3.3 版本中新增。

如果设置了此源文件属性，它可以覆盖传递给 cuda_wrap_srcs() 的格式（OBJ、PTX、CUBIN 或 FATBIN）。如果输入源文件不是 .cu 文件，设置此属性将导致它被视为 .cu 文件。有关如何设置此属性，请参阅 set_source_files_properties 的文档。

CUDA_USE_STATIC_CUDA_RUNTIME (默认值：ON)

3.3 版本中新增。

启用后，将在 CUDA_LIBRARIES 中使用静态版本的 CUDA 运行时库。如果配置的 CUDA 版本不支持此选项，它将被静默禁用。

CUDA_VERBOSE_BUILD (默认值：OFF)

设置为 ON 以查看构建 CUDA 文件时使用的所有命令。使用 Makefile（构建脚本） 生成器时，该值默认为 VERBOSE（运行 make VERBOSE=1 以查看输出），尽管将 CUDA_VERBOSE_BUILD 设置为 ON 将始终打印输出。

cuda_select_nvcc_arch_flags(<out_variable> [<target_CUDA_architecture> ...])

代码说明

  • cuda_select_nvcc_arch_flags(<out_variable> [<target_CUDA_architecture> ...])：调用 `cuda_select_nvcc_arch_flags()` 命令，具体行为由这一行传入的参数决定。

根据 target_CUDA_architecture 为 nvcc 选择 GPU 架构标志。

target_CUDA_architecture 的值

  • Auto：在运行时检测本地机器的 GPU 计算架构。

  • Common 和 All：涵盖常见和完整的架构子集。

  • <name>：Fermi、Kepler、Maxwell、Kepler+Tegra、Kepler+Tesla、Maxwell+Tegra、Pascal 之一。

  • <ver>、<ver>(<ver>)、<ver>+PTX，其中 <ver> 是 2.0、2.1、3.0、3.2、3.5、3.7、5.0、5.2、5.3、6.0、6.2 之一。

返回在 <out_variable> 中添加到 CUDA_NVCC_FLAGS 的标志列表。此外，将 <out_variable>_readable 设置为生成的数字列表。

示例

cuda_select_nvcc_arch_flags(ARCH_FLAGS "3.0" "3.5+PTX" "5.2(5.0)" "Maxwell")
list(APPEND CUDA_NVCC_FLAGS ${ARCH_FLAGS})

代码说明

cuda_wrap_srcs(<cuda_target> <format> <generated_files> <file>...
               [STATIC | SHARED | MODULE] [OPTIONS ...])

代码说明

  • cuda_wrap_srcs(<cuda_target> <format> <generated_files> <file>... [STATIC | SHARED | MODULE] [OPTIONS ...])：调用 `cuda_wrap_srcs()` 命令，具体行为由这一行传入的参数决定。

这就是所有魔法发生的地方。cuda_add_executable()、cuda_add_library()、cuda_compile() 和 cuda_compile_ptx() 都在内部调用此函数。

给定文件列表 <file>...，此宏生成用于产生 PTX 或可链接对象的自定义命令（使用 PTX 或 OBJ 作为 <format> 参数进行切换）。不以 .cu 结尾或具有 HEADER_FILE_ONLY 属性的文件将被忽略。

OPTIONS 之后传入的参数是提供给 nvcc 的额外命令行选项。您也可以通过指定配置名称后跟选项来指定每个配置的选项。通用选项必须先于配置特定选项。不需要指定所有配置，仅使用提供的配置。例如：

cuda_add_executable(...
  OPTIONS -DFLAG=2 "-DFLAG_OTHER=space in flag"
  DEBUG -g
  RELEASE --use_fast_math
  RELWITHDEBINFO --use_fast_math;-g
  MINSIZEREL --use_fast_math)

代码说明

  • cuda_add_executable(... OPTIONS -DFLAG=2 "-DFLAG_OTHER=space in flag" DEBUG -g RELEASE --use_fast_math RELWITHDEBINFO --use_fast_math;-g MINSIZEREL --use_fast_math)：调用 `cuda_add_executable()` 命令，具体行为由这一行传入的参数决定。

对于某些配置（即在设置了 CUDA_ATTACH_VS_BUILD_RULE_TO_CUDA_FILE 为 ON 的情况下 VS 生成对象文件），给定的 cuda 文件不会产生生成的文件。这是因为当您将 cuda 文件添加到 Visual Studio（集成开发环境） 时，它知道该文件会产生一个对象文件，并会自动链接生成的对象文件。

此脚本还将生成一个单独的 cmake 脚本，该脚本在构建时用于调用 nvcc。原因有几个：

  • nvcc 可能会返回负数作为返回值，这会误导 Visual Studio（集成开发环境） 认为命令成功。该脚本现在会检查错误代码，并在出现问题时生成错误。

  • 众所周知，nvcc 在遇到问题时不会删除不完整的输出。这会误导构建系统认为目标已生成，而实际上存在一个不可用的文件。如果发生错误，该脚本现在会删除输出文件。

  • 通过将所有影响构建的选项放入一个文件，然后使构建规则依赖于该文件，当选项发生变化时，输出文件将被重新生成。

该脚本还查看可选参数 STATIC、SHARED 或 MODULE，以确定何时针对共享库进行对象编译。BUILD_SHARED_LIBS 在 cuda_wrap_srcs() 中被忽略，但在 cuda_add_library() 中会受到尊重。在某些系统上，会为构建旨在用于共享库的对象添加特殊标志。当检测到共享库编译时，会定义预处理器宏 <target_name>_EXPORTS。

使用 -D 或 /D 传递给 add_definitions 的标志也会传递给 nvcc。

结果变量¶

该脚本定义了以下变量

CUDA_VERSION_MAJOR

nvcc 报告的 CUDA 主要版本号。

CUDA_VERSION_MINOR

次要版本号。

CUDA_VERSION, CUDA_VERSION_STRING

X.Y 格式的完整版本号。

CUDA_HAS_FP16

3.6 版本新增：是否支持短浮点数（float16, fp16）。

CUDA_TOOLKIT_ROOT_DIR

CUDA 工具包的路径（如果未设置则定义）。

CUDA_SDK_ROOT_DIR

CUDA SDK 的路径。使用此路径在 SDK 中查找文件。此脚本不会直接支持查找特定的库或头文件，因为 NVIDIA 不支持这样做。如果您想在路径更改时更改库，请参阅 FindCUDA.cmake 脚本，了解如何清除这些变量的示例。如果愿意（风险自负），还有关于如何使用 CUDA_SDK_ROOT_DIR 定位头文件或库的示例。

CUDA_INCLUDE_DIRS

CUDA 头文件的包含目录。自动为 cuda_add_executable() 和 cuda_add_library() 添加。

CUDA_LIBRARIES

Cuda RT 库。

CUDA_CUFFT_LIBRARIES

用于 Cuda FFT 实现的设备或仿真库（cuda_add_cufft_to_target() 宏的替代方案）

CUDA_CUBLAS_LIBRARIES

用于 Cuda BLAS 实现的设备或仿真库（cuda_add_cublas_to_target() 宏的替代方案）。

CUDA_cudart_static_LIBRARY

可静态链接的 cuda 运行时库。仅适用于 CUDA 5.5+ 版本。

CUDA_cudadevrt_LIBRARY

3.7 版本新增：设备运行时库。可分离编译必需。

CUDA_cupti_LIBRARY

CUDA 分析工具接口库。仅适用于 CUDA 4.0+ 版本。

CUDA_curand_LIBRARY

CUDA 随机数生成库。仅适用于 CUDA 3.2+ 版本。

CUDA_cusolver_LIBRARY

3.2 版本新增：CUDA 直接求解器库。仅适用于 CUDA 7.0+ 版本。

CUDA_cusparse_LIBRARY

CUDA 稀疏矩阵库。仅适用于 CUDA 3.2+ 版本。

CUDA_npp_LIBRARY

NVIDIA 性能原语库。仅适用于 CUDA 4.0+ 版本。

CUDA_nppc_LIBRARY

NVIDIA 性能原语库（核心）。仅适用于 CUDA 5.5+ 版本。
