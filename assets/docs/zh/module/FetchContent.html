概述¶

该模块支持通过 ExternalProject 模块支持的任何方法，在配置时填充内容。与在构建时进行下载的 ExternalProject_Add() 不同，FetchContent 模块使内容立即可用，从而允许配置步骤在诸如 add_subdirectory()、include() 或 file() 操作等命令中使用这些内容。

内容填充的细节应与执行实际填充的命令分开定义。这种分离确保了在任何可能尝试使用它们来填充内容之前，所有的依赖细节都已经定义完成。这在复杂的项目层次结构中尤为重要，因为依赖项可能在多个项目之间共享。

以下示例展示了定义某些依赖项的内容细节，然后通过单独的调用确保它们被填充的典型案例：

FetchContent_Declare(
  googletest
  GIT_REPOSITORY https://github.com/google/googletest.git
  GIT_TAG        703bd9caab50b139428cea1aaff9974ebee5742e # release-1.10.0
)
FetchContent_Declare(
  myCompanyIcons
  URL      https://intranet.mycompany.com/assets/iconset_1.12.tar.gz
  URL_HASH MD5=5588a7b18261c20068beabfb4f530b87
)

FetchContent_MakeAvailable(googletest myCompanyIcons)

FetchContent_MakeAvailable() 命令确保指定的依赖项已被填充（通过之前的调用，或者由它自己执行填充）。执行填充时，如果可能，它还会将它们添加到主构建中，以便主构建可以使用已填充项目的目标等。有关如何执行这些步骤的详细信息，请参阅该命令的文档。

使用分层项目结构时，层次结构中更高级别的项目可以覆盖在项目层次结构中任何较低级别处指定的已声明内容细节。对于给定的依赖项，最先声明的细节具有优先权，无论该声明发生在项目层次结构中的何处。同样，第一个尝试填充依赖项的调用会“获胜”，后续的填充将复用第一次的结果，而不是再次重复填充过程。请参阅 示例，其中演示了此场景。

FetchContent 模块还支持在单次调用中定义和填充内容，而不检查该内容是否已在其他地方被填充。这不应在项目中执行，但对于在 CMake 脚本模式 下填充内容可能是合适的。有关详细信息，请参阅 FetchContent_Populate()。

命令¶

FetchContent_Declare¶
FetchContent_Declare(

  <contentOptions>...
  [EXCLUDE_FROM_ALL]
  [SYSTEM]
  [OVERRIDE_FIND_PACKAGE |
   FIND_PACKAGE_ARGS args...]
)

FetchContent_Declare() 函数记录描述如何填充指定内容的选项。如果此类细节已在本项目的较早位置记录（无论项目层次结构中的位置如何），则此调用以及后续对相同 <name> 内容的所有调用都将被忽略。这种“先记录者获胜”的方法允许分层项目由父项目覆盖子项目的内容细节。

内容 <name> 可以是任何不带空格的字符串，但最佳实践是仅使用字母、数字和下划线。名称将不区分大小写，且应能直观反映其代表的内容。它通常是子项目的名称，或者是传递给其顶级 project() 命令的值（如果是 CMake 项目）。对于知名的公共项目，名称通常应为项目的官方名称。选择一个不寻常的名称会降低其他需要相同内容的项目使用相同名称的可能性，从而导致内容被重复填充多次。

<contentOptions> 可以是 ExternalProject_Add() 命令所理解的任何下载、更新或补丁选项。配置、构建、安装和测试步骤被显式禁用，因此与这些步骤相关的选项被禁止，如果提供将被丢弃。SOURCE_SUBDIR 选项是一个例外，有关它如何影响行为的详细信息，请参阅 FetchContent_MakeAvailable()。

版本 3.30 中更改：当策略 CMP0168 设置为 NEW 时，某些与输出和目录相关的选项将被忽略。有关详细信息，请参阅策略文档。

在大多数情况下，<contentOptions> 将只是定义下载方法以及特定方法细节（如提交标签或归档哈希）的几个选项。例如

  googletest
  GIT_REPOSITORY https://github.com/google/googletest.git
  GIT_TAG        703bd9caab50b139428cea1aaff9974ebee5742e # release-1.10.0
)

FetchContent_Declare(
  myCompanyIcons
  URL      https://intranet.mycompany.com/assets/iconset_1.12.tar.gz
  URL_HASH MD5=5588a7b18261c20068beabfb4f530b87
)

FetchContent_Declare(
  myCompanyCertificates
  SVN_REPOSITORY svn+ssh://svn.mycompany.com/srv/svn/trunk/certs
  SVN_REVISION   -r12345

版本 3.22 中添加：CMAKE_TLS_VERIFY、CMAKE_TLS_CAINFO、CMAKE_NETRC 和 CMAKE_NETRC_FILE 变量现在提供其对应内容选项的默认值，就像它们对 ExternalProject_Add() 所做的那样。以前，这些变量会被 FetchContent 模块忽略。

版本 3.24 中添加

FIND_PACKAGE_ARGS

此选项用于 FetchContent_MakeAvailable() 命令可能首先尝试调用 find_package() 来满足 <name> 依赖项的场景。默认情况下，此类调用仅为 find_package(<name>)，但 FIND_PACKAGE_ARGS 可用于提供在 <name> 之后追加的附加参数。FIND_PACKAGE_ARGS 后面也可以不带任何内容，这表示如果 FETCHCONTENT_TRY_FIND_PACKAGE_MODE 设置为 OPT_IN 或者未设置，仍然可以调用 find_package()。

通常不适合在 FIND_PACKAGE_ARGS 后的附加参数中指定 REQUIRED。这样做意味着 find_package() 调用必须成功，因此 FetchContent_Declare() 调用中指定的任何其他细节将没有机会作为回退使用。

FIND_PACKAGE_ARGS 关键字之后的所有内容都会追加到 find_package() 调用中，因此所有其他 <contentOptions> 必须出现在 FIND_PACKAGE_ARGS 关键字之前。如果在调用 FetchContent_Declare() 时 CMAKE_FIND_PACKAGE_TARGETS_GLOBAL 变量设置为 true，则如果未指定 GLOBAL 关键字，它将被追加到 find_package() 参数中。如果未给出 FIND_PACKAGE_ARGS，但 FETCHCONTENT_TRY_FIND_PACKAGE_MODE 设置为 ALWAYS，它也会被追加。

当给出 FIND_PACKAGE_ARGS 时，不能使用 OVERRIDE_FIND_PACKAGE。

依赖提供程序 讨论了重定向 FetchContent_MakeAvailable() 调用的另一种方式。FIND_PACKAGE_ARGS 旨在用于项目控制，而依赖提供程序则允许用户覆盖项目行为。

OVERRIDE_FIND_PACKAGE

当 FetchContent_Declare(<name> ...) 调用包含此选项时，后续对 find_package(<name> ...) 的调用将确保已调用 FetchContent_MakeAvailable(<name>)，然后使用 CMAKE_FIND_PACKAGE_REDIRECTS_DIR 目录中的配置包文件（通常由 FetchContent_MakeAvailable() 创建）。这有效地使 FetchContent_MakeAvailable() 覆盖了命名依赖项的 find_package()，从而允许前者满足后者的包要求。当给出 OVERRIDE_FIND_PACKAGE 时，不能使用 FIND_PACKAGE_ARGS。

如果已设置 依赖提供程序 且项目为 <name> 依赖项调用了 find_package()，OVERRIDE_FIND_PACKAGE 不会阻止提供程序看到该调用。依赖提供程序始终有机会拦截任何对 find_package() 的直接调用，除非该调用包含 BYPASS_PROVIDER 选项。

版本 3.25 中添加

SYSTEM

如果提供了 SYSTEM 参数，则由 FetchContent_MakeAvailable() 添加的子目录的 SYSTEM 目录属性将被设置为 true。这将影响作为该命令一部分创建的非导入目标。有关效果的更详细讨论，请参阅 SYSTEM 目标属性文档。

版本 3.28 中添加

EXCLUDE_FROM_ALL

如果提供了 EXCLUDE_FROM_ALL 参数，则由 FetchContent_MakeAvailable() 添加的子目录中的目标将默认不会包含在 ALL 目标中，并且可能会被排除在 IDE 项目文件之外。有关效果的详细讨论，请参阅目录属性 EXCLUDE_FROM_ALL 的文档。

FetchContent_MakeAvailable¶

3.14 版新增。

FetchContent_MakeAvailable(<name1> [<name2>...])

代码说明

  • FetchContent_MakeAvailable(<name1> [<name2>...])：调用 FetchContent_MakeAvailable() 命令，具体行为由这一行传入的参数决定。

此命令确保每个命名的依赖项在返回时都已对项目可用。必须已为每个依赖项调用了 FetchContent_Declare()，并且第一个此类调用将控制该依赖项如何变得可用，如下所述。

如果未设置 <lowercaseName>_SOURCE_DIR

  • 版本 3.24 中添加：如果设置了 依赖提供程序，则使用 FETCHCONTENT_MAKEAVAILABLE_SERIAL 作为第一个参数，后跟 FetchContent_Declare() 对 <name> 的首次调用的参数来调用提供程序的命令。如果 SOURCE_DIR 或 BINARY_DIR 不在原始声明的参数中，它们将以默认值添加。如果细节声明时 FETCHCONTENT_TRY_FIND_PACKAGE_MODE 设置为 NEVER，则任何 FIND_PACKAGE_ARGS 都将被省略。OVERRIDE_FIND_PACKAGE 关键字也总是被省略。如果提供程序满足了请求，FetchContent_MakeAvailable() 将认为该依赖项已处理，跳过下方的剩余步骤，并继续处理列表中的下一个依赖项。

  • 版本 3.24 中添加：如果允许，将调用 find_package(<name> [<args>...])，其中 <args>... 可能由 FetchContent_Declare() 中的 FIND_PACKAGE_ARGS 选项提供。调用 FetchContent_Declare() 时 FETCHCONTENT_TRY_FIND_PACKAGE_MODE 变量的值决定了 FetchContent_MakeAvailable() 是否可以调用 find_package()。如果在调用 FetchContent_MakeAvailable() 时 CMAKE_FIND_PACKAGE_TARGETS_GLOBAL 变量设置为 true，它仍然会影响该调用依次调用 find_package() 时创建的任何导入目标，即使在声明相应细节时该变量为 false。

如果依赖项未由提供程序或 find_package() 调用满足，FetchContent_MakeAvailable() 然后使用以下逻辑使依赖项可用

  • 如果该依赖项已在本次运行中较早前填充，则按照 FetchContent_GetProperties() 的调用方式设置 <lowercaseName>_POPULATED、<lowercaseName>_SOURCE_DIR 和 <lowercaseName>_BINARY_DIR 变量，然后跳过下方的剩余步骤，继续处理列表中的下一个依赖项。

  • 使用由较早的 FetchContent_Declare() 调用记录的细节填充依赖项。如果没有记录此类细节，则以致命错误终止。FETCHCONTENT_SOURCE_DIR_<uppercaseName> 可用于覆盖已声明的细节，并改用指定位置提供的内容。

  • 版本 3.24 中添加：确保 CMAKE_FIND_PACKAGE_REDIRECTS_DIR 目录包含一个 <lowercaseName>-config.cmake 和一个 <lowercaseName>-config-version.cmake 文件（或者等效地，<name>Config.cmake 和 <name>ConfigVersion.cmake）。CMAKE_FIND_PACKAGE_REDIRECTS_DIR 变量指向的目录在每次 CMake 运行开始时都会被清除。如果在上一步填充依赖项后不存在配置文件，则会写入一个最小化的文件，该文件使用 OPTIONAL 标志 包含 任何 <lowercaseName>-extra.cmake 或 <name>Extra.cmake 文件（因此这些文件可以缺失且不会产生警告）。类似地，如果不存在配置版本文件，则会写入一个非常简单的版本文件，将 PACKAGE_VERSION_COMPATIBLE 和 PACKAGE_VERSION_EXACT 设置为 true。这确保了未来对该依赖项的所有 find_package() 调用都将使用重定向的配置文件，而无需考虑任何版本要求。CMake 无法自动确定任意依赖项的版本，因此无法设置 PACKAGE_VERSION。当依赖项在下一步通过 add_subdirectory() 引入时，它可以选择使用同时设置了 PACKAGE_VERSION 的文件覆盖 CMAKE_FIND_PACKAGE_REDIRECTS_DIR 中生成的配置版本文件。依赖项还可以编写一个 <lowercaseName>-extra.cmake 或 <name>Extra.cmake 文件来执行自定义处理，或定义其正常的（已安装的）包配置文件通常会定义的任何变量（许多项目不进行任何自定义处理或设置任何变量，因此无需执行此操作）。如有需要，如果依赖项目没有这样做，主项目可以改为编写这些文件。这允许主项目添加来自尚未或无法更新以支持此功能的旧依赖项的缺失细节。有关示例，请参阅 与 find_package() 集成。

  • 如果已填充内容的顶级目录包含一个 CMakeLists.txt 文件，则调用 add_subdirectory() 将其添加到主构建中。没有 CMakeLists.txt 文件并不是错误，这允许该命令用于那些使下载内容在已知位置可用，但不需要或不支持直接添加到构建中的依赖项。

    版本 3.18 中添加：可以在声明的细节中提供 SOURCE_SUBDIR 选项以在顶级目录下的某处查找（即 ExternalProject_Add() 命令使用 SOURCE_SUBDIR 的方式相同）。随 SOURCE_SUBDIR 提供的路径必须是相对路径，它将被视为相对于顶级目录。它也可以指向一个不包含 CMakeLists.txt 文件的目录，甚至是一个不存在的目录。这可用于避免添加在顶级目录中包含 CMakeLists.txt 文件的项目。

    版本 3.25 中添加：如果 SYSTEM 关键字包含在对 FetchContent_Declare() 的调用中，SYSTEM 关键字将被添加到 add_subdirectory() 命令中。

    版本 3.28 中添加：如果 EXCLUDE_FROM_ALL 关键字包含在对 FetchContent_Declare() 的调用中，EXCLUDE_FROM_ALL 关键字将被添加到 add_subdirectory() 命令中。

    版本 3.29 中添加：在调用 add_subdirectory() 之前，CMAKE_EXPORT_FIND_PACKAGE_NAME 被设置为依赖项名称。

项目应旨在为它们可能使用的所有依赖项声明细节，然后再为它们中的任何一个调用 FetchContent_MakeAvailable()。这确保了如果任何依赖项也是其他一个或多个依赖项的子依赖项，主项目仍然可以控制将要使用的细节（因为它会在依赖项有机会声明它们之前首先声明它们）。在以下代码示例中，假设 uses_other 依赖项也使用 FetchContent 在内部添加 other 依赖项。

# WRONG: Should declare all details first
FetchContent_Declare(uses_other ...)
FetchContent_MakeAvailable(uses_other)

FetchContent_Declare(other ...)    # Will be ignored, uses_other beat us to it
FetchContent_MakeAvailable(other)  # Would use details declared by uses_other

  • FetchContent_Declare(uses_other ...)：调用 FetchContent_Declare() 命令，具体行为由这一行传入的参数决定。

  • FetchContent_MakeAvailable(uses_other)：调用 FetchContent_MakeAvailable() 命令，具体行为由这一行传入的参数决定。

  • FetchContent_Declare(other ...) # Will be ignored, uses_other beat us to it：这一行继续补充当前命令或表达式的内容。

  • FetchContent_MakeAvailable(other) # Would use details declared by uses_other：这一行继续补充当前命令或表达式的内容。

FetchContent_Declare(uses_other ...)
FetchContent_Declare(other ...)

FetchContent_Populate¶

FetchContent_Populate() 命令是一个自包含的调用，可用于将内容填充作为独立操作执行。它很少是正确使用的命令，项目几乎应该始终使用 FetchContent_Declare() 和 FetchContent_MakeAvailable()。 FetchContent_Populate() 的主要用例是在 CMake 脚本模式 下，作为实现其他更高级自定义功能的一部分。

  • EXCLUDE_FROM_ALL

  • SYSTEM

  • OVERRIDE_FIND_PACKAGE

  • FIND_PACKAGE_ARGS

上面签名中显示的少数选项要么是 FetchContent_Populate() 特有的，要么其行为与 ExternalProject_Add() 处理它们的方式略有不同。

QUIET

可以提供 QUIET 选项来隐藏与填充指定内容相关的输出。如果填充失败，无论是否提供了此选项，输出都将显示，以便可以诊断失败的原因。FETCHCONTENT_QUIET 变量对这种形式的直接提供内容细节的 FetchContent_Populate() 调用没有影响。

版本 3.30 中更改：当策略 CMP0168 设置为 NEW 时，QUIET 选项和 FETCHCONTENT_QUIET 变量无效。在此情况下输出默认仍为静默，但冗长程度由消息记录级别控制（请参阅 CMAKE_MESSAGE_LOG_LEVEL 和 --log-level）。

SUBBUILD_DIR

可以提供 SUBBUILD_DIR 参数来更改为执行填充而创建的子构建的位置。默认值为 ${CMAKE_CURRENT_BINARY_DIR}/<lowercaseName>-subbuild，覆盖此默认值是不寻常的。如果指定了相对路径，它将被解释为相对于 CMAKE_CURRENT_BINARY_DIR。此选项不应与 SOURCE_SUBDIR 选项混淆，后者仅影响 FetchContent_MakeAvailable() 命令。

版本 3.30 中更改：当策略 CMP0168 设置为 NEW 时，SUBBUILD_DIR 被忽略，因为在这种情况下没有子构建。

SOURCE_DIR, BINARY_DIR

SOURCE_DIR 和 BINARY_DIR 参数受 ExternalProject_Add() 支持，但 FetchContent_Populate() 使用不同的默认值。SOURCE_DIR 默认为 ${CMAKE_CURRENT_BINARY_DIR}/<lowercaseName>-src，BINARY_DIR 默认为 ${CMAKE_CURRENT_BINARY_DIR}/<lowercaseName>-build。如果指定了相对路径，它将被解释为相对于 CMAKE_CURRENT_BINARY_DIR。

除了上述显式选项外，任何其他未识别的选项都会原封不动地传递给 ExternalProject_Add() 以设置下载、补丁和更新步骤。以下选项被明确禁止（它们被 FetchContent_Populate() 命令禁用）

  • CONFIGURE_COMMAND

  • BUILD_COMMAND

  • INSTALL_COMMAND

  • TEST_COMMAND

使用此形式，FETCHCONTENT_FULLY_DISCONNECTED 和 FETCHCONTENT_UPDATES_DISCONNECTED 变量以及策略 CMP0170 将被忽略。

当这种形式的 FetchContent_Populate() 返回时，以下变量将在调用者的作用域内设置

<lowercaseName>_SOURCE_DIR

返回时可以找到已填充内容的位置。

<lowercaseName>_BINARY_DIR

原本打算用作相应构建目录的目录，但在使用此形式的命令时不太可能相关。

如果在 CMake 脚本模式 内使用 FetchContent_Populate()，请注意实现会设置一个子构建，因此需要可用的 CMake 生成器和构建工具。如果这些不能默认找到，则需要适当地在调用脚本的命令行上设置 CMAKE_GENERATOR 和潜在的 CMAKE_MAKE_PROGRAM 变量。

版本 3.30 中更改：如果策略 CMP0168 设置为 NEW，则不使用子构建。在 CMake 脚本模式 下，这允许在没有任何构建工具或 CMake 生成器的情况下调用 FetchContent_Populate()。

版本 3.18 中添加：增加了对 DOWNLOAD_NO_EXTRACT 选项的支持。

该命令支持另一种形式，尽管不应再使用它

变量¶

许多缓存变量会影响使用 FetchContent_Declare() 调用的细节来填充内容时的行为。

注意

所有这些变量均旨在供开发人员自定义行为。它们通常不应由项目设置。

FETCHCONTENT_BASE_DIR¶

在大多数情况下，保存的细节不会指定任何有关用于内部子构建、最终源和构建区域的目录的选项。最好将这些决定留给 FetchContent 模块来代表项目处理。FETCHCONTENT_BASE_DIR 缓存变量控制收集所有内容填充目录的点，但在大多数情况下，开发人员不需要更改此项。默认位置是 ${CMAKE_BINARY_DIR}/_deps，但如果开发人员更改此值，他们应设法保持路径较短，并位于构建树的顶级下方，以避免在 Windows（微软桌面系统） 上遇到路径长度问题。

FETCHCONTENT_QUIET¶

填充期间的日志输出可能非常详细，使得配置阶段非常嘈杂。此缓存选项（默认为 ON）会隐藏所有填充输出，除非遇到错误。如果遇到下载挂起的问题，暂时关闭此选项可能有助于诊断是哪种内容填充导致了问题。

版本 3.30 中更改：如果策略 CMP0168 设置为 NEW，则 FETCHCONTENT_QUIET 被忽略。在此情况下输出默认仍为静默，但冗长程度由消息记录级别控制（请参阅 CMAKE_MESSAGE_LOG_LEVEL 和 --log-level）。

FETCHCONTENT_FULLY_DISCONNECTED¶

启用此选项时，不会尝试下载或更新任何内容。假定所有内容已在之前的运行中填充，或者源目录已指向开发人员手动提供的现有内容（使用下文描述的选项）。当开发人员知道内容细节未发生更改时，将此选项设置为 ON 可以加快配置阶段。默认情况下为 OFF。

注意

FETCHCONTENT_FULLY_DISCONNECTED 变量不是防止在构建目录中首次运行时进行任何网络访问的合适方式。这样做会破坏项目，导致误导性的错误消息，并隐藏隐蔽的填充失败。此变量专门旨在仅在 CMake 首次运行*之后*开启。如果您想防止网络访问，即使在首次运行时也是如此，请使用 依赖提供程序 并改为从本地内容填充依赖项。

版本 3.30 中更改：当 FETCHCONTENT_FULLY_DISCONNECTED 为 true 时，源目录必须已填充的约束现在已强制执行。请参阅策略 CMP0170。

FETCHCONTENT_UPDATES_DISCONNECTED¶

与 FETCHCONTENT_FULLY_DISCONNECTED 相比，这是一种限制较轻的下载/更新控制方式。与跳过所有下载和更新逻辑不同，FETCHCONTENT_UPDATES_DISCONNECTED 仅在使用 git 或 hg 下载方法时，防止更新步骤连接远程服务器。如果更新步骤的细节发生变化，更新仍会发生，但更新尝试只会利用本地已有的信息（因此切换到本地已获取的不同标签或提交将成功，但切换到未知的提交哈希值则会失败）。下载步骤不受影响，因此如果内容之前未下载，即使启用了此选项，它仍会被下载。这可以加快配置步骤的速度，但效果不如 FETCHCONTENT_FULLY_DISCONNECTED。FETCHCONTENT_UPDATES_DISCONNECTED 默认值为 OFF。

FETCHCONTENT_TRY_FIND_PACKAGE_MODE¶

在 3.24 版本中添加。

该变量会修改 FetchContent_Declare() 为给定依赖项记录的详细信息。虽然它最终控制 FetchContent_MakeAvailable() 的行为，但在调用 FetchContent_Declare() 时，该变量的值才会被使用。在调用 FetchContent_MakeAvailable() 时，该变量设置为什么值并不重要。由于该变量通常仅由用户设置而不应由项目直接设置，因此它在整个过程中通常具有相同的值，所以这种区别通常不明显。

FETCHCONTENT_TRY_FIND_PACKAGE_MODE 最终控制 FetchContent_MakeAvailable() 是否允许调用 find_package() 来满足依赖项。该变量可以设置为以下值之一：

OPT_IN

FetchContent_MakeAvailable() 仅在 FetchContent_Declare() 调用中包含 FIND_PACKAGE_ARGS 关键字时，才会调用 find_package()。如果未设置 FETCHCONTENT_TRY_FIND_PACKAGE_MODE，这也是默认行为。

ALWAYS

find_package() 可由 FetchContent_MakeAvailable() 调用，无论 FetchContent_Declare() 调用是否包含 FIND_PACKAGE_ARGS 关键字。如果未提供 FIND_PACKAGE_ARGS 关键字，其行为将等同于提供了 FIND_PACKAGE_ARGS 且其后没有任何附加参数的情况。

NEVER

FetchContent_MakeAvailable() 不会调用 find_package()。传递给 FetchContent_Declare() 调用的任何 FIND_PACKAGE_ARGS 都将被忽略。

作为一种特殊情况，如果 FETCHCONTENT_SOURCE_DIR_<uppercaseName> 变量对于某个依赖项具有非空值，则假定用户正在覆盖所有其他使该依赖项可用的方法。FETCHCONTENT_TRY_FIND_PACKAGE_MODE 对该依赖项将不起作用，且 FetchContent_MakeAvailable() 不会尝试为其调用 find_package()。

除上述内容外，每个内容名称还定义了以下变量：

FETCHCONTENT_SOURCE_DIR_<uppercaseName>¶

如果设置了此项，则不会为指定的内容执行下载或更新步骤，并且返回给调用者的 <lowercaseName>_SOURCE_DIR 变量将指向此位置。这为开发人员提供了一种单独检出内容的方法，他们可以自由修改这些内容而不会受到构建过程的干扰。构建过程直接使用现有的源代码，但它仍然定义了 <lowercaseName>_BINARY_DIR 以指向其自身的构建区域。强烈建议开发人员使用此机制，而不是编辑默认位置中填充的源代码，因为当项目更改内容填充细节时，默认位置中源代码的更改可能会丢失。

FETCHCONTENT_UPDATES_DISCONNECTED_<uppercaseName>¶

这是 FETCHCONTENT_UPDATES_DISCONNECTED 的特定于内容的版本。如果全局选项或此选项为 ON，则 git 和 hg 方法的更新将不会为指定内容联系任何远程服务器。它们将仅使用本地已有的信息。为单个内容禁用更新对于那些细节很少更改的内容非常有用，同时仍然保持其他经常更改的内容处于启用更新的状态。

上述内容中应注意以下几个关键点：

  • projB 和 projC 为 projD 定义了不同的内容详细信息，但 projA 也为 projD 定义了一组内容详细信息。由于 projA 会首先定义它们，因此来自 projB 和 projC 的详细信息将不会被使用。由 projA 定义的覆盖详细信息不需要与 projB 或 projC 中的任何一个匹配，但由较高级别的项目确保其定义的详细信息对子项目仍然有意义，是其责任。

  • 在 projA 对 FetchContent_MakeAvailable() 的调用中，projD 列在 projB 和 projC 之前，因此它将在 projB 或 projC 之前被填充。projA 不强制要求这样做，但这样做可以确保 projA 完全控制 projD 被引入构建的环境（目录属性特别相关）。

  • 虽然 projA 定义了 projE 的内容详细信息，但它本身不需要显式调用 FetchContent_MakeAvailable(projE) 或 FetchContent_Populate(projD)。相反，它将这些留给子项目 projB。对于较高级别的项目，通常只需定义覆盖内容详细信息并将实际填充留给子项目即可。这避免了在项目层级的每一层重复相同的事情，但只有在预期依赖项设置的目录属性不会影响共享依赖项（本例中为 projE）的填充时才应这样做。

填充内容而不将其添加到构建中¶

项目并不总是需要将填充的内容添加到构建中。有时项目只是想让下载的内容在可预测的位置可用。下一个示例确保了一组标准的公司工具链文件（甚至可能包括工具链二进制文件本身）足够早地可用，以便用于同一构建。
