GoogleTest¶

版本 3.9 中添加。

此模块提供了一些命令，用于帮助使用 Google Test 基础设施。

在 CMake 项目中加载此模块，使用

include(GoogleTest)

代码说明

  • include(GoogleTest)：调用 `include()` 命令，加载并执行另一个 CMake 脚本。

模块提供了两种添加测试的机制。gtest_add_tests() 已存在一段时间，最初通过 find_package(GTest) 提供。gtest_discover_tests() 则是在 CMake 3.10 中引入的。

（较旧的）gtest_add_tests() 通过扫描源代码来识别测试。这通常很有效，但也存在一些局限性，包括在交叉编译环境中，且为测试设置额外属性较为繁琐。此外，它对参数化测试的处理不够全面，并且需要重新运行 CMake 才能检测到测试列表的变化。

（较新的）gtest_discover_tests() 通过让已编译的测试可执行文件枚举其测试来发现测试。这种方法更健壮，对参数化测试的处理更好，并且当测试更改时不需要重新运行 CMake。然而，它可能无法在交叉编译环境中工作，且设置测试属性的便利性稍逊一筹。

更多详细信息可在各函数的文档中找到。

这两个命令旨在取代使用 add_test() 来注册测试，并会为每个 Google Test 测试用例创建一个独立的 CTest 测试。请注意，在某些情况下这样做效率较低，因为在同一实例中执行的多个测试用例无法共享通用的设置（set-up）和清理（tear-down）逻辑。不过，它为 CTest 提供了更细粒度的通过/失败信息，这通常被认为更有益处。默认情况下，CTest 测试名称与 Google Test 名称相同（即 suite.testcase）；另请参阅 TEST_PREFIX 和 TEST_SUFFIX。

gtest_add_tests¶

通过扫描源代码中的 Google Test 宏，自动使用 CTest 添加测试

gtest_add_tests(TARGET target
                [SOURCES src1...]
                [EXTRA_ARGS args...]
                [WORKING_DIRECTORY dir]
                [TEST_PREFIX prefix]
                [TEST_SUFFIX suffix]
                [SKIP_DEPENDENCY]
                [TEST_LIST outVar]
)

代码说明

  • gtest_add_tests(TARGET target [SOURCES src1...] [EXTRA_ARGS args...] [WORKING_DIRECTORY dir] [TEST_PREFIX prefix] [TEST_SUFFIX suffix] [SKIP_DEPENDENCY] [TEST_LIST outVar] )：调用 `gtest_add_tests()` 命令，具体行为由这一行传入的参数决定。

gtest_add_tests 尝试通过扫描源文件来识别测试。尽管这通常很有效，但它仅使用基础的正则表达式匹配，非典型的测试声明可能会导致匹配失败，并且无法完全“拆分”参数化测试。此外，它要求重新运行 CMake 以发现任何新增、移除或重命名的测试（默认情况下，这意味着当任何测试源文件更改时，CMake 都会重新运行，但请参阅 SKIP_DEPENDENCY）。不过，它的优势在于可以在 CMake 配置时声明测试，这在一定程度上简化了设置测试的额外属性，并且始终可以在交叉编译环境中工作。

选项包括

TARGET target

指定 Google Test 可执行文件，必须是一个已知的 CMake 可执行目标。在运行测试时，CMake 将替换为已构建可执行文件的路径。

SOURCES src1...

提供此项后，仅会扫描所列的文件以查找测试用例。如果不提供此选项，将使用指定 target 的 SOURCES 属性来获取源文件列表。

EXTRA_ARGS args...

传递给每个测试用例命令行的任何额外参数。

3.31 版本变更: args... 中的空值会被保留，详见 CMP0178。

WORKING_DIRECTORY dir

指定运行所发现测试用例的目录。如果不提供此选项，则使用当前二进制目录。

TEST_PREFIX prefix

指定要在每个发现的测试用例名称前添加的 prefix。当在多次调用 gtest_add_test() 时使用相同的源文件但采用不同的 EXTRA_ARGS 时，这非常有用。

TEST_SUFFIX suffix

与 TEST_PREFIX 类似，不同之处在于 suffix 会附加到每个发现的测试用例名称末尾。可以同时指定 TEST_PREFIX 和 TEST_SUFFIX。

SKIP_DEPENDENCY

通常，该函数会创建一个依赖项，导致如果扫描的任何源文件发生更改，CMake 就会重新运行。这是为了确保发现的测试列表得到更新。如果不希望这种行为（例如在编写测试用例时），可以使用此选项来阻止添加该依赖项。

TEST_LIST outVar

由 outVar 命名的变量将在调用范围内填充所发现的测试用例列表。这允许调用者执行诸如操作所发现测试的属性之类的任务。

3.31 版本变更: TEST_LAUNCHER 和 CROSSCOMPILING_EMULATOR 目标属性中的空值会被保留，详见策略 CMP0178。

使用示例

include(GoogleTest)
add_executable(FooTest FooUnitTest.cxx)
gtest_add_tests(TARGET      FooTest
                TEST_SUFFIX .noArgs
                TEST_LIST   noArgsTests
)
gtest_add_tests(TARGET      FooTest
                EXTRA_ARGS  --someArg someValue
                TEST_SUFFIX .withArgs
                TEST_LIST   withArgsTests
)
set_tests_properties(${noArgsTests}   PROPERTIES TIMEOUT 10)
set_tests_properties(${withArgsTests} PROPERTIES TIMEOUT 20)

代码说明

  • include(GoogleTest) add_executable(FooTest FooUnitTest.cxx) gtest_add_tests(TARGET FooTest TEST_SUFFIX .noArgs TEST_LIST noArgsTests ) gtest_add_tests(TARGET FooTest EXTRA_ARGS --someArg someValue TEST_SUFFIX .withArgs TEST_LIST withArgsTests ) set_tests_properties(${noArgsTests} PROPERTIES TIMEOUT 10) set_tests_properties(${withArgsTests} PROPERTIES TIMEOUT 20)：调用 `include()` 命令，加载并执行另一个 CMake 脚本。

为了向后兼容，也支持以下形式

gtest_add_tests(exe args files...)

代码说明

  • gtest_add_tests(exe args files...)：调用 `gtest_add_tests()` 命令，具体行为由这一行传入的参数决定。

exe

测试可执行文件的路径或 CMake 目标的名称。

args

传递给可执行文件的额外参数的分号分隔列表。整个列表必须作为单个参数传递。请将其括在引号中，或者如果不需要参数则传递 ""。

files...

要搜索测试和测试夹具的源文件列表。或者，使用 AUTO 来指定 exe 是一个 CMake 可执行目标的名称，其源文件应该被扫描。

include(GoogleTest)
set(FooTestArgs --foo 1 --bar 2)
add_executable(FooTest FooUnitTest.cxx)
gtest_add_tests(FooTest "${FooTestArgs}" AUTO)

代码说明

  • include(GoogleTest) set(FooTestArgs --foo 1 --bar 2) add_executable(FooTest FooUnitTest.cxx) gtest_add_tests(FooTest "${FooTestArgs}" AUTO)：调用 `include()` 命令，加载并执行另一个 CMake 脚本。

gtest_discover_tests¶

通过查询已编译的测试可执行文件以获取可用测试，从而自动使用 CTest 添加测试

gtest_discover_tests(target
                     [EXTRA_ARGS args...]
                     [WORKING_DIRECTORY dir]
                     [TEST_PREFIX prefix]
                     [TEST_SUFFIX suffix]
                     [TEST_FILTER expr]
                     [NO_PRETTY_TYPES] [NO_PRETTY_VALUES]
                     [PROPERTIES name1 value1...]
                     [TEST_LIST var]
                     [DISCOVERY_TIMEOUT seconds]
                     [XML_OUTPUT_DIR dir]
                     [DISCOVERY_MODE <POST_BUILD|PRE_TEST>]
                     [DISCOVERY_EXTRA_ARGS args...]
)

代码说明

  • gtest_discover_tests(target [EXTRA_ARGS args...] [WORKING_DIRECTORY dir] [TEST_PREFIX prefix] [TEST_SUFFIX suffix] [TEST_FILTER expr] [NO_PRETTY_TYPES] [NO_PRETTY_VALUES] [PROPERTIES name1 value1...] [TEST_LIST var] [DISCOVERY_TIMEOUT seconds] [XML_OUTPUT_DIR dir] [DISCOVERY_MODE <POST_BUILD|PRE_TEST>] [DISCOVERY_EXTRA_ARGS args...] )：调用 `gtest_discover_tests()` 命令，具体行为由这一行传入的参数决定。

3.10 版本新增。

gtest_discover_tests() 在测试可执行文件上设置了一个构建后或测试前命令，通过解析以 --gtest_list_tests 参数运行测试可执行文件的输出来生成测试列表。与 gtest_add_tests() 的源代码解析方法相比，这确保了能够获得完整的测试列表，包括参数化测试的实例化。由于测试发现发生在构建或测试阶段，当测试列表发生更改时，无需重新运行 CMake。但是，它要求正确设置 CROSSCOMPILING_EMULATOR 才能在交叉编译环境中工作。

此外，在测试上设置属性的便利性稍逊，因为这些测试在 CMake 配置时不可用。可以使用 PROPERTIES 选项将额外的测试属性作为一个整体分配给这组测试。如果需要更细粒度的测试控制，可以通过使用 TEST_INCLUDE_FILES 目录属性的外部 CTest 脚本提供自定义内容。发现的测试集通过 <target>_TESTS 变量对脚本可见（请参阅下文关于 TEST_LIST 选项的进一步讨论和限制）。

选项包括

目标

指定 Google Test 可执行文件，必须是一个已知的 CMake 可执行目标。在运行测试时，CMake 将替换为已构建可执行文件的路径。

EXTRA_ARGS args...

传递给每个测试用例命令行的任何额外参数。

3.31 版本变更: args... 中的空值会被保留，详见 CMP0178。

WORKING_DIRECTORY dir

指定运行所发现测试用例的目录。如果不提供此选项，则使用当前二进制目录。

TEST_PREFIX prefix

指定要在每个发现的测试用例名称前添加的 prefix。当在多次调用 gtest_discover_tests() 时使用相同的测试可执行文件但采用不同的 EXTRA_ARGS 时，这非常有用。

TEST_SUFFIX suffix

与 TEST_PREFIX 类似，不同之处在于 suffix 会附加到每个发现的测试用例名称末尾。可以同时指定 TEST_PREFIX 和 TEST_SUFFIX。

TEST_FILTER expr

3.22 版本新增。

在测试发现期间作为 --gtest_filter 参数传递的过滤表达式。请注意，该表达式是一种基于通配符的格式，用于匹配 gtest 使用的原始测试名称。对于类型或值参数化测试，这些名称可能与 ctest 使用的可能经过美化打印的测试名称不同。

NO_PRETTY_TYPES

默认情况下，类型参数化测试的类型索引在 CTest 测试名称中会被实际的类型名称替换。如果不希望这种行为（例如因为类型名称太复杂），此选项将禁止该行为。

NO_PRETTY_VALUES

默认情况下，值参数化测试的值索引在 CTest 测试名称中会被实际值替换。如果不希望这种行为（例如因为值字符串太复杂），此选项将禁止该行为。

PROPERTIES name1 value1...

指定要为本次 gtest_discover_tests() 调用所发现的所有测试设置的附加属性。

TEST_LIST var

使测试列表在变量 var 中可用，而不是默认的 <target>_TESTS。当在多次调用 gtest_discover_tests() 时使用同一个测试可执行文件时，这非常有用。注意，此变量仅在 CTest 中可用。

由于 CMake 解析规则的限制，任何名称中带有方括号的测试都将从存储在此变量的测试列表中省略。不过，此类测试仍将由 ctest 正常定义和执行。

DISCOVERY_TIMEOUT num

在 3.10.3 版本中添加。

指定 CMake 等待测试枚举可用测试的时长（秒）。如果测试耗时超过此时间，发现（以及您的构建）将失败。大多数测试可执行文件会非常快地枚举其测试，但在某些特殊情况下，测试可能需要更长的超时时间。默认值为 5。另请参阅 execute_process() 的 TIMEOUT 选项。

注意

在 CMake 3.10.1 和 3.10.2 版本中，此选项称为 TIMEOUT。这与 TIMEOUT 测试属性冲突，后者是通常会使用 PROPERTIES 关键字设置的常见属性之一，通常会导致合规但意料之外的行为。关键字在 CMake 3.10.3 中更改为 DISCOVERY_TIMEOUT 以解决此问题。3.10.1 和 3.10.2 版本中 TIMEOUT 关键字的歧义行为未被保留。

XML_OUTPUT_DIR dir

在 3.18 版本中新增。

如果指定，该参数将与 --gtest_output=xml: 一起传递给测试可执行文件。实际的文件名与测试目标相同，包括前缀和后缀。应使用此选项代替 EXTRA_ARGS --gtest_output=xml，以避免在使用并行测试执行时写入 XML 结果输出时的竞态条件。

DISCOVERY_MODE

在 3.18 版本中新增。

提供对 gtest_discover_tests() 何时执行测试发现的更好控制。默认情况下，POST_BUILD 设置一个构建后命令，在构建时执行测试发现。在某些场景（如交叉编译）中，这种 POST_BUILD 行为是不希望出现的。相比之下，PRE_TEST 将测试发现延迟到测试执行之前。这样，测试发现发生在目标环境中，测试更有机会找到合适的运行时依赖项。

如果调用 gtest_discover_tests() 时未传递此项，DISCOVERY_MODE 将默认为 CMAKE_GTEST_DISCOVER_TESTS_DISCOVERY_MODE 变量的值。这提供了一种全局选择首选测试发现行为的机制，而无需修改每个调用点。

DISCOVERY_EXTRA_ARGS args...

在版本 3.31 中添加。

传递给发现命令行的任何额外参数。

3.29 版本添加: TEST_LAUNCHER 目标属性在测试发现和测试执行期间会被采纳。

3.31 版本变更: TEST_LAUNCHER 和 CROSSCOMPILING_EMULATOR 目标属性中的空值会被保留，详见策略 CMP0178。
