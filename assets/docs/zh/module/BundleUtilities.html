BundleUtilities¶

此模块提供了用于使用 CMake 组装独立、捆绑式（bundle-style）应用程序的实用命令，例如 macOS（苹果桌面系统） 的 .app 捆绑包，或其他操作系统上类似的基于目录的应用程序捆绑包。

在 CMake 安装中通过以下方式加载此模块：

include(BundleUtilities)

代码说明

  • include(BundleUtilities)：调用 `include()` 命令，加载并执行另一个 CMake 脚本。

注意

请勿在配置时（即在 CMakeLists.txt 中）使用此模块。相反，应在 install(CODE) 或 install(SCRIPT) 中引入并调用其命令。

fixup_bundle¶

准备一个捆绑包以进行分发，方法是修复其内部依赖项

fixup_bundle(<app> <libs> <dirs> [IGNORE_ITEM <files>...])

代码说明

  • fixup_bundle(<app> <libs> <dirs> [IGNORE_ITEM <files>...])：调用 `fixup_bundle()` 命令，具体行为由这一行传入的参数决定。

此命令会对 <app> 捆绑包进行就地修改，使其成为独立且可移植的，以便可以拖放到另一台机器上运行（前提是所有系统库均兼容）。

此命令会收集捆绑包内所有可执行文件和库的依赖项（键）。对于每个依赖项，它会将所需文件复制到捆绑包中，并根据它们各自的先决条件进行调整。完成后，它会清除收集到的键，并调用 verify_app() 命令以确保最终的捆绑包确实是独立的。

参数为

<app>

要修复的捆绑包路径。这可以是一个 .app 目录或指向可执行文件的直接路径。

<libs>

必须修复但无法通过 otool 输出分析自动确定的库列表（例如 插件）。如果将插件作为此参数传递给此命令，则应在调用此命令之前将它们安装或复制到捆绑包中。

<dirs>

可能找到库的路径列表。当给定没有路径信息的目标时，首先搜索这些路径。然后还会搜索标准系统位置：PATH、框架位置、/usr/lib 等。

IGNORE_ITEM <files>...

3.6 版本新增。

可选的要忽略的文件名列表（例如 IGNORE_ITEM "vcredist_x86.exe;vcredist_x64.exe"）。

copy_and_fixup_bundle¶

复制捆绑包并就地修复新的拷贝

copy_and_fixup_bundle(<src> <dst> <libs> <dirs>)

代码说明

  • copy_and_fixup_bundle(<src> <dst> <libs> <dirs>)：调用 `copy_and_fixup_bundle()` 命令，具体行为由这一行传入的参数决定。

此命令在位置 <dst> 创建捆绑包 <src> 的副本，然后在 <dst> 处就地修复该新拷贝。

参数为

<src>

正在复制的捆绑包的目录。

<dst>

捆绑包拷贝的目标目录。

<libs>

必须修复但无法通过 otool 输出分析自动确定的库列表（例如 插件）。如果将插件作为此参数传递给此命令，则应在调用此命令之前将它们安装或复制到捆绑包中。

<dirs>

可能找到库的路径列表。当给定没有路径信息的目标时，首先搜索这些路径。然后还会搜索标准系统位置：PATH、框架位置、/usr/lib 等。

verify_app¶

通过运行分析工具验证应用程序捆绑包是否有效

verify_app(<app> [IGNORE_ITEM <files>...])

代码说明

  • verify_app(<app> [IGNORE_ITEM <files>...])：调用 `verify_app()` 命令，具体行为由这一行传入的参数决定。

如果应用程序未能通过验证，则会发出 message(FATAL_ERROR)，从而停止安装过程。

参数为

<app>

IGNORE_ITEM <files>...

3.6 版本新增。

可选的要忽略的文件名列表（例如 IGNORE_ITEM "vcredist_x86.exe;vcredist_x64.exe"）。

get_bundle_main_executable¶

检索给定应用程序捆绑包中的主可执行文件

get_bundle_main_executable(<bundle> <result-var>)

代码说明

  • get_bundle_main_executable(<bundle> <result-var>)：调用 `get_bundle_main_executable()` 命令，具体行为由这一行传入的参数决定。

结果存储在 <result-var> 变量中，并将包含捆绑包主可执行文件的完整路径名，如果无法确定，则为以 error: 为前缀的字符串。

get_dotapp_dir¶

定位给定可执行文件的封闭 .app 目录

get_dotapp_dir(<exe> <dotapp-dir-var>)

代码说明

  • get_dotapp_dir(<exe> <dotapp-dir-var>)：调用 `get_dotapp_dir()` 命令，具体行为由这一行传入的参数决定。

此命令在给定可执行文件的完整路径时，检索名称以 .app 结尾的最近父目录，并将其存储到 <dotapp-dir-var> 变量中。如果没有这样的父目录，则它仅检索包含可执行文件的目录。

检索到的目录可能存在，也可能不存在。

get_bundle_and_executable¶

接受 .app 目录名或嵌套在 .app 目录中的可执行文件名，并检索 .app 目录的路径及其主可执行文件的路径

get_bundle_and_executable(<app> <bundle-var> <executable-var> <valid-var>)

代码说明

  • get_bundle_and_executable(<app> <bundle-var> <executable-var> <valid-var>)：调用 `get_bundle_and_executable()` 命令，具体行为由这一行传入的参数决定。

参数为

<app>

正在处理的应用程序名称。

<bundle-var>

用于存储生成的 .app 目录路径的变量名。如果发生任何错误，此变量将包含以字符串 error: 为前缀的错误消息。

<executable-var>

用于存储生成的对应主可执行文件的变量名。如果发生任何错误，此变量将包含以字符串 error: 为前缀的错误消息。

<valid-var>

用于存储布尔结果的变量名，表示此命令是否成功。

get_item_key¶

为给定项生成唯一键

get_item_key(<item> <key-var>)

代码说明

  • get_item_key(<item> <key-var>)：调用 `get_item_key()` 命令，具体行为由这一行传入的参数决定。

给定 <item> 文件名，此命令生成一个 <key-var> 键，考虑到需要复制或修复以使捆绑包独立的一组库，该键应该是唯一的。本质上，这是包含扩展名的文件名，并将 . 替换为 _。

此键用作 CMake 变量的前缀，以便一组变量可以基于其键与给定项关联。

clear_bundle_keys¶

清除所有与键关联的变量

clear_bundle_keys(<keys-var>)

代码说明

  • clear_bundle_keys(<keys-var>)：调用 `clear_bundle_keys()` 命令，具体行为由这一行传入的参数决定。

此命令遍历 <keys-var> 键列表，清除与每个键关联的所有变量。循环结束后，它会清除键列表本身。当处理完键列表后，应在 get_bundle_keys() 命令之后调用此命令。

set_bundle_key_values¶

将键添加到键列表中

set_bundle_key_values(
  <keys-var>
  <context>
  <item>
  <exepath>
  <dirs>
  <copyflag>
  [<rpaths>]
)

代码说明

  • set_bundle_key_values( <keys-var> <context> <item> <exepath> <dirs> <copyflag> [<rpaths>] )：调用 `set_bundle_key_values()` 命令，具体行为由这一行传入的参数决定。

此命令将 <keys-var> 键添加到给定项的列表中（如果需要）。如果添加，还会设置与该键关联的所有变量。

参数为

<keys-var>

用于持有要添加到给定项列表中的键名的变量名。

<context>

用于 Apple 操作系统上 @loader_path 替换的顶层加载路径。解析项时，@loader_path 引用将相对于给定上下文值（大概是另一个库）的目录进行解析。

<item>

要为其添加键的项。

<exepath>

用于 Apple 操作系统上 @executable_path 替换的顶层可执行文件路径。

<dirs>

可能找到库的路径列表。当给定没有路径信息的目标时，首先搜索这些路径。然后还会搜索标准系统位置：PATH、框架位置、/usr/lib 等。

<copyflag>

如果设置为 1，则将保留库的符号链接结构。

<rpaths>

可执行文件或库的可选运行时搜索路径，有助于查找文件。

get_bundle_keys¶

获取捆绑包键

get_bundle_keys(<app> <libs> <dirs> <keys-var> [IGNORE_ITEM <files>...])

代码说明

  • get_bundle_keys(<app> <libs> <dirs> <keys-var> [IGNORE_ITEM <files>...])：调用 `get_bundle_keys()` 命令，具体行为由这一行传入的参数决定。

此命令遍历 <app> 捆绑包（以及作为额外 <libs> 给出的项）内的所有可执行文件和库文件，并积累代表它们的键列表。它设置与每个键关联的值，以便可以遍历所有这些值，并将先决条件库复制到捆绑包中，然后执行适当的 install_name_tool 修复。

参数为

<app>

要修复的捆绑包路径。这可以是一个 .app 目录或指向可执行文件的直接路径。

<libs>

必须修复但无法通过 otool 输出分析自动确定的库列表（例如 插件）。如果将插件作为此参数传递给此命令，则应在调用此命令之前将它们安装或复制到捆绑包中。

<dirs>

可能找到库的路径列表。当给定没有路径信息的目标时，首先搜索这些路径。然后还会搜索标准系统位置：PATH、框架位置、/usr/lib 等。

<keys-var>

用于持有代表捆绑包内所有可执行文件和库文件的键列表的变量名。

IGNORE_ITEM <files>...

3.6 版本新增。

可选的要忽略的文件名列表（例如 IGNORE_ITEM "vcredist_x86.exe;vcredist_x64.exe"）。

copy_resolved_framework_into_bundle¶

如有必要，将已解析的框架复制到捆绑包中

copy_resolved_framework_into_bundle(<resolved-item> <resolved-embedded-item>)

代码说明

  • copy_resolved_framework_into_bundle(<resolved-item> <resolved-embedded-item>)：调用 `copy_resolved_framework_into_bundle()` 命令，具体行为由这一行传入的参数决定。

如果 <resolved-item> 与 <resolved-embedded-item> “相同”，则无需复制。

在调用此命令之前，可以设置以下变量

BU_COPY_FULL_FRAMEWORK_CONTENTS

默认情况下，未设置此变量。如果应将完整框架嵌入到捆绑包中，请在调用 fixup_bundle() 命令之前将此变量设置为布尔真值。默认情况下，此命令会复制框架 dylib 本身以及框架的 Resources 目录。

fixup_bundle_item¶

修复捆绑包项

fixup_bundle_item(<resolved-embedded-item> <exepath> <dirs>)

代码说明

  • fixup_bundle_item(<resolved-embedded-item> <exepath> <dirs>)：调用 `fixup_bundle_item()` 命令，具体行为由这一行传入的参数决定。

此命令获取 <resolved-embedded-item> 的直接/非系统先决条件，并为每个先决条件将其引用方式更改为该先决条件的 _EMBEDDED_ITEM 键控变量的值。最有可能更改为 @executable_path 样式的引用。

此命令要求 <resolved-embedded-item> 已经在捆绑包内部。换句话说，如果插件作为其 <libs> 参数传递给 fixup_bundle() 命令，则应在调用 fixup_bundle() 命令之前将它们安装或复制到捆绑包中。

此外，它会将正在修复的项的 ID 更改为其自己的 _EMBEDDED_ITEM 值。

更改会累积在局部变量中，并在本命令结束时对 install_name_tool 命令行工具进行一次调用，同时应用所有更改。

参数为

<resolved-embedded-item>

要修复的捆绑包项。

<exepath>

用于 Apple 操作系统上 @executable_path 替换的顶层可执行文件路径。

<dirs>

可能找到库的路径列表。当给定没有路径信息的目标时，首先搜索这些路径。然后还会搜索标准系统位置：PATH、框架位置、/usr/lib 等。

在调用此命令之前，可以设置以下变量

BU_CHMOD_BUNDLE_ITEMS

如果此变量设置为布尔真值，则在 install_name_tool 工具尝试更改它们之前，捆绑包项将被标记为可写。

  • verify_bundle_prerequisites( <bundle> <result-var> <info-var> [IGNORE_ITEM <files>...] )：调用 `verify_bundle_prerequisites()` 命令，具体行为由这一行传入的参数决定。

verify_bundle_symlinks¶

verify_bundle_symlinks(<bundle> <result-var> <info-var>)

代码说明

  • verify_bundle_symlinks(<bundle> <result-var> <info-var>)：调用 `verify_bundle_symlinks()` 命令，具体行为由这一行传入的参数决定。

示例¶

在安装阶段执行的安装代码中使用此模块

CMakeLists.txt¶
# ...

install(CODE "
  include(BundleUtilities)
  set(BU_CHMOD_BUNDLE_ITEMS TRUE)
  fixup_bundle(
    \"${fixup_exe}\"
    \"${plugins}\"
    \"${bin_dir};${library_dir};${binary_dir}\"
  )
")

代码说明

  • # ... install(CODE " include(BundleUtilities) set(BU_CHMOD_BUNDLE_ITEMS TRUE) fixup_bundle( \"${fixup_exe}\" \"${plugins}\" \"${bin_dir};${library_dir};${binary_dir}\" ) ")：这是注释，用来说明：... install(CODE " include(BundleUtilities) set(BU_CHMOD_BUNDLE_ITEMS TRUE) fixup_bundle( \"${fixup_exe}\" \"${plugins}\" \"${bin_dir};${library_dir};${binary_dir}\" ) ")。
