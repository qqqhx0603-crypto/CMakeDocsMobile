WriteCompilerDetectionHeader¶

自 3.20 版本起弃用：仅当策略 CMP0120 未设置为 NEW 时，此模块才可用。请勿在新代码中使用它。

版本 3.1 中新增。

该模块提供了一个用于生成包含预处理器宏的头文件的命令。

在 CMake 项目中加载此模块，使用

include(WriteCompilerDetectionHeader)

命令¶

此模块提供以下命令

write_compiler_detection_header¶

此函数可用于生成一个适用于预处理器包含的文件，其中包含可在源代码中使用的宏。

write_compiler_detection_header(
          FILE <file>
          PREFIX <prefix>
          [OUTPUT_FILES_VAR <output_files_var> OUTPUT_DIR <output_dir>]
          COMPILERS <compiler> [...]
          FEATURES <feature> [...]
          [BARE_FEATURES <feature> [...]]
          [VERSION <version>]
          [PROLOG <prolog>]
          [EPILOG <epilog>]
          [ALLOW_UNKNOWN_COMPILERS]
          [ALLOW_UNKNOWN_COMPILER_VERSIONS]
)

这将生成文件 <file>，其中的所有宏均带有前缀 <prefix>。

默认情况下，所有内容直接写入 <file>。可以指定 OUTPUT_FILES_VAR 以便将特定于编译器的内容写入单独的文件。这些单独的文件随后可在 <output_files_var> 中找到，并可供调用者用于安装等目的。 OUTPUT_DIR 指定了从主文件 <file> 到这些特定于编译器的文件的相对路径。例如：

write_compiler_detection_header(
  FILE climbingstats_compiler_detection.h
  PREFIX ClimbingStats
  OUTPUT_FILES_VAR support_files
  OUTPUT_DIR compilers
  COMPILERS GNU Clang MSVC Intel
  FEATURES cxx_variadic_templates
)
install(FILES
  ${CMAKE_CURRENT_BINARY_DIR}/climbingstats_compiler_detection.h
  DESTINATION include
)
install(FILES
  ${support_files}
  DESTINATION include/compilers
)

VERSION 可用于指定要生成的 API 版本。CMake 的未来版本可能会引入替代 API。给定 API 的选择依据是任何大于或等于引入该 API 的 CMake 版本，且小于引入其后续 API 的 CMake 版本的 <version> 值。如果未显式指定版本，则使用 CMAKE_MINIMUM_REQUIRED_VERSION 变量的值。（截至 CMake 4.3.0 版本，仅存在一个 API 版本。）

PROLOG 可指定为要写入头文件开头的文本内容。EPILOG 可指定为要写入头文件末尾的文本内容。

必须至少列出一个 <compiler> 和一个 <feature>。CMake 已知但未指定的编译器会被检测到，并为其生成预处理器 #error。对于 CMake 已知的每个编译器，都会生成一个匹配 <PREFIX>_COMPILER_IS_<compiler> 的预处理器宏，其值包含 0 或 1。

可能的编译器标识符记录在 CMAKE_<LANG>_COMPILER_ID 变量中。此 CMake 版本中可用的特性列在 CMAKE_C_KNOWN_FEATURES 和 CMAKE_CXX_KNOWN_FEATURES 全局属性中。有关编译特性的信息，请参阅 cmake-compile-features(7) 手册。

3.2 版本新增：增加了对 MSVC 和 AppleClang 编译器的支持。

3.6 版本新增：增加了对 Intel 编译器的支持。

3.8 版本变更：如果请求了 {c,cxx}_std_* 元特性，它们将被忽略。

3.8 版本新增：ALLOW_UNKNOWN_COMPILERS 和 ALLOW_UNKNOWN_COMPILER_VERSIONS 会使该模块生成一些条件，将未知编译器视为仅缺乏所有特性。如果不使用这些选项，默认行为是为未知编译器和版本生成 #error。

3.12 版本新增：BARE_FEATURES 将使用语言标准较新版本中的名称来定义兼容性宏，以便代码可以无条件地使用新特性名称。

特性测试宏¶

对于每个编译器，都会生成一个匹配 <PREFIX>_COMPILER_IS_<compiler> 的预处理器宏，其内容根据所使用的编译器为 0 或 1。如果定义了编译器版本组件，则会生成匹配 <PREFIX>_COMPILER_VERSION_MAJOR、<PREFIX>_COMPILER_VERSION_MINOR 和 <PREFIX>_COMPILER_VERSION_PATCH 的预处理器宏，其中包含对应编译器版本组件的十进制值。

会根据编译器版本生成一个预处理器测试，以指示每个特性是否已启用。生成的预处理器宏匹配 <PREFIX>_COMPILER_<FEATURE>（其中 <FEATURE> 是大写的 <feature> 名称），其值根据所使用的编译器是否支持该特性而包含 0 或 1。

write_compiler_detection_header(
  FILE climbingstats_compiler_detection.h
  PREFIX ClimbingStats
  COMPILERS GNU Clang AppleClang MSVC Intel
  FEATURES cxx_variadic_templates
)

#if ClimbingStats_COMPILER_CXX_VARIADIC_TEMPLATES
template<typename... T>
void someInterface(T t...) { /* ... */ }
#else
// Compatibility versions
template<typename T1>
void someInterface(T1 t1) { /* ... */ }
template<typename T1, typename T2>
void someInterface(T1 t1, T2 t2) { /* ... */ }
template<typename T1, typename T2, typename T3>
void someInterface(T1 t1, T2 t2, T3 t3) { /* ... */ }
#endif

符号宏¶

针对某些特定特性创建了一些额外的符号定义，用作可以有条件地定义为空的符号。

class MyClass ClimbingStats_FINAL
{
    ClimbingStats_CONSTEXPR int someInterface() { return 42; }
};

如果编译器（及其标志）支持 cxx_final 特性，则 ClimbingStats_FINAL 宏将扩展为 final；如果支持 cxx_constexpr，则 ClimbingStats_CONSTEXPR 宏将扩展为 constexpr。

如果将 BARE_FEATURES cxx_final 作为参数，则 final 关键字也将为旧编译器定义。

兼容性实现宏¶

如果编译器不支持某项特性，则适合将某些特性包装在带有向后兼容性实现的宏中。

当编译器不提供 cxx_static_assert 特性时，可通过函数式宏 <PREFIX>_STATIC_ASSERT(COND) 和 <PREFIX>_STATIC_ASSERT_MSG(COND, MSG) 使用兼容性实现。这些宏在编译器支持该特性时扩展为 static_assert，否则扩展为兼容性实现。在第一种形式中，条件被字符串化为 static_assert 的消息字段。在第二种形式中，消息 MSG 被传递给 static_assert 的消息字段，或者在使用向后兼容性实现时被忽略。

cxx_attribute_deprecated 特性提供了一个宏定义 <PREFIX>_DEPRECATED，它扩展为标准 [[deprecated]] 属性或 GNU 编译器使用的特定于编译器的修饰符（如 __attribute__((__deprecated__))）。

cxx_alignas 特性提供了一个宏定义 <PREFIX>_ALIGNAS，它扩展为标准 alignas 修饰符或 GNU 编译器使用的特定于编译器的修饰符（如 __attribute__ ((__aligned__))）。

cxx_alignof 特性提供了一个宏定义 <PREFIX>_ALIGNOF，它扩展为标准 alignof 修饰符或 GNU 编译器使用的特定于编译器的修饰符（如 __alignof__）。

此类弃用宏产生的一个用例是弃用整个库。在这种情况下，库中的所有公共 API 都可以用 <PREFIX>_DEPRECATED 宏进行修饰。这会导致在构建库本身时产生大量的构建输出噪音，因此在构建被弃用的库时，可以将该宏定义为空。

add_library(compat_support ${srcs})
target_compile_definitions(compat_support
  PRIVATE
    CompatSupport_DEPRECATED=
)

示例用法¶

注意

本节从 cmake-compile-features(7) 手册迁移而来，因为它依赖于被策略 CMP0120 移除的 WriteCompilerDetectionHeader 模块。

如果可用，可能优先使用编译特性，而不必将其设为硬性要求。例如，一个库可以根据是否支持 cxx_variadic_templates 特性来提供不同的实现。

#if Foo_COMPILER_CXX_VARIADIC_TEMPLATES
template<int I, int... Is>
struct Interface;

template<int I>
struct Interface<I>
{
  static int accumulate()
  {
    return I;
  }
};

template<int I, int... Is>
struct Interface
{
  static int accumulate()
  {
    return I + Interface<Is...>::accumulate();
  }
};
#else
template<int I1, int I2 = 0, int I3 = 0, int I4 = 0>
struct Interface
{
  static int accumulate() { return I1 + I2 + I3 + I4; }
};
#endif

这样的接口依赖于使用正确的编译器特性预处理器定义。CMake 可以使用 WriteCompilerDetectionHeader 模块生成包含此类定义的头文件。该模块包含 write_compiler_detection_header 函数，它接受用于控制生成的头文件内容的参数。

write_compiler_detection_header(
  FILE "${CMAKE_CURRENT_BINARY_DIR}/foo_compiler_detection.h"
  PREFIX Foo
  COMPILERS GNU
  FEATURES
    cxx_variadic_templates
)

这样的头文件可以在项目的源代码内部使用，也可以被安装并在库代码的接口中使用。

对于 FEATURES 中列出的每个特性，都会在头文件中创建一个预处理器定义，并将其定义为 1 或 0。

此外，某些特性（如 cxx_final 和 cxx_override）需要额外的定义。与其在 #ifdef 代码中使用，不如用一个符号来抽象 final 关键字，该符号被定义为 final、特定的编译器等效项或为空。这样，C++ 代码就可以无条件地使用该符号编写，而由编译器支持情况决定它被扩展为什么。

struct Interface {
  virtual void Execute() = 0;
};

struct Concrete Foo_FINAL {
  void Execute() Foo_OVERRIDE;
};

在这种情况下，如果编译器支持该关键字，Foo_FINAL 将扩展为 final，否则扩展为空。

在此用例中，项目代码可能希望在编译器支持的情况下启用特定的语言标准。CXX_STANDARD 目标属性可以设置为特定目标的所需语言标准，CMAKE_CXX_STANDARD 变量则可用于影响所有后续目标。

write_compiler_detection_header(
  FILE "${CMAKE_CURRENT_BINARY_DIR}/foo_compiler_detection.h"
  PREFIX Foo
  COMPILERS GNU
  FEATURES
    cxx_final cxx_override
)

# Includes foo_compiler_detection.h and uses the Foo_FINAL symbol
# which will expand to 'final' if the compiler supports the requested
# CXX_STANDARD.
add_library(foo foo.cpp)
set_property(TARGET foo PROPERTY CXX_STANDARD 11)

# Includes foo_compiler_detection.h and uses the Foo_FINAL symbol
# which will expand to 'final' if the compiler supports the feature,
# even though CXX_STANDARD is not set explicitly.  The requirement of
# cxx_constexpr causes CMake to set CXX_STANDARD internally, which
# affects the compile flags.
add_library(foo_impl foo_impl.cpp)
target_compile_features(foo_impl PRIVATE cxx_constexpr)

write_compiler_detection_header 函数还会为其他具有标准等价物的特性创建兼容性代码。例如，cxx_static_assert 特性是使用模板进行模拟，并通过 <PREFIX>_STATIC_ASSERT 和 <PREFIX>_STATIC_ASSERT_MSG 函数宏进行抽象的。
