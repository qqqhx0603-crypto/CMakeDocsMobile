INTERFACE_LINK_LIBRARIES_DIRECT¶

在 3.24 版本中添加。

库的消费者应将其视为直接链接依赖项的库列表。

此目标属性可以设置为在依赖目标的最终直接链接依赖项集合中包含某些项。请参阅 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 目标属性以排除某些项。

依赖目标的直接链接依赖项的初始集合由其 LINK_LIBRARIES 目标属性指定。间接链接依赖项由直接链接依赖项的 INTERFACE_LINK_LIBRARIES 属性的传递闭包指定。任何链接依赖项都可以使用 INTERFACE_LINK_LIBRARIES_DIRECT 目标属性指定额外的直接链接依赖项。随后，直接链接依赖项的集合会被过滤，以排除任何依赖项的 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 目标属性所命名的项。

INTERFACE_LINK_LIBRARIES_DIRECT 的值可以使用 生成器表达式。

注意

INTERFACE_LINK_LIBRARIES_DIRECT 目标属性旨在用于高级用例，例如将静态插件注入到消费方的可执行文件中。它不应被用作组织对 target_link_libraries() 的常规调用的替代品。

作为使用要求的直接链接依赖项¶

INTERFACE_LINK_LIBRARIES_DIRECT 和 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 目标属性是 使用要求。它们的效果会传递性地传播到依赖目标，因此可能会影响依赖库链中每个目标的直接链接依赖项。每当某个库目标 X 链接到另一个库目标 Y，且该目标的直接或传递性使用要求中包含 INTERFACE_LINK_LIBRARIES_DIRECT 或 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 时，这些属性就可能影响 X 的直接链接依赖项列表。

  • 如果 X 是共享库或可执行文件，则会链接其依赖项。它们还会影响编译 X 的源文件时所使用的使用要求。

  • 如果 X 是静态库或对象库，则它实际上不会进行链接，因此其依赖项最多只会影响编译 X 的源文件时所使用的使用要求。

这些属性还可能影响 X 的从属目标的直接链接依赖项列表。

  • 如果 X 公开（publicly）链接 Y

    target_link_libraries(X PUBLIC Y)

    代码说明

    • target_link_libraries(X PUBLIC Y)：调用 `target_link_libraries()` 命令，为目标链接依赖库。

    那么 Y 会被放置在 X 的 INTERFACE_LINK_LIBRARIES 中，因此 Y 的使用要求（包括 INTERFACE_LINK_LIBRARIES_DIRECT、INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE，以及它们所添加的直接链接依赖项所声明的使用要求）都会传播到 X 的从属目标。

  • 如果 X 是静态库或对象库，且私有（privately）链接 Y

    target_link_libraries(X PRIVATE Y)

    代码说明

    • target_link_libraries(X PRIVATE Y)：调用 `target_link_libraries()` 命令，为目标链接依赖库。

    那么 $<LINK_ONLY:Y> 会被放置在 X 的 INTERFACE_LINK_LIBRARIES 中。Y 的链接要求（包括 INTERFACE_LINK_LIBRARIES_DIRECT、INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 以及它们所添加的直接链接依赖项所声明的传递性链接依赖项）会传播到 X 的从属目标。然而，Y 的非链接使用要求会被 LINK_ONLY 生成器表达式阻塞，且不会传播到 X 的从属目标。

  • 如果 X 是共享库或可执行文件，且私有（privately）链接 Y

    target_link_libraries(X PRIVATE Y)

    代码说明

    • target_link_libraries(X PRIVATE Y)：调用 `target_link_libraries()` 命令，为目标链接依赖库。

    那么 Y 不会被放置在 X 的 INTERFACE_LINK_LIBRARIES 中，因此 Y 的使用要求（即使是 INTERFACE_LINK_LIBRARIES_DIRECT 和 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE）也不会传播到 X 的从属目标。

  • 在所有情况下，X 的 INTERFACE_LINK_LIBRARIES 的内容都不会受到 Y 的 INTERFACE_LINK_LIBRARIES_DIRECT 或 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 的影响。

可以使用 TARGET_PROPERTY 生成器表达式将 INTERFACE_LINK_LIBRARIES_DIRECT 和 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 的影响限制在依赖目标的子集内。例如，要将影响限制在可执行目标上，请使用以下形式的条目

"$<$<STREQUAL:$<TARGET_PROPERTY:TYPE>,EXECUTABLE>:...>"

代码说明

  • "$<$<STREQUAL:$<TARGET_PROPERTY:TYPE>,EXECUTABLE>:...>"：这一行继续补充当前命令或表达式的内容。

同样，要将影响限制在特定目标上，请使用以下形式的条目

"$<$<BOOL:$<TARGET_PROPERTY:USE_IT>>:...>"

代码说明

  • "$<$<BOOL:$<TARGET_PROPERTY:USE_IT>>:...>"：这一行继续补充当前命令或表达式的内容。

此条目将仅影响将其 USE_IT 目标属性设置为真值的目标。

直接链接依赖项排序¶

目标的直接链接依赖项列表是根据其 LINK_LIBRARIES 目标属性中的初始有序列表计算出来的。对于每个项，会从其直接和传递性的 INTERFACE_LINK_LIBRARIES_DIRECT 使用要求中发现额外的直接链接依赖项。每个发现的项都会被插入到指定它的项之前。然而，一个发现的项最多只会添加一次，且仅当它未出现在初始列表中的任何位置时才会添加。这使得 LINK_LIBRARIES 能够控制其明确指定的直接链接依赖项的排序。

一旦收集完所有直接链接依赖项，所有其 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 使用要求中命名的项都会从最终列表中移除。这不会影响剩余项的顺序。

示例：静态插件¶

考虑一个静态库 Foo，它向消费方的应用程序可执行文件提供一个静态插件 FooPlugin，其中插件的实现依赖于 Foo 和其他内容。在这种情况下，应用程序应该直接链接到 FooPlugin，排在 Foo 之前。然而，应用程序开发者只知道 Foo。我们可以按如下方式表达

# Core library used by other components.
add_library(Core STATIC core.cpp)

# Foo is a static library for use by applications.
# Implementation of Foo depends on Core.
add_library(Foo STATIC foo.cpp foo_plugin_helper.cpp)
target_link_libraries(Foo PRIVATE Core)

# Extra parts of Foo for use by its static plugins.
# Implementation of Foo's extra parts depends on both Core and Foo.
add_library(FooExtras STATIC foo_extras.cpp)
target_link_libraries(FooExtras PRIVATE Core Foo)

# The Foo library has an associated static plugin
# that should be linked into the final executable.
# Implementation of the plugin depends on Core, Foo, and FooExtras.
add_library(FooPlugin STATIC foo_plugin.cpp)
target_link_libraries(FooPlugin PRIVATE Core Foo FooExtras)

# An app that links Foo should link Foo's plugin directly.
set_property(TARGET Foo PROPERTY INTERFACE_LINK_LIBRARIES_DIRECT FooPlugin)

# An app does not need to link Foo directly because the plugin links it.
set_property(TARGET Foo PROPERTY INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE Foo)

应用程序 app 只需指定它链接到 Foo

add_executable(app main.cpp)
target_link_libraries(app PRIVATE Foo)

代码说明

  • add_executable(app main.cpp) target_link_libraries(app PRIVATE Foo)：调用 `add_executable()` 命令，定义一个可执行目标。

Foo 上的 INTERFACE_LINK_LIBRARIES_DIRECT 目标属性告诉 CMake 假定 app 也直接链接到 FooPlugin。Foo 上的 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 目标属性告诉 CMake 假定 app 没有直接链接到 Foo。相反，Foo 将作为 FooPlugin 的依赖项进行链接。app 的最终链接行将按以下顺序链接库

  • FooPlugin 作为 app 的直接链接依赖项（通过 Foo 的使用要求）。

  • FooExtras 作为 FooPlugin 的依赖项。

  • Foo 作为 FooPlugin 和 FooExtras 的依赖项。

  • Core 作为 FooPlugin、FooExtras 和 Foo 的依赖项。

注意，如果没有 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 目标属性，Foo 将被链接两次：一次作为 app 的直接依赖项，一次作为 FooPlugin 的依赖项。

示例：选择性加入的静态插件¶

在上面的 示例：静态插件 中，app 可执行文件指定它直接链接到 Foo。在实际应用程序中，可能存在一个中间库

add_library(app_impl STATIC app_impl.cpp)
target_link_libraries(app_impl PRIVATE Foo)

add_executable(app main.cpp)
target_link_libraries(app PRIVATE app_impl)

代码说明

  • add_library(app_impl STATIC app_impl.cpp) target_link_libraries(app_impl PRIVATE Foo) add_executable(app main.cpp) target_link_libraries(app PRIVATE app_impl)：调用 `add_library()` 命令，定义一个库目标。

在这种情况下，我们不希望 Foo 的 INTERFACE_LINK_LIBRARIES_DIRECT 和 INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE 目标属性影响 app_impl 的直接依赖项。为了避免这种情况，我们可以修改属性值以使其效果为“选择性加入”

# An app that links Foo should link Foo's plugin directly.
set_property(TARGET Foo PROPERTY INTERFACE_LINK_LIBRARIES_DIRECT
  "$<$<BOOL:$<TARGET_PROPERTY:FOO_STATIC_PLUGINS>>:FooPlugin>"
)

# An app does not need to link Foo directly because the plugin links it.
set_property(TARGET Foo PROPERTY INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE
  "$<$<BOOL:$<TARGET_PROPERTY:FOO_STATIC_PLUGINS>>:Foo>"
)

代码说明

  • # An app that links Foo should link Foo's plugin directly. set_property(TARGET Foo PROPERTY INTERFACE_LINK_LIBRARIES_DIRECT "$<$<BOOL:$<TARGET_PROPERTY:FOO_STATIC_PLUGINS>>:FooPlugin>" ) # An app does not need to link Foo directly because the plugin links it. set_property(TARGET Foo PROPERTY INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE "$<$<BOOL:$<TARGET_PROPERTY:FOO_STATIC_PLUGINS>>:Foo>" )：这是注释，用来说明：An app that links Foo should link Foo's plugin directly. set_property(TARGET Foo PROPERTY INTERFACE_LINK_LIBRARIES_DIRECT "$<$<BOOL:$<TARGET_PROPERTY:FOO_STATIC_PLUGINS>>:FooPlugin>" ) # An app does not need to link Foo directly because the plugin links it. set_property(TARGET Foo PROPERTY INTERFACE_LINK_LIBRARIES_DIRECT_EXCLUDE "$<$<BOOL:$<TARGET_PROPERTY:FOO_STATIC_PLUGINS>>:Foo>" )。

现在，app 可执行文件可以选择加入以获取 Foo 的插件

set_property(TARGET app PROPERTY FOO_STATIC_PLUGINS 1)

代码说明

  • set_property(TARGET app PROPERTY FOO_STATIC_PLUGINS 1)：调用 `set_property()` 命令，设置目标、目录、源文件等对象的属性。

app 的最终链接行现在将按以下顺序链接库

  • FooPlugin 作为 app 的直接链接依赖项（通过 Foo 的使用要求）。

  • app_impl 作为 app 的直接链接依赖项。

  • FooExtras 作为 FooPlugin 的依赖项。

  • Foo 作为 app_impl、FooPlugin 和 FooExtras 的依赖项。

  • Core 作为 FooPlugin、FooExtras 和 Foo 的依赖项。
