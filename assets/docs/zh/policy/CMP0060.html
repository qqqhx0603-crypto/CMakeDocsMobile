CMP0060¶

警告

此策略的 OLD 行为已在 CMake 4.0 版本中移除。此策略必须通过调用 cmake_minimum_required() 或 cmake_policy() 来设置为 NEW。

3.3 版本中新增。

即使在隐式目录中也按完整路径链接库。

引入策略 CMP0003 的初衷是，当向 target_link_libraries() 命令提供完整路径时，始终通过完整路径链接库文件。然而，在某些平台（例如 HP-UX）上，编译器前端会为当前架构添加替代的库搜索路径（例如 /usr/lib/<arch> 包含针对当前架构的 /usr/lib 中库的替代版本）。在这些平台上，find_library() 可能会找到类似 /usr/lib/libfoo.so 这样不属于当前架构的库。

在策略 CMP0003 之前，项目在上述情况下仍然可以构建，因为错误的库路径会被转换为链接行上的 -lfoo，链接器会在编译器前端隐式提供的特定架构搜索路径中找到正确的库。当时我们选择通过始终将隐式链接目录中找到的库文件转换为 -lfoo 标志来保持与此类项目的兼容性，以要求链接器搜索它们。这种方法允许现有项目在继续构建的同时，仍能通过完整路径链接到隐式链接目录之外的库（例如构建树中的库）。

CMake 允许项目通过使用 IMPORTED 库目标 并将其 IMPORTED_LOCATION 属性设置为所需的库文件完整路径来覆盖此行为。事实上，许多 Find 模块 正在学习提供 导入目标，而不是仅仅提供列出库文件的传统 Foo_LIBRARIES 变量。然而，这使得 Find 模块找到的库所生成的链接行取决于它是否通过导入目标进行链接，这是不一致的。此外，这种行为一直是困惑的来源，因为库文件生成的链接行取决于其位置。对于试图进行静态链接的项目来说，这也是有问题的，因为诸如 -Wl,-Bstatic -lfoo -Wl,-Bdynamic 之类的标志可能被用来帮助链接器选择 libfoo.a 而不是 libfoo.so，但随后会将动态链接泄漏到后续的库中。（关于该问题的典型解决方案，请参阅 LINK_SEARCH_END_STATIC 目标属性。）

当首次引入隐式链接目录中库的特殊情况时，隐式链接目录列表只是硬编码的（例如 /lib, /usr/lib 等）。从那时起，CMake 已经学会了检测编译器前端使用的隐式链接目录。如有必要，find_library() 命令可以被教导使用这些信息来帮助找到正确架构的库。

由于这些原因，CMake 3.3 及更高版本倾向于放弃这种特殊情况，即使库位于隐式链接目录中，也按完整路径进行链接。策略 CMP0060 为现有项目提供了兼容性。

此策略的 OLD 行为是要求链接器搜索已知位于隐式链接目录中的库。此策略的 NEW 行为是即使库位于隐式链接目录中，也按完整路径链接库。

此策略是在 CMake 3.3 版本中引入的。在 CMake 4.0 版本移除之前，它可以通过 cmake_policy() 或 cmake_minimum_required() 进行设置。如果不进行设置，CMake 默认不发出警告，并使用 OLD 行为。

请参阅 CMAKE_POLICY_WARNING_CMP0060 变量的文档，以控制在 4.0 之前的 CMake 版本中的警告。
