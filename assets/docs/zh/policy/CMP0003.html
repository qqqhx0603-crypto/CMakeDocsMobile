CMP0003¶

警告

此策略的 OLD 行为已在 CMake 4.0 版本中移除。此策略必须通过调用 cmake_minimum_required() 或 cmake_policy() 来设置为 NEW。

通过全路径链接的库不再产生链接器搜索路径。

此策略影响在链接时如何查找全路径“未知”的库，但它的创建是由于 CMake 处理全路径“已知”库的方式发生了变化。考虑以下代码：

target_link_libraries(myexe /path/to/libA.so)

CMake 2.4 及以下版本在链接到全路径已知的库时，通过将它们拆分为链接器搜索路径和库名称这两个独立的部分来实现。上述示例代码可能会产生类似以下内容的结果：

... -L/path/to -lA ...

以便链接到库 A。虽然通过分析可以对多个链接目录进行排序，从而使链接器能在预期的位置找到库 A，但有些情况下这种做法并不奏效。CMake 2.6 及以上版本采用了更可靠的方法，即在大多数情况下直接将库的全路径传递给链接器。示例代码现在会产生类似以下内容的结果：

... /path/to/libA.so ....

遗憾的是，这种变化可能会破坏类似如下的代码：

target_link_libraries(myexe /path/to/libA.so B)

其中 B 旨在找到 /path/to/libB.so。这段代码是错误的，因为用户要求链接器查找库 B，但并未提供链接器搜索路径（这可以通过 link_directories 命令添加）。然而，使用旧的链接实现时，该代码能“意外地”工作，因为为库 A 添加的链接器搜索路径使得库 B 也能被找到。

为了支持依赖于通过链接全路径已知库而添加的链接器搜索路径的项目，此策略的 OLD 行为将继续添加这些链接器搜索路径，即使它们对于其自身的库而言并非必需。当此策略设置为 OLD 时，CMake 将生成如下链接行：

... -L/path/to /path/to/libA.so -lB ...

这将允许库 B 像以前一样被找到。当此策略设置为 NEW 时，CMake 将生成如下链接行：

... /path/to/libA.so -lB ...

这更准确地匹配了项目所指定的意图。

生成链接行时所使用的策略设置，取决于调用 add_executable 或 add_library 命令创建目标时该策略的生效状态。对于上述示例，代码：

cmake_policy(SET CMP0003 OLD) # or cmake_policy(VERSION 2.4)
add_executable(myexe myexe.c)
target_link_libraries(myexe /path/to/libA.so B)

将可以正常工作并抑制此策略的警告。它也可以更新为使用更正后的链接方式：

cmake_policy(SET CMP0003 NEW) # or cmake_policy(VERSION 2.6)
link_directories(/path/to) # needed to find library B
add_executable(myexe myexe.c)
target_link_libraries(myexe /path/to/libA.so B)

更好的做法是，通过全路径指定库 B：

add_executable(myexe myexe.c)
target_link_libraries(myexe /path/to/libA.so /path/to/libB.so)

当链接行上的所有项目都具有已知路径时，CMake 不会检查此策略，因此它不起作用。

请注意，此策略的警告最多只会针对一个目标发出。这避免了在用户只需设置一次策略即可修复所有目标的情况下，却被每个目标的警告消息所淹没。

此策略在 CMake 2.6.0 版本中引入。在 CMake 4.0 版本移除之前，可以通过 cmake_policy() 或 cmake_minimum_required() 进行设置。如果未设置，CMake 会发出警告并使用 OLD 行为。
