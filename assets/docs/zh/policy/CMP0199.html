CMP0199¶

版本 4.2 中添加。

$<CONFIG:cfgs> 不匹配未被选中的映射配置。

从历史上看，当 $<CONFIG:cfgs> 生成器表达式出现在导入目标的属性中时，如果 cfgs 中的任何一个匹配以下 *任意* 一项，它就会匹配（即求值为 1）：

  1. 所消费的导入目标被选中的配置。

  2. 消费目标的配置。

  3. 所消费的导入目标的 MAP_IMPORTED_CONFIG_<CONFIG> 中包含的 *任意* 配置（其中 <CONFIG> 是消费目标的配置），*无论这些配置对于该导入目标是否有效*。

这可能导致本应互斥的表达式被同时求值。如果编译定义的取值旨在由配置决定，这尤其成问题，因为缺乏互斥性可能导致重新定义。

CMake 4.2 及以上版本倾向于 *仅* 考虑消费目标的配置以及（适用时）导入目标被选中的配置；即上述列表中的 (2) 和 (1)。遗憾的是，由于用户同时依赖这两者，该策略无法完全防止多个唯一的 $<CONFIG:cfg> 表达式同时匹配。

此策略为依赖历史行为的项目提供了兼容性。此策略的 OLD 行为是保留上述历史行为。 NEW 行为是仅考虑消费目标和被消费目标的配置。

注意

此策略仅适用于作为非通用包规范（Common Package Specification）包导入的导入目标的使用需求（usage requirements）的一部分而进行求值的生成器表达式。

对于非导入目标，历史行为和当前行为均仅考虑消费目标的配置。（非导入目标的选中配置始终是当前的构建配置，它必然与消费目标的配置相同。）

对于从通用包规范包导入的目标，无论策略设置如何，*仅*考虑被消费的导入目标的配置。

此策略引入于 CMake 4.2 版本。它可以通过 cmake_policy() 或 cmake_minimum_required() 进行设置。如果未设置，CMake 会发出警告，并使用 OLD 行为。

注意

策略的 OLD 行为根据定义已被弃用，并可能在未来版本的 CMake 中被移除。

示例¶

考虑以下导入库：

add_library(test1 INTERFACE IMPORTED)
set_target_properties(test1 PROPERTIES
  IMPORTED_CONFIGURATIONS "DEBUG"
  INTERFACE_COMPILE_DEFINITIONS
    "$<$<CONFIG:debug>:DEBUG>;$<$<CONFIG:release>:RELEASE>"
)

add_library(test2 INTERFACE IMPORTED)
set_target_properties(test2 PROPERTIES
  IMPORTED_CONFIGURATIONS "TEST"
  INTERFACE_COMPILE_DEFINITIONS
    "$<$<CONFIG:debug>:DEBUG>;$<$<CONFIG:example>:EXAMPLE>;$<$<CONFIG:test>:TEST>"
  MAP_IMPORTED_CONFIG_RELEASE "DEBUG;EXAMPLE;TEST"
)

代码说明

  • add_library(test1 INTERFACE IMPORTED) set_target_properties(test1 PROPERTIES IMPORTED_CONFIGURATIONS "DEBUG" INTERFACE_COMPILE_DEFINITIONS "$<$<CONFIG:debug>:DEBUG>;$<$<CONFIG:release>:RELEASE>" ) add_library(test2 INTERFACE IMPORTED) set_target_properties(test2 PROPERTIES IMPORTED_CONFIGURATIONS "TEST" INTERFACE_COMPILE_DEFINITIONS "$<$<CONFIG:debug>:DEBUG>;$<$<CONFIG:example>:EXAMPLE>;$<$<CONFIG:test>:TEST>" MAP_IMPORTED_CONFIG_RELEASE "DEBUG;EXAMPLE;TEST" )：调用 `add_library()` 命令，定义一个库目标。

假设消费项目以 Release 配置构建。 test1 的消费者无论策略设置如何，都会看到 DEBUG 和 RELEASE 同时被定义；$<CONFIG:debug> 求值为 1，因为 test1 的选中配置是 DEBUG；$<CONFIG:release> 求值为 1，因为消费者的配置是 Release（请记住配置匹配是不区分大小写的）。

在 OLD 策略下，test2 的消费者会看到 DEBUG、EXAMPLE 和 TEST 同时被定义；$<CONFIG:debug>、$<CONFIG:example> 和 $<CONFIG:test> 全部求值为 1，因为所有这些配置都出现在 MAP_IMPORTED_CONFIG_RELEASE 中。

在 NEW 策略下，test2 的消费者将仅看到 TEST 被定义。
