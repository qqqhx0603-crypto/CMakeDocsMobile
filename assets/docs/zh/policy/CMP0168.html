CMP0168¶

3.30 版本新增。

FetchContent 模块直接实现步骤，而不是通过子构建（sub-build）实现。

CMake 3.29 及以下版本将 FetchContent 实现为独立的子构建。这需要配置该独立项目并使用构建工具。这种方法在某些生成器和操作系统上可能非常缓慢。CMake 3.30 及以上版本倾向于将下载、更新和补丁步骤直接作为主项目的一部分来实现。

NEW 行为具有以下特点：

  • 不使用子构建。所有操作均直接从主项目的 CMake 配置步骤中实现。在 CMake 脚本模式下运行时，无需提供构建工具。

  • 不支持生成器表达式和 $(SOMEVAR) 形式的 GNU Make（GNU 构建工具） 变量。它们不应在 FetchContent_Declare() 或 FetchContent_Populate() 的任何参数中使用。

  • 所有 LOG_... 和 USES_TERMINAL_... 选项、QUIET 选项以及 FETCHCONTENT_QUIET 变量都会被忽略。FetchContent 的输出始终是主项目配置输出的一部分。这也意味着它现在遵循消息日志级别（请参阅 CMAKE_MESSAGE_LOG_LEVEL 和 --log-level）。默认消息日志级别应该与 OLD 策略设置中使用 QUIET 的效果相当，不同之处在于现在会显示警告信息。

  • PREFIX、TMP_DIR、STAMP_DIR、LOG_DIR 和 DOWNLOAD_DIR 选项及其相关的目录属性将被忽略。FetchContent 模块在内部控制这些位置。

  • cmake --fresh 将删除用于跟踪和填充依赖项的标记文件和脚本文件。这将强制重新执行依赖项的下载、更新和补丁步骤。cmake --fresh 不会影响用于下载的目录，因此 URL 下载方法中先前下载的文件仍然可以重复使用。

OLD 行为具有以下特点：

  • 始终使用子构建来实现下载、更新和补丁步骤。即使在 CMake 脚本模式下使用 FetchContent_Populate() 时，也必须提供构建工具。

  • 可以使用生成器表达式和 $(SOMEVAR) 形式的 GNU Make（GNU 构建工具） 变量，尽管这种使用几乎总是不合适的。它们是在子构建中求值的，因此它们无法获取主构建中的任何信息。

  • 支持所有与下载、更新或补丁步骤相关的日志记录、终端控制和目录选项。

  • 如果使用了 QUIET 选项，或者 FETCHCONTENT_QUIET 变量被设置为 true，则输出中不会显示警告。

  • cmake --fresh 对依赖项的标记文件或脚本文件没有影响。先前执行的步骤只有在依赖项详情发生变化时才会重新运行。

用户有相当大的机会可以将 CMAKE_POLICY_DEFAULT_CMP0168 变量设置为 NEW，从而全局切换到 NEW 行为，同时等待项目及其依赖项更新以默认使用 NEW 策略设置。项目通常不会利用 NEW 行为不再支持的功能，即使是那些使用了这些功能的项目，在这些选项被忽略时通常也能正常工作。在全局设置此行为之前，请检查是否有任何 FetchContent_Declare() 或 FetchContent_Populate() 调用以某种会改变可观察行为的方式使用了被忽略的选项（将临时文件或内部生成的文件放在不同位置的情况除外）。

此策略引入于 CMake 版本 3.30。它可以通过 cmake_policy() 或 cmake_minimum_required() 设置。如果未设置，CMake *不会* 警告，并使用 OLD 行为。

注意

策略的 OLD 行为根据定义已被弃用，并可能在未来版本的 CMake 中被移除。
