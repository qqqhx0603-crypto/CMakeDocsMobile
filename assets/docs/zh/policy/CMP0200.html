CMP0200¶

版本 4.2 中添加。

导入目标（imported targets（目标））的定位和配置选择更加一致。

CMake 历史上选择导入目标所用配置的方式，优先考虑候选配置的位置属性，仅将 IMPORTED_CONFIGURATIONS 作为后备选项。这可能导致不正确的配置选择，尤其是对于 INTERFACE 库。

CMake 4.2 及更高版本将 IMPORTED_CONFIGURATIONS 视为可用配置的最终列表，无论是否为该库提供了特定于配置的位置。此外，当未指定特定于配置的位置时，CMake 将尊重非特定于配置的位置。

此策略为依赖于历史行为的项目提供了兼容性。策略设置应用于目标，并在创建导入目标时记录。因此，导入的包可能会覆盖消费者为其创建的目标所设置的策略。特别是，从“通用包规范”（Common Package Specification）包中导入的目标始终使用 NEW 行为。

此策略的 OLD 行为是保留历史行为。而 NEW 行为则根据公布的可用配置列表优先进行选择。两种行为都在下文中有详细描述。

此策略在 CMake 4.2 版本中引入。它可以通过 cmake_policy() 或 cmake_minimum_required() 进行设置。如果未设置，CMake 将发出警告并使用 OLD 行为。

注意

策略的 OLD 行为根据定义已被弃用，并可能在未来版本的 CMake 中被移除。

映射配置选择¶

如果 MAP_IMPORTED_CONFIG_<CONFIG>（其中 <CONFIG> 是消费目标的配置）在导入目标上被设置，CMake 过去会从该列表中选择第一个提供特定于配置位置的配置。如果不存在这样的配置，且导入目标是 INTERFACE 库，CMake 会选择消费目标的配置。否则，CMake 会认为该目标没有合适的配置。

对于不提供位置的 INTERFACE 库，这导致 CMake 总是选择消费目标的配置，并有效地忽略了 MAP_IMPORTED_CONFIG_<CONFIG>。这种行为与提供位置的导入目标的配置选择方式不一致。

在 NEW 行为下，CMake 会从映射中选择出现在 IMPORTED_CONFIGURATIONS 中的第一个配置。如果未设置 IMPORTED_CONFIGURATIONS，CMake 会选择映射中第一个“可用”的配置。对于非 INTERFACE 库，“可用”意味着有一个位置（无论是特定于配置的还是与配置无关的）是可用的。INTERFACE 库始终被认为是“可用”的。

如果没有找到匹配项，CMake 会认为该目标没有合适的配置。

非映射配置选择¶

如果未设置 MAP_IMPORTED_CONFIG_<CONFIG>，CMake 过去会从以下选项中选择第一个提供位置的配置：

  • 消费目标的配置，或者

  • 空配置，或者

  • IMPORTED_CONFIGURATIONS 中的配置列表。

作为一种实现产物，这导致 CMake 为不提供位置的 INTERFACE 库选择了 IMPORTED_CONFIGURATIONS 中的最后一个配置。同样，这种行为与提供位置的导入目标的配置选择方式不一致。

在 NEW 行为下，如果设置了 IMPORTED_CONFIGURATIONS，如果其中存在消费目标的配置，CMake 将选择它，否则将选择第一个导入的配置。如果未设置 IMPORTED_CONFIGURATIONS，如果消费目标的配置是“可用”的（如上一节所定义），CMake 将选择它；否则，CMake 会认为该目标没有合适的配置。

示例¶

考虑以下导入的库：

add_library(test INTERFACE IMPORTED)
set_target_properties(test PROPERTIES
  IMPORTED_CONFIGURATIONS "RELEASE;DEBUG"
  INTERFACE_COMPILE_DEFINITIONS "$<$<CONFIG:debug>:DEBUG>"
)

代码说明

  • add_library(test INTERFACE IMPORTED)：调用 add_library() 命令，定义一个库目标。

  • set_target_properties(test PROPERTIES：这一行继续补充当前命令或表达式的内容。

  • IMPORTED_CONFIGURATIONS "RELEASE;DEBUG"：这一行继续补充当前命令或表达式的内容。

  • INTERFACE_COMPILE_DEFINITIONS "$<$<CONFIG:debug>:DEBUG>"：这一行继续补充当前命令或表达式的内容；生成器表达式：$<CONFIG:...> 在生成构建系统时按当前目标和配置计算结果；运算符：> 表示大于比较。

  • )：这一行继续补充当前命令或表达式的内容。

在 OLD 策略下，CMake 将为任何链接到 test 的目标选择 test 的 DEBUG 配置（从而定义符号 DEBUG），因为 CMake 不认为任何配置是“有效的”，并且作为一种实现产物，最后考虑的配置会被接受。

在 NEW 策略下，如果消费项目以 Debug 以外的任何配置构建（请记住配置匹配是不区分大小写的），则将选择 RELEASE 配置。这是因为如果消费者的配置也是 DEBUG，则会优先选择 DEBUG；否则，由于 RELEASE 在 IMPORTED_CONFIGURATIONS 中排在第一位，并且其出现使其成为 INTERFACE 库的“有效”配置，因此将优先选择 RELEASE。
