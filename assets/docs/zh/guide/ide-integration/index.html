简介¶

集成开发环境 (IDE) 可能希望与 CMake 集成，以改善 CMake 用户的使用体验。本文档列出了此类集成的推荐最佳实践。

捆绑¶

许多 IDE 供应商希望将 CMake 的副本与其 IDE 捆绑在一起。捆绑了 CMake 的 IDE 应向用户提供使用外部 CMake 安装而非捆绑版本的选项，以防捆绑版本过时，且用户希望使用更新的版本。

虽然 IDE 供应商可能倾向于在其应用程序中捆绑不同版本的 CMake，但不建议这样做。CMake 具有强大的向后兼容性保证，因此没有理由不使用比项目要求更新的版本，甚至直接使用最新版本。因此，建议将 CMake 捆绑在应用程序中的 IDE 供应商，在发布时始终包含 CMake 的最新补丁版本。

建议 IDE 也可以在 CMake 旁边随附一份 Ninja（构建工具） 构建系统的副本。Ninja（构建工具） 在所有支持 CMake 的平台上性能卓越且受支持良好。捆绑 Ninja（构建工具） 的 IDE 应使用 Ninja（构建工具） 1.10 或更高版本，因为它包含了支持 Fortran 构建所需的功能。

预设 (Presets)¶

CMake 支持一种称为 CMakePresets.json 的文件格式，以及其用户特定的对应文件 CMakeUserPresets.json。该文件包含用户可能需要的各种配置预设信息。每个预设可能有不同的编译器、构建标志等。此格式的详细信息在 cmake(1) 手册中进行了说明。

鼓励 IDE 供应商以 CMake 相同的方式读取和评估此文件，并将文件中列出的预设呈现给用户。用户应能够查看（并可能编辑）为给定预设定义的 CMake 缓存变量、环境变量和命令行选项。然后，IDE 应根据这些设置构造适当的 cmake(1) 命令行参数列表，而不是直接使用 --preset= 选项。--preset= 选项仅旨在作为命令行用户的便捷前端，不应由 IDE 使用。

例如，如果名为 ninja 的预设指定 Ninja（构建工具） 作为生成器，并指定 ${sourceDir}/build 作为构建目录，IDE 不应运行

cmake -S /path/to/source --preset=ninja

代码说明

  • cmake -S /path/to/source --preset=ninja：这是一条命令行输入，调用 CMake 命令。

而是应该计算 ninja 预设的设置，然后运行

cmake -S /path/to/source -B /path/to/source/build -G Ninja

代码说明

  • cmake -S /path/to/source -B /path/to/source/build -G Ninja：这是一条命令行输入，调用 CMake 命令。

如果预设包含大量缓存变量，且将所有变量作为 -D 标志传递会导致超过平台的命令行长度限制，IDE 则应构造一个临时缓存脚本并使用 -C 标志传递它。

虽然读取、解析和评估 CMakePresets.json 的内容很简单，但这并不容易。除了文档之外，IDE 供应商还可以参考 CMake 源代码和测试用例，以更好地了解如何实现该格式。此文件 提供了 CMakePresets.json 格式的机器可读 JSON 模式，IDE 供应商可能会发现它对验证和提供编辑辅助很有用。

配置¶

调用 cmake(1) 执行配置步骤的 IDE 可能希望接收有关构建将产生的制品、以及用于构建制品的包含目录、编译定义等的信息。此类信息可以通过使用 文件 API (File API) 获得。文件 API 的手册页包含了有关该 API 及其调用方式的更多信息。服务器模式 (Server mode) 已在 CMake 3.20 中移除，不应在 CMake 3.14 或更高版本中使用。

IDE 应避免创建多于必要的构建树，并且仅在用户希望切换到不同编译器、使用不同编译标志等情况下才创建多个构建树。特别地，IDE 不应创建除了 CMAKE_BUILD_TYPE 不同之外所有属性都相同的多个单配置构建树，从而有效地创建多配置环境。相反，应为此目的使用 Ninja（构建工具） Multi-Config 生成器，并结合 文件 API 来获取构建配置列表。

IDE 不应将“额外生成器 (extra generators)”与 Makefile（构建脚本） 或 Ninja（构建工具） 生成器一起使用，因为除了 Makefile（构建脚本） 或 Ninja（构建工具） 文件外，它们还会生成 IDE 项目文件。相反，应使用 文件 API 来获取构建制品列表。

构建¶

如果使用 Makefile（构建脚本） 或 Ninja（构建工具） 生成器来生成构建树，则不建议直接调用 make 或 ninja。相反，建议 IDE 调用带有 --build 参数的 cmake(1)，这将依次调用适当的构建工具。

如果使用 IDE 项目生成器（例如 Xcode 或某个 Visual Studio（集成开发环境） 生成器），且 IDE 理解所使用的项目格式，则 IDE 应读取项目文件并以其原有的方式进行构建。

文件 API 可用于从构建树中获取构建配置列表，IDE 应将此列表呈现给用户以供选择构建配置。

测试¶

ctest(1) 支持输出有关可用测试和测试配置信息的 JSON 格式。想要运行 CTest 的 IDE 应获取此信息，并将其用于向用户展示测试列表。

IDE 不应调用生成的构建系统的 test 目标。相反，它们应该直接调用 ctest(1)。

集成 CMake 的 IDE¶

以下 IDE 原生支持 CMake

  • CLion

  • KDevelop

  • QtCreator

  • Vim（通过插件）

  • Visual Studio（集成开发环境）

  • VSCode（通过插件）

此外，CMake 对某些 IDE 具有内置支持

  • IDE 构建工具生成器：生成 IDE 原生构建系统，例如 Visual Studio（集成开发环境） 或 Xcode。

  • 额外生成器：扩展 命令行构建工具生成器，以生成挂载到命令行构建系统的 IDE 项目文件。已被 文件 API 取代。
