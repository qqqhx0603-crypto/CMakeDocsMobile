简介¶

CMake Instrumentation API 允许在 CMake 项目的配置、生成、构建、测试和安装步骤中收集计时数据、目标信息和系统诊断信息。

此功能仅适用于使用 Makefile（构建脚本） 生成器、Ninja（构建工具） 生成器 或 FASTBuild 的项目。

所有与 CMake Instrumentation API 的交互都必须指定 API 版本和数据版本。目前支持的 API 版本是 API v1；目前支持的数据版本是 1.0。

数据收集¶

每当启用 Instrumentation 时执行命令，项目构建树中就会创建一个包含该命令特定数据的 v1 片段文件。这些文件将保留到 索引 发生之后。

CMake 设置了 RULE_LAUNCH_COMPILE、RULE_LAUNCH_LINK 和 RULE_LAUNCH_CUSTOM 全局属性，以将每个编译、链接和自定义命令调用包装在执行 Instrumentation 并写入 v1 片段文件 的启动程序中。如果项目已使用 CTestUseLaunchers 配置，则启动程序除了执行该模块通常处理的通信外，还将收集 Instrumentation 数据。

索引¶

索引是整理生成的 Instrumentation 数据的过程。索引发生在称为“钩子”(hooks) 的特定间隔，例如在每次构建之后。这些钩子作为 v1 查询文件 的一部分进行配置。每当触发钩子时，就会生成一个索引文件，其中包含比上次索引更新的片段文件列表。

索引也可以通过手动调用 ctest --collect-instrumentation 来执行。

回调¶

作为 v1 查询文件 的一部分，用户可以提供一系列回调函数，旨在处理此功能收集的数据。

每当 索引 发生时，每个提供的回调函数都会被执行，并将生成的索引文件的路径作为参数传递。

这些定义在用户级别或项目级别的回调函数应读取 Instrumentation 数据并执行任何需要的处理。一旦所有回调完成，索引文件及其列出的片段将被 CMake 自动删除。请注意，回调绝不应手动移动或删除这些数据文件，因为其他回调可能需要它们。

启用 Instrumentation¶

可以为单个 CMake 项目启用 Instrumentation，也可以为用户配置和构建的所有 CMake 项目启用。在这两种情况下，请参阅 v1 查询文件 以了解配置此功能的详细信息。

在项目级别启用 Instrumentation¶

项目代码可以使用 cmake_instrumentation() 命令包含 Instrumentation 查询。

此外，查询文件可以手动放置在构建树顶部的 <build>/.cmake/instrumentation/<version>/query/ 下。此版本的 CMake 仅支持一个版本方案，即 API v1。

在用户级别启用 Instrumentation¶

可以通过将查询文件放置在 CMAKE_CONFIG_DIR 下的 <config_dir>/instrumentation/<version>/query/ 中，在用户级别配置 Instrumentation。

API v1¶

API 版本指定了 Instrumentation 目录的布局，以及查询文件和 cmake_instrumentation() 命令参数的一般格式。

Instrumentation API v1 位于项目构建树中 <build>/.cmake/（用于输出数据和项目级查询）或 <config_dir>/（用于用户级查询）下的 instrumentation/v1/ 目录中。此目录的 v1 组件代表 API 版本。它具有以下子目录：

query/

存放用户或客户端编写的查询文件。任何带有 .json 文件扩展名的文件都将被识别为查询文件。这些文件由创建它们的客户端或用户所有。

query/generated/

存放由 CMake 项目通过 cmake_instrumentation() 命令或 CTEST_USE_INSTRUMENTATION 变量生成的查询文件。这些文件由 CMake 所有，并在 CMake 配置步骤中自动删除和重新生成。

data/

存放项目收集的 Instrumentation 数据。CMake 拥有所有数据文件，它们不应被其他进程删除。此处收集的数据将保留到 索引 发生且所有 回调 执行完毕之后。

data/index/

收集数据的子集，包含任何 v1 索引文件。

data/content/

收集数据的子集，包含任何 v1 CMake 内容文件。

data/trace/

收集数据的子集，包含从最近的 索引 创建的 Google Trace 文件。与其他数据文件不同，最近的跟踪文件在 索引 发生且所有 回调 执行完毕后仍然保留，直到下一次 索引 发生。

cdash/

存放内部用于生成要提交给 CDash 的 XML 内容的临时文件。

v1 查询文件¶

任何在 instrumentation/v1/query/ 目录下带有 .json 扩展名的文件都被识别为 Instrumentation 数据查询。

这些文件必须包含一个具有以下键的 JSON 对象。version 键是必需的，所有其他字段都是可选的。

version

要生成的片段文件的数据版本。

在查询文件中，它可以写成整数形式的主版本号，也可以写成包含 major 和 minor 成员的对象。例如，1 和 { "major": 1, "minor": 0 } 都表示请求版本 1.0。次版本号可以省略。即使请求较早的次版本，CMake 也始终生成最新次版本的插桩数据。

目前唯一支持的版本是 1.0。未知数据版本的查询文件会被忽略。

callbacks

用于处理收集到的 Instrumentation 数据的 回调 命令行字符串列表。每当执行这些回调时，生成的 v1 索引文件 的完整路径会被追加到字符串中包含的参数中。

hooks

指定何时应自动进行 索引 的字符串列表。这些是应该整理 Instrumentation 数据并调用用户 回调 来处理数据的间隔。此列表中的元素应为以下之一：

  • postGenerate

  • preBuild（在调用 ninja 或 make 时调用）

  • postBuild（在 ninja 或 make 完成时调用）

  • preCMakeBuild（在调用 cmake --build 时调用）

  • postCMakeBuild（在 cmake --build 完成时调用）

  • postCMakeInstall

  • postCMakeWorkflow

  • postCTest

当使用 MSYS Makefile（构建脚本）s 或 FASTBuild 生成器时，不支持 preBuild 和 postBuild。此外，当通过 cmake --build 调用构建工具时，它们不会被触发。

options

用于启用某些可选行为的字符串列表，包括收集某些额外数据。此列表中的元素应为以下之一：

staticSystemInformation

启用收集有关运行 CMake 的宿主机机器的静态信息。此数据在 索引 期间收集，并包含在生成的 v1 索引文件 中。

dynamicSystemInformation

启用收集有关运行 CMake 的宿主机机器的动态信息。数据针对 CMake 生成的每个 v1 片段文件 进行收集，并包括命令执行前后立即的信息。

cdashSubmit

启用将 Instrumentation 数据包含在 CDash 中。这等同于启用了 CTEST_USE_INSTRUMENTATION 环境变量。

cdashVerbose

启用在提交给 CDash 的数据中包含完整的未截断命令。等同于启用了 CTEST_USE_VERBOSE_INSTRUMENTATION 和 CTEST_USE_INSTRUMENTATION 环境变量。

trace

启用在 索引 期间生成 Google Trace 文件，以可视化收集到的 v1 片段文件 中的数据。

列出的 callbacks 将在指定的钩子处至少被调用一次。当存在多个查询文件时，它们之间的 callbacks、hooks 和 options 将被合并。因此，如果任何查询文件包含任何 hooks，则跨所有查询文件的每个 callback 将在所有查询文件的每个 hook 处执行。此外，如果任何查询文件使用 options 字段请求可选数据，任何相关数据都将出现在所有片段文件中。用户编写的 callbacks 应该能够处理这种可选数据的存在，因为它可能是由不相关的查询请求的。

JSON 格式由 此 JSON 模式 以机器可读的形式描述。

示例

{
  "version": 1,
  "callbacks": [
    "/usr/bin/python callback.py",
    "/usr/bin/cmake -P callback.cmake arg",
  ],
  "hooks": [
    "postCMakeBuild",
    "postCMakeInstall"
  ],
  "options": [
    "staticSystemInformation",
    "dynamicSystemInformation",
    "cdashSubmit",
    "trace"
  ]
}

代码说明

  • { "version": 1, "callbacks": [ "/usr/bin/python callback.py", "/usr/bin/cmake -P callback.cmake arg", ], "hooks": [ "postCMakeBuild", "postCMakeInstall" ], "options": [ "staticSystemInformation", "dynamicSystemInformation", "cdashSubmit", "trace" ] }：这一行给键 `{ "version` 指定对应的值。

在此示例中，在每次 cmake --build 或 cmake --install 调用后，将在 <build>/.cmake/instrumentation/v1/data/index 中生成索引文件 index-<timestamp>.json，其中包含自上次索引以来创建的数据片段文件列表。命令 /usr/bin/python callback.py index-<timestamp>.json 和 /usr/bin/cmake -P callback.cmake arg index-<timestamp>.json 将按该顺序执行。索引文件将包含 staticSystemInformation 数据，索引中列出的每个片段文件都将包含 dynamicSystemInformation 数据。此外，索引文件将包含生成的 Google Trace 文件 的路径。一旦两个回调都完成，索引文件及其列出的数据文件（包括片段文件，但不包括跟踪文件）将从项目构建树中删除。Instrumentation 数据将出现在提交给 CDash 的 XML 文件中，但由于未启用 cdashVerbose，命令字符串将被截断。

数据版本¶

数据版本指定了 API v1 在 数据收集 和 索引 过程中生成的输出文件内容。

v1 查询文件 或 cmake_instrumentation() 调用应请求特定的数据版本；CMake 会生成对应版本的 v1 数据文件，并发送给该查询定义的用户 回调。

目前唯一支持的版本是 1.0。当删除或重新格式化已有数据、导致解析这些数据的脚本可能与新格式不兼容时，会创建新的主版本号；当新增数据可用时，会创建新的次版本号。

v1 片段文件¶

片段文件是为作为 CMake 构建或安装步骤一部分调用的每个编译、链接和自定义命令生成的，并包含有关执行的命令的 Instrumentation 数据。此外，以下情况也会创建片段文件：

  • CMake 配置步骤

  • CMake 生成步骤

  • 整个构建步骤（使用 cmake --build 执行）

  • 整个安装步骤（使用 cmake --install 执行）

  • 每次调用 ctest 来 运行测试 时（即使未找到测试）

  • ctest 执行的每个单独测试

这些文件将保留在构建树中，直到 索引 发生且任何用户指定的 回调 执行完毕。

注意

配置和生成片段文件在生成步骤完成之前不会由 CMake 写入。当使用 cmake-gui(1) 或 ccmake(1) 时，仅触发配置步骤而不生成项目文件将不会生成任何配置片段。一旦运行了生成步骤，每次运行配置步骤就会有一个配置片段。

片段文件的文件名语法为 <role>-<hash>-<timestamp>.json，并包含以下数据：

version

片段文件的 数据版本。目前始终写为 { "major": 1, "minor": 0 }。

命令

执行的完整命令。当 role 为 build 时排除。

workingDir

执行 command 的工作目录。

result

命令的退出代码，为一个整数。当 role 为 build 时，这将为 null。

role

执行的命令类型，将为以下值之一：

  • configure：CMake 配置步骤

  • generate：CMake 生成步骤

  • compile：构建期间调用的单个编译步骤

  • link：构建期间调用的单个链接步骤

  • custom：构建期间调用的单个自定义命令

  • build：完整的 make 或 ninja 调用（非通过 cmake --build）。

  • cmakeBuild：完整的 cmake --build 调用

  • cmakeInstall：完整的 cmake --install 调用

  • install：单个 cmake -P cmake_install.cmake 调用

  • ctest：完整的 ctest 命令调用

  • test：ctest 执行的单个测试

目标

与命令关联的 CMake 目标。仅在 role 为 compile 或 link 时，或者当 role 为 custom 且自定义命令通过 add_custom_command(TARGET) 附加到目标时包含。结合 cmakeContent，这可用于查找目标 TYPE 和 LABELS。

timeStart

命令开始的时间，表示为自系统纪元以来的毫秒数。

duration

命令运行的持续时间，以毫秒表示。

outputs

命令的输出文件，为一个数组。仅当 role 为以下之一时包含：compile、link、custom。

outputSizes

以字节为单位的 outputs 大小，为一个数组。对于不存在的文件，大小为 0。在与 outputs 字段相同的条件下包含。

源

正在编译的源文件。仅当 role 为 compile 时包含。

language

正在编译的源文件的语言。仅当 role 为 compile 或 link 时包含。

testName

正在执行的测试名称。仅当 role 为 test 时包含。

config

构建配置，例如 Release 或 Debug。仅当 role 为以下之一时包含：compile、link、custom、install、test。

dynamicSystemInformation

指定收集有关运行 CMake 的宿主机机器的动态信息。针对 CMake 生成的每个片段文件收集数据，并包括命令执行前后立即的数据。仅当通过 v1 查询文件 启用时包含。

beforeHostMemoryUsed

timeStart 时已使用的宿主机内存（以 KiB 为单位）。

afterHostMemoryUsed

timeStart + duration 时已使用的宿主机内存（以 KiB 为单位）。

beforeCPULoadAverage

timeStart 时的平均 CPU 负载，如果无法确定则为 null。

afterCPULoadAverage

timeStart + duration 时的平均 CPU 负载，如果无法确定则为 null。

cmakeContent

位于 data 下的 v1 CMake 内容文件 的路径，其中包含有关负责生成此片段中 command 的 CMake 配置和生成步骤的信息。使用 cmake-gui(1) 或 ccmake(1) 时，此字段对于生成步骤之前的所有配置步骤可能为 null。

showOnly

一个布尔值，表示是否将 --show-only 选项传递给了 ctest。仅当 role 为 ctest 时包含。

v1 索引文件¶

索引文件包含 v1 片段文件 的列表。它充当导航 Instrumentation 数据的入口点。它们在 索引 发生时生成，并在任何用户指定的 回调 执行后删除。

version

索引文件的 数据版本。目前始终写为 { "major": 1, "minor": 0 }。

buildDir

CMake 项目的构建目录。

dataDir

<build>/.cmake/instrumentation/v1/data/ 目录的完整路径。

hook

负责生成索引文件的钩子名称。除了可以通过 v1 查询文件 之一指定的钩子外，如果通过调用 ctest --collect-instrumentation 执行索引，则此值可能设置为 manual。

snippets

包含 v1 片段文件 的列表。这包括自上一个索引文件创建以来生成的所有片段文件。文件路径相对于 dataDir。

trace

包含 Google Trace 文件 的路径。这包括来自索引文件中所有相应 snippets 的数据。文件路径相对于 dataDir。仅当通过 v1 查询文件 启用时包含。

staticSystemInformation

指定收集有关运行 CMake 的宿主机机器的静态信息。如果 CMake 无法确定任何给定字段的值，则它将为 null。有关以下每个字段的描述，请参阅 cmake_host_system_information()。

仅当通过 v1 查询文件 启用时包含。

  • OSName

  • OSPlatform

  • OSRelease

  • OSVersion

  • familyId

  • hostname

  • is64Bits

  • modelId

  • modelName

  • numberOfLogicalCPU

  • numberOfPhysicalCPU

  • processorAPICID

  • processorCacheSize

  • processorClockFrequency

  • processorName

  • totalPhysicalMemory

  • totalVirtualMemory

  • vendorID

  • vendorString

v1 CMake 内容文件¶

CMake 内容文件包含有关 CMake 配置和生成步骤的信息。每个 v1 片段文件 都提供指向其中一个文件的路径，该文件对应于负责生成其命令的 CMake 调用。

每个 CMake 内容文件包含以下内容：

project

CMAKE_PROJECT_NAME 的值。

自定义

一个对象，其中包含用户通过 cmake_instrumentation() 命令的 自定义 CMake 内容 功能指定的任意 JSON 数据。

version

CMake 内容文件的 数据版本。目前始终写为 { "major": 1, "minor": 0 }。

targets（目标）

一个包含按名称索引且具有相应 Instrumentation 数据的 CMake 目标的对象。每个目标包含以下内容：

type

目标的 TYPE 属性。仅包含 EXECUTABLE、STATIC_LIBRARY、SHARED_LIBRARY、MODULE_LIBRARY 和 OBJECT_LIBRARY 目标。

labels

目标的 LABELS 属性。

Google Trace 文件¶

跟踪文件遵循 Google 跟踪事件格式。它们包括当前索引文件中列出的所有 v1 片段文件 中的数据。这些文件即使在 索引 发生且所有 回调 执行完毕后仍保留在构建树中，直到下一次 索引 发生。

跟踪文件以 JSON 数组格式 存储，其中每个 v1 片段文件 对应于单个跟踪事件对象。每个跟踪事件包含以下数据：

名称

由 CMake 根据给定的片段数据生成的描述性名称。

cat

v1 片段文件 中的 role。

ph

目前始终为 "X"，表示“完成事件”。

ts

v1 片段文件 中的 timeStart，从毫秒转换为微秒。

dur

v1 片段文件 中的 duration，从毫秒转换为微秒。

pid

未使用（始终为零）。

tid

一个介于零与正在索引的进程运行的并发作业数之间的整数。这是 CMake 根据正在索引的所有片段文件的 ts 和 dur 计算出的合成 ID，旨在产生更有用的进程并发可视化。

args

包含来自与此跟踪事件对应的 v1 片段文件 的所有数据。
