简介¶

生成器表达式在构建系统生成期间求值，以产生特定于每个构建配置的信息。它们的形式为 $<...>。例如：

target_include_directories(tgt PRIVATE /opt/include/$<CXX_COMPILER_ID>)

代码说明

  • target_include_directories(tgt PRIVATE /opt/include/$<CXX_COMPILER_ID>)：调用 `target_include_directories()` 命令，为目标配置头文件搜索目录。

根据所使用的 C++ 编译器，这会扩展为 /opt/include/GNU、/opt/include/Clang 等。

许多目标属性上下文中允许使用生成器表达式，例如 LINK_LIBRARIES、INCLUDE_DIRECTORIES、COMPILE_DEFINITIONS 等。它们也可以在填充这些属性的命令中使用，例如 target_link_libraries()、target_include_directories()、target_compile_definitions() 等。它们实现了条件链接、编译时的条件定义、条件包含目录等功能。这些条件可以基于构建配置、目标属性、平台信息或任何其他可查询的信息。

生成器表达式可以嵌套。

target_compile_definitions(tgt PRIVATE
  $<$<VERSION_LESS:$<CXX_COMPILER_VERSION>,4.2.0>:OLD_COMPILER>
)

代码说明

  • target_compile_definitions(tgt PRIVATE $<$<VERSION_LESS:$<CXX_COMPILER_VERSION>,4.2.0>:OLD_COMPILER> )：调用 `target_compile_definitions()` 命令，为目标设置编译宏。

如果 CMAKE_CXX_COMPILER_VERSION 小于 4.2.0，上述表达式将扩展为 OLD_COMPILER。

空白与引用¶

生成器表达式通常在命令参数解析后进行求值。如果生成器表达式包含空格、换行符、分号或其他可能被解释为命令参数分隔符的字符，在传递给命令时，整个表达式应加引号括起来。否则，表达式可能会被拆分，从而不再被识别为生成器表达式。

使用 add_custom_command() 或 add_custom_target() 时，使用 VERBATIM 和 COMMAND_EXPAND_LISTS 选项可以获得稳健的参数拆分和引用效果。

# WRONG: Embedded space will be treated as an argument separator.
# This ends up not being seen as a generator expression at all.
add_custom_target(run_some_tool
  COMMAND some_tool -I$<JOIN:$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>, -I>
  VERBATIM
)

代码说明

  • # WRONG: Embedded space will be treated as an argument separator. # This ends up not being seen as a generator expression at all. add_custom_target(run_some_tool COMMAND some_tool -I$<JOIN:$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>, -I> VERBATIM )：这是注释，用来说明：WRONG: Embedded space will be treated as an argument separator. # This ends up not being seen as a generator expression at all. add_custom_target(run_some_tool COMMAND some_tool -I$<JOIN:$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>, -I> VERBATIM )。

# Better, but still not robust. Quotes prevent the space from splitting the
# expression. However, the tool will receive the expanded value as a single
# argument.
add_custom_target(run_some_tool
  COMMAND some_tool "-I$<JOIN:$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>, -I>"
  VERBATIM
)

使用变量构建复杂的生成器表达式也是减少错误并提高可读性的好方法。上面的示例可以这样进一步改进：

# The $<BOOL:...> check prevents adding anything if the property is empty,
# assuming the property value cannot be one of CMake's false constants.
set(prop "$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>")
add_custom_target(run_some_tool
  COMMAND some_tool "$<$<BOOL:${prop}>:-I$<JOIN:${prop},;-I>>"
  COMMAND_EXPAND_LISTS
  VERBATIM
)

代码说明

  • # The $<BOOL:...> check prevents adding anything if the property is empty, # assuming the property value cannot be one of CMake's false constants. set(prop "$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>") add_custom_target(run_some_tool COMMAND some_tool "$<$<BOOL:${prop}>:-I$<JOIN:${prop},;-I>>" COMMAND_EXPAND_LISTS VERBATIM )：这是注释，用来说明：The $<BOOL:...> check prevents adding anything if the property is empty, # assuming the property value cannot be one of CMake's false constants. set(prop "$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>") add_custom_target(run_some_tool COMMAND some_tool "$<$<BOOL:${prop}>:-I$<JOIN:${prop},;-I>>" COMMAND_EXPAND_LISTS VERBATIM )。

最后，上面的例子可以使用替代的生成器表达式以更简单、更稳健的方式表示：

add_custom_target(run_some_tool
  COMMAND some_tool "$<LIST:TRANSFORM,$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>,PREPEND,-I>"
  COMMAND_EXPAND_LISTS
  VERBATIM
)

代码说明

  • add_custom_target(run_some_tool COMMAND some_tool "$<LIST:TRANSFORM,$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>,PREPEND,-I>" COMMAND_EXPAND_LISTS VERBATIM )：调用 `add_custom_target()` 命令，定义一个没有直接输出文件的自定义目标。

对于那些期望 -I 的值作为一个单独参数的工具，请使用前面学到的分号技巧：

add_custom_target(run_some_tool
  COMMAND some_tool "$<LIST:TRANSFORM,$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>,PREPEND,-I;>"
  COMMAND_EXPAND_LISTS
  VERBATIM
)

代码说明

  • add_custom_target(run_some_tool COMMAND some_tool "$<LIST:TRANSFORM,$<TARGET_PROPERTY:tgt,INCLUDE_DIRECTORIES>,PREPEND,-I;>" COMMAND_EXPAND_LISTS VERBATIM )：调用 `add_custom_target()` 命令，定义一个没有直接输出文件的自定义目标。

一个常见的错误是试图在多行上拆分生成器表达式并进行缩进：

# WRONG: New lines and spaces all treated as argument separators, so the
# generator expression is split and not recognized correctly.
target_compile_definitions(tgt PRIVATE
  $<$<AND:
      $<CXX_COMPILER_ID:GNU>,
      $<VERSION_GREATER_EQUAL:$<CXX_COMPILER_VERSION>,5>
    >:HAVE_5_OR_LATER>
)

代码说明

  • # WRONG: New lines and spaces all treated as argument separators, so the # generator expression is split and not recognized correctly. target_compile_definitions(tgt PRIVATE $<$<AND: $<CXX_COMPILER_ID:GNU>, $<VERSION_GREATER_EQUAL:$<CXX_COMPILER_VERSION>,5> >:HAVE_5_OR_LATER> )：这是注释，用来说明：WRONG: New lines and spaces all treated as argument separators, so the # generator expression is split and not recognized correctly. target_compile_definitions(tgt PRIVATE $<$<AND: $<CXX_COMPILER_ID:GNU>, $<VERSION_GREATER_EQUAL:$<CXX_COMPILER_VERSION>,5> >:HAVE_5_OR_LATER> )。

再次建议，使用名称恰当的辅助变量来构建可读的表达式：

set(is_gnu "$<CXX_COMPILER_ID:GNU>")
set(v5_or_later "$<VERSION_GREATER_EQUAL:$<CXX_COMPILER_VERSION>,5>")
set(meet_requirements "$<AND:${is_gnu},${v5_or_later}>")
target_compile_definitions(tgt PRIVATE
  "$<${meet_requirements}:HAVE_5_OR_LATER>"
)

代码说明

  • set(is_gnu "$<CXX_COMPILER_ID:GNU>") set(v5_or_later "$<VERSION_GREATER_EQUAL:$<CXX_COMPILER_VERSION>,5>") set(meet_requirements "$<AND:${is_gnu},${v5_or_later}>") target_compile_definitions(tgt PRIVATE "$<${meet_requirements}:HAVE_5_OR_LATER>" )：调用 `set()` 命令，把变量 `is_gnu` 设为给定的值。

调试¶

由于生成器表达式是在生成构建系统期间求值的，而不是在处理 CMakeLists.txt 文件时求值，因此无法使用 message() 命令检查其结果。一种生成调试消息的方法是添加一个自定义目标：

add_custom_target(genexdebug COMMAND ${CMAKE_COMMAND} -E echo "$<...>")

代码说明

  • add_custom_target(genexdebug COMMAND ${CMAKE_COMMAND} -E echo "$<...>")：调用 `add_custom_target()` 命令，定义一个没有直接输出文件的自定义目标。

运行 cmake 后，您可以构建 genexdebug 目标来打印 $<...> 表达式的结果（即运行命令 cmake --build ... --target genexdebug）。

另一种方法是使用 file(GENERATE) 将调试消息写入文件：

file(GENERATE OUTPUT filename CONTENT "$<...>")

代码说明

  • file(GENERATE OUTPUT filename CONTENT "$<...>")：调用 `file()` 命令，执行文件系统相关操作。

生成器表达式参考¶

注意

本参考手册偏离了大多数 CMake 文档的惯例，即省略了占位符（如 condition、string、target 等）周围的尖括号 <...>。这是为了防止这些占位符被误解为生成器表达式。

条件表达式¶

生成器表达式的一个基础类别与条件逻辑有关。支持两种形式的条件生成器表达式：

$<condition:true_string>¶

如果 condition 为 1，则求值为 true_string；如果 condition 求值为 0，则求值为空字符串。condition 的任何其他值都会导致错误。

$<IF:condition,true_string,false_string>¶

版本 3.8 新增。

如果 condition 为 1，则求值为 true_string；如果 condition 为 0，则求值为 false_string。condition 的任何其他值都会导致错误。

3.28 版本新增：此生成器表达式采用短路求值，即当 condition 为 1 时，false_string 中的生成器表达式不会被求值；当 condition 为 0 时，true_string 中的生成器表达式不会被求值。

通常，condition 本身就是一个生成器表达式。例如，当使用 Debug 配置时，以下表达式扩展为 DEBUG_MODE，而在所有其他配置中则为空字符串：

$<$<CONFIG:Debug>:DEBUG_MODE>

代码说明

  • $<$<CONFIG:Debug>:DEBUG_MODE>：这一行继续补充当前命令或表达式的内容。

除 1 或 0 之外的布尔型 condition 值可以通过用 $<BOOL:...> 生成器表达式包装来处理：

$<BOOL:string>¶

将 string 转换为 0 或 1。如果以下任一情况为真，则求值为 0：

  • string 为空，

  • string 不区分大小写地等于 0、FALSE、OFF、N、NO、IGNORE 或 NOTFOUND，或者

  • string 以后缀 -NOTFOUND 结尾（区分大小写）。

否则求值为 1。

$<BOOL:...> 生成器表达式通常在 condition 由 CMake 变量提供时使用：

$<$<BOOL:${HAVE_SOME_FEATURE}>:-DENABLE_SOME_FEATURE>

代码说明

  • $<$<BOOL:${HAVE_SOME_FEATURE}>:-DENABLE_SOME_FEATURE>：这一行继续补充当前命令或表达式的内容。

逻辑运算符¶

支持常见的布尔逻辑运算符：

$<AND:conditions>¶

其中 conditions 是以逗号分隔的布尔表达式列表，所有表达式必须求值为 1 或 0。如果所有条件均为 1，则整个表达式求值为 1。如果任何条件为 0，则整个表达式求值为 0。

$<OR:conditions>¶

其中 conditions 是以逗号分隔的布尔表达式列表，所有表达式必须求值为 1 或 0。如果至少一个 conditions 为 1，则整个表达式求值为 1。如果所有 conditions 均求值为 0，则整个表达式求值为 0。

$<NOT:condition>¶

condition 必须为 0 或 1。如果 condition 为 1，则表达式结果为 0，否则为 1。

3.28 版本新增：逻辑运算符采用短路求值，即一旦可以确定返回值，参数列表中的生成器表达式将不会被求值。

基础比较表达式¶

CMake 支持多种用于比较事物的生成器表达式。本节介绍最主要和最常用的比较类型。其他更具体的比较类型将在下文的各个章节中单独说明。

数值比较¶

$<EQUAL:value1,value2>¶

如果 value1 和 value2 在数值上相等，则为 1，否则为 0。

版本比较¶

$<VERSION_LESS:v1,v2>¶

如果 v1 的版本小于 v2，则为 1，否则为 0。

$<VERSION_GREATER:v1,v2>¶

如果 v1 的版本大于 v2，则为 1，否则为 0。

$<VERSION_EQUAL:v1,v2>¶

如果 v1 与 v2 版本相同，则为 1，否则为 0。

$<VERSION_LESS_EQUAL:v1,v2>¶

3.7 版本中新增。

如果 v1 的版本小于或等于 v2，则为 1，否则为 0。

$<VERSION_GREATER_EQUAL:v1,v2>¶

3.7 版本中新增。

如果 v1 的版本大于或等于 v2，则为 1，否则为 0。

字符串表达式¶

本节中的大多数表达式都与 string() 命令紧密相关，提供了相同的功能，但形式为生成器表达式。

在以下每个与字符串相关的生成器表达式中，如果该生成器表达式期望在 string 之后提供某些内容，则 string 不得包含任何逗号。例如，表达式 $<STRING:FIND,string,value> 需要在 string 之后有一个 value。由于逗号用于分隔 string 和 value，因此 string 本身不能包含逗号。此限制不适用于 string() 命令，它仅针对字符串处理生成器表达式。可以使用 $<COMMA> 生成器表达式来在字符串相关生成器表达式的参数中指定逗号。

字符串比较¶

这些比较区分大小写。对于不区分大小写的比较，请结合使用 字符串转换生成器表达式。例如，如果 ${foo} 是 BAR、Bar、bar 等中的任意一个，则以下表达式求值为 1：

$<STREQUAL:$<STRING:TOUPPER,${foo}>,BAR>

代码说明

  • $<STREQUAL:$<STRING:TOUPPER,${foo}>,BAR>：这一行继续补充当前命令或表达式的内容。

$<STREQUAL:string1,string2>¶

如果 string1 和 string2 在字典序上相等，则为 1，否则为 0。

$<STRLESS:string1,string2>¶

4.3 版本新增。

如果 string1 在字典序上小于 string2，则为 1，否则为 0。

$<STRGREATER:string1,string2>¶

4.3 版本新增。

如果 string1 在字典序上大于 string2，则为 1，否则为 0。

$<STRLESS_EQUAL:string1,string2>¶

4.3 版本新增。

如果 string1 在字典序上小于或等于 string2，则为 1，否则为 0。

$<STRGREATER_EQUAL:string1,string2>¶

4.3 版本新增。

如果 string1 在字典序上大于或等于 string2，则为 1，否则为 0。

字符串查询¶

$<STRING:LENGTH,string>¶

4.3 版本新增。

给定字符串的字节长度。请注意，这意味着如果 string 包含多字节字符，结果将*不是*字符的数量。

$<STRING:SUBSTRING,string,begin,length>

4.3 版本新增。

给定 string 的子字符串。如果 length 为 -1 或大于 string 的长度，则返回从 begin 开始的剩余字符串。

begin 和 length 均按字节计算，因此如果 string 可能包含多字节字符，则必须小心。

$<STRING:FIND,string[,FROM:(BEGIN|END)],substring>

4.3 版本新增。

在所提供的 string 中找到 substring 的位置。如果未找到 substring，则返回位置 -1。

FROM: 选项定义搜索执行方式：

BEGIN

搜索将从 string 的开头开始。这是默认设置。

END

搜索将从 string 的末尾开始。

$<STRING:FIND> 生成器表达式将所有字符串视为仅含 ASCII 字符。返回的索引也将按字节计数，因此包含多字节字符的字符串可能会导致意外结果。

$<STRING:MATCH,string[,SEEK:(ONCE|ALL)],regular_expression>

4.3 版本新增。

在 string 中匹配 regular_expression。

SEEK: 选项指定匹配行为：

ONCE

仅匹配第一次出现的情况。这是默认设置。

ALL

尽可能多次匹配，并将匹配结果作为列表返回。

有关 regular_expression 参数的语法，请参阅 正则表达式规范。

字符串生成¶

$<STRING:JOIN,glue,input[,input]...>

4.3 版本新增。

使用 glue 字符串将所有 input 参数连接在一起。

$<STRING:ASCII,number[,number]...>

4.3 版本新增。

将 1-255 范围内的所有数字转换为相应的 ASCII 字符。超出此范围的任何数字都会引发错误。

$<STRING:TIMESTAMP[,(UTC|format)]...>

4.3 版本新增。

生成当前日期和/或时间的字符串表示形式。

如果生成器表达式无法获取时间戳，结果将为空字符串 ""。

可选的 UTC 标志请求以协调世界时 (UTC) 而非本地时间表示当前日期/时间。

如果设置了 SOURCE_DATE_EPOCH 环境变量，其值将代替当前时间使用。详细信息请参阅 https://reproducible-builds.org/specs/source-date-epoch/。

可选的 <format> 可能包含以下格式说明符：

%%

字面百分号 (%)。

%d

当前月份的日期 (01-31)。

%H

24 小时制的小时 (00-23)。

%I

12 小时制的小时 (01-12)。

%j

当前年份的一天 (001-366)。

%m

当前年份的月份 (01-12)。

%b

缩写的月份名称（例如 Oct）。

%B

完整的月份名称（例如 October）。

%M

当前小时的分钟 (00-59)。

%s

自 1970 年 1 月 1 日午夜 (UTC) 以来的秒数（UNIX 时间）。

%S

当前分钟的秒。60 代表闰秒。(00-60)

%f

当前秒的微秒 (000000-999999)。

%U

当前年份的周数 (00-53)。

%V

当前年份的 ISO 8601 周数 (01-53)。

%w

当前周的日期。0 是星期日。(0-6)

%a

缩写的星期几名称（例如 Fri）。

%A

完整的星期几名称（例如 Friday）。

%y

当前年份的最后两位数字 (00-99)。

%Y

当前年份。

%z

时区相对于 UTC 的偏移量，以小时和分钟表示，格式为 +hhmm 或 -hhmm。

%Z

时区名称。

未知的格式说明符将被忽略，并按原样复制到输出中。

如果未给出显式的 format，则默认为：

  • %Y-%m-%dT%H:%M:%S（本地时间）。

  • %Y-%m-%dT%H:%M:%SZ（UTC）。

$<STRING:RANDOM[,(LENGTH:length|ALPHABET:alphabet|RANDOM_SEED:seed)]...>

4.3 版本新增。

本节中的大多数表达式都与 list() 命令紧密相关，提供了相同的功能，但形式为生成器表达式。

将 ... 的内容转换为 shell（命令解释器） 路径样式。例如，在 Windows（微软桌面系统） shell（命令解释器） 中斜杠被转换为反斜杠，在 MSYS shell（命令解释器） 中驱动器盘符被转换为 posix 路径。... 必须是绝对路径。

add_library(myapp_c foo.c)

target_compile_options(myapp_cxx PUBLIC -fno-exceptions)
add_executable(myapp main.cpp)
target_link_libraries(myapp myapp_c myapp_cxx)

链接语言和 ID (Link Language and ID)¶

$<LINK_LANGUAGE>¶

在 3.18 版本中新增。

评估链接选项时目标的链接语言。有关此生成器表达式的可移植性说明，请参阅 相关的布尔表达式 $<LINK_LANGUAGE:languages>。

注意

链接库属性不支持此生成器表达式，以避免因这些属性的双重评估而产生的副作用。

$<LINK_LANGUAGE:languages>

在 3.18 版本中新增。

当用于链接步骤的语言与 languages 中的任何一个逗号分隔条目匹配时，返回 1，否则返回 0。此表达式可用于为目标中特定语言指定链接库、链接选项、链接目录和链接依赖项。例如

add_library(api_C ...)
add_library(api_CXX ...)
add_library(api INTERFACE)
target_link_options(api   INTERFACE $<$<LINK_LANGUAGE:C>:-opt_c>
                                    $<$<LINK_LANGUAGE:CXX>:-opt_cxx>)
target_link_libraries(api INTERFACE $<$<LINK_LANGUAGE:C>:api_C>
                                    $<$<LINK_LANGUAGE:CXX>:api_CXX>)

add_executable(myapp1 main.c)
target_link_options(myapp1 PRIVATE api)

add_executable(myapp2 main.cpp)
target_link_options(myapp2 PRIVATE api)

代码说明

  • add_library(api_C ...) add_library(api_CXX ...) add_library(api INTERFACE) target_link_options(api INTERFACE $<$<LINK_LANGUAGE:C>:-opt_c> $<$<LINK_LANGUAGE:CXX>:-opt_cxx>) target_link_libraries(api INTERFACE $<$<LINK_LANGUAGE:C>:api_C> $<$<LINK_LANGUAGE:CXX>:api_CXX>) add_executable(myapp1 main.c) target_link_options(myapp1 PRIVATE api) add_executable(myapp2 main.cpp) target_link_options(myapp2 PRIVATE api)：调用 `add_library()` 命令，定义一个库目标。

这指定使用 api 目标来链接 myapp1 和 myapp2。实际上，myapp1 将与目标 api_C 和选项 -opt_c 链接，因为它将使用 C 作为链接语言。而 myapp2 将与 api_CXX 和选项 -opt_cxx 链接，因为 CXX 将是其链接语言。

注意

为了确定目标的链接语言，需要传递性地收集将与其链接的所有目标。因此，对于链接库属性，将进行两次评估。在第一次评估期间，$<LINK_LANGUAGE:..> 表达式将始终返回 0。在第一次通过后计算出的链接语言将用于进行第二次通过。为了避免不一致，要求第二次通过不应更改链接语言。此外，为了避免意外的副作用，要求将完整的实体指定为 $<LINK_LANGUAGE:..> 表达式的一部分。例如

add_library(lib STATIC file.cxx)
add_library(libother STATIC file.c)

# bad usage
add_executable(myapp1 main.c)
target_link_libraries(myapp1 PRIVATE lib$<$<LINK_LANGUAGE:C>:other>)

# correct usage
add_executable(myapp2 main.c)
target_link_libraries(myapp2 PRIVATE $<$<LINK_LANGUAGE:C>:libother>)

代码说明

  • add_library(lib STATIC file.cxx) add_library(libother STATIC file.c) # bad usage add_executable(myapp1 main.c) target_link_libraries(myapp1 PRIVATE lib$<$<LINK_LANGUAGE:C>:other>) # correct usage add_executable(myapp2 main.c) target_link_libraries(myapp2 PRIVATE $<$<LINK_LANGUAGE:C>:libother>)：调用 `add_library()` 命令，定义一个库目标。

在此示例中，对于 myapp1，第一次通过将意外确定链接语言为 CXX，因为生成器表达式的求值将是一个空字符串，因此 myapp1 将依赖于 lib，它是 C++ 目标。相反，对于 myapp2，第一次求值将给出 C 作为链接语言，因此第二次通过将正确地将目标 libother 添加为链接依赖项。

$<LINK_LANG_AND_ID:language,compiler_ids>¶

在 3.18 版本中新增。

当用于链接步骤的语言与 language 匹配，且 CMake 的该语言链接器 ID 与 compiler_ids 中的任意一个逗号分隔条目匹配时，返回 1，否则返回 0。此表达式是 $<LINK_LANGUAGE:language> 和 $<LANG_COMPILER_ID:compiler_ids> 组合的简写形式。此表达式可用于为目标中特定语言和链接器组合指定链接库、链接选项、链接目录和链接依赖项。例如

add_library(libC_Clang ...)
add_library(libCXX_Clang ...)
add_library(libC_Intel ...)
add_library(libCXX_Intel ...)

add_executable(myapp main.c)
if (CXX_CONFIG)
  target_sources(myapp PRIVATE file.cxx)
endif()
target_link_libraries(myapp
  PRIVATE $<$<LINK_LANG_AND_ID:CXX,Clang,AppleClang>:libCXX_Clang>
          $<$<LINK_LANG_AND_ID:C,Clang,AppleClang>:libC_Clang>
          $<$<LINK_LANG_AND_ID:CXX,Intel>:libCXX_Intel>
          $<$<LINK_LANG_AND_ID:C,Intel>:libC_Intel>)

代码说明

  • add_library(libC_Clang ...) add_library(libCXX_Clang ...) add_library(libC_Intel ...) add_library(libCXX_Intel ...) add_executable(myapp main.c) if (CXX_CONFIG) target_sources(myapp PRIVATE file.cxx) endif() target_link_libraries(myapp PRIVATE $<$<LINK_LANG_AND_ID:CXX,Clang,AppleClang>:libCXX_Clang> $<$<LINK_LANG_AND_ID:C,Clang,AppleClang>:libC_Clang> $<$<LINK_LANG_AND_ID:CXX,Intel>:libCXX_Intel> $<$<LINK_LANG_AND_ID:C,Intel>:libC_Intel>)：调用 `add_library()` 命令，定义一个库目标。

这指定基于编译器 ID 和链接语言使用不同的链接库。在此示例中，当 Clang 或 AppleClang 是 CXX 链接器时，会将目标 libCXX_Clang 添加为链接依赖项；当 Intel 是 CXX 链接器时，使用 libCXX_Intel。同样，当 C 链接器是 Clang 或 AppleClang 时，目标 libC_Clang 将被添加为链接依赖项；当 Intel 是 C 链接器时，添加 libC_Intel。

有关使用此生成器表达式的限制，请参阅 与 $<LINK_LANGUAGE:language> 相关的说明。

链接特性 (Link Features)¶

$<LINK_LIBRARY:feature,library-list>¶

在 3.24 版本中添加。

指定一组要链接到目标的内容，以及一个提供关于如何链接它们的信息的 feature。例如

add_library(lib1 STATIC ...)
add_library(lib2 ...)
target_link_libraries(lib2 PRIVATE "$<LINK_LIBRARY:WHOLE_ARCHIVE,lib1>")

代码说明

  • add_library(lib1 STATIC ...) add_library(lib2 ...) target_link_libraries(lib2 PRIVATE "$<LINK_LIBRARY:WHOLE_ARCHIVE,lib1>")：调用 `add_library()` 命令，定义一个库目标。

这指定 lib2 应链接到 lib1，并在此过程中使用 WHOLE_ARCHIVE 特性。

特性名称区分大小写，只能包含字母、数字和下划线。全大写的特性名称为 CMake 内置特性保留。预定义的内置库特性有

DEFAULT

此功能对应于标准链接，本质上等同于不使用任何功能。它通常仅与 LINK_LIBRARY_OVERRIDE 和 LINK_LIBRARY_OVERRIDE_<LIBRARY> 目标属性一起使用。

WHOLE_ARCHIVE

当链接为 可执行文件、共享库 和 模块库 的依赖项时，强制包含静态库的所有成员。此特性仅在以下平台受支持，并有如下说明的限制

  • Linux。

  • 所有 BSD 变体。

  • SunOS。

  • 所有 Apple 变体。库必须指定为 CMake 目标名称、库文件名（例如 libfoo.a）或库文件路径（例如 /path/to/libfoo.a）。由于 Apple 链接器的限制，它不能指定为纯库名称，例如 foo（其中 foo 不是 CMake 目标）。

  • Windows（微软桌面系统）。使用 MSVC 或类似 MSVC 工具链时，MSVC 版本必须大于 1900。

  • Cygwin。

  • MSYS。

注意

由于 静态库 是存档而非链接的二进制文件，CMake 在链接消费二进制文件时记录其链接依赖项以供传递使用。因此，WHOLE_ARCHIVE 不会导致静态库的对象被包含在其他静态库中。请为此使用 对象库。

FRAMEWORK

此选项告诉链接器使用 -framework 链接器选项搜索指定的框架。它只能在 Apple 平台上使用，并且只能与理解该选项的链接器一起使用（即 Xcode 提供的链接器，或与之兼容的链接器）。

框架可以指定为 CMake 框架目标、裸框架名称或文件路径。如果给定一个目标，则该目标必须具有 FRAMEWORK 目标属性设置为 true。对于文件路径，如果它包含目录部分，则该目录将被添加为框架搜索路径。

add_library(lib SHARED ...)
target_link_libraries(lib PRIVATE "$<LINK_LIBRARY:FRAMEWORK,/path/to/my_framework>")

# The constructed linker command line will contain:
#   -F/path/to -framework my_framework

代码说明

  • add_library(lib SHARED ...) target_link_libraries(lib PRIVATE "$<LINK_LIBRARY:FRAMEWORK,/path/to/my_framework>") # The constructed linker command line will contain: # -F/path/to -framework my_framework：这一行继续补充当前命令或表达式的内容。

文件路径必须符合以下模式之一（* 是通配符，可选部分显示为 [...]）：

  • [/path/to/]FwName[.framework]

  • [/path/to/]FwName.framework/FwName[suffix]

  • [/path/to/]FwName.framework/Versions/*/FwName[suffix]

请注意，即使不使用 $<LINK_LIBRARY:FRAMEWORK,...> 表达式，CMake 也能识别并自动处理框架目标。如果项目希望明确指定，仍可将该生成器表达式与 CMake 目标一起使用，但不是必须的。链接器命令行在使用或不使用该生成器表达式之间可能会有一些差异，但最终结果应该是相同的。另一方面，如果给出了文件路径，CMake 会自动识别某些路径，但不能处理所有情况。项目可能希望将 $<LINK_LIBRARY:FRAMEWORK,...> 用于文件路径，以便明确预期的行为。

版本 3.25 中添加: FRAMEWORK_MULTI_CONFIG_POSTFIX_<CONFIG> 目标属性以及框架库名称的 suffix 现在受 FRAMEWORK 功能支持。

NEEDED_FRAMEWORK

这类似于 FRAMEWORK 功能，除了它强制链接器链接框架，即使没有使用它的任何符号。它使用 -needed_framework 选项，并且具有与 FRAMEWORK 相同的链接器约束。

REEXPORT_FRAMEWORK

这类似于 FRAMEWORK 功能，除了它告诉链接器框架应该可供链接到正在创建的库的客户端使用。它使用 -reexport_framework 选项，并且具有与 FRAMEWORK 相同的链接器约束。

WEAK_FRAMEWORK

这类似于 FRAMEWORK 功能，除了它强制链接器将框架及其所有引用标记为弱导入。它使用 -weak_framework 选项，并且具有与 FRAMEWORK 相同的链接器约束。

NEEDED_LIBRARY

这类似于 NEEDED_FRAMEWORK 功能，除了它用于非框架目标或库（仅限 Apple 平台）。它使用适当的 -needed_library 或 -needed-l 选项，并且具有与 NEEDED_FRAMEWORK 相同的链接器约束。

REEXPORT_LIBRARY

这类似于 REEXPORT_FRAMEWORK 功能，除了它用于非框架目标或库（仅限 Apple 平台）。它使用适当的 -reexport_library 或 -reexport-l 选项，并且具有与 REEXPORT_FRAMEWORK 相同的链接器约束。

WEAK_LIBRARY

这类似于 WEAK_FRAMEWORK 功能，除了它用于非框架目标或库（仅限 Apple 平台）。它使用适当的 -weak_library 或 -weak-l 选项，并且具有与 WEAK_FRAMEWORK 相同的链接器约束。

  • CMAKE_<LANG>_LINK_LIBRARY_USING_<FEATURE>

  • CMAKE_LINK_LIBRARY_USING_<FEATURE>_SUPPORTED

  • CMAKE_LINK_LIBRARY_USING_<FEATURE>

  1. 如果特定语言的 CMAKE_<LANG>_LINK_LIBRARY_USING_<FEATURE>_SUPPORTED 变量为真，则该 feature 必须由相应的 CMAKE_<LANG>_LINK_LIBRARY_USING_<FEATURE> 变量定义。

  2. 如果不支持特定语言的 feature，则 CMAKE_LINK_LIBRARY_USING_<FEATURE>_SUPPORTED 变量必须为真，且该 feature 必须由相应的 CMAKE_LINK_LIBRARY_USING_<FEATURE> 变量定义。

应注意以下限制

  • library-list 可以指定 CMake 目标或库。任何类型为 OBJECT 或 INTERFACE 的 CMake 目标将忽略该表达式的特性方面，并以标准方式链接。

  • $<LINK_LIBRARY:...> 生成器表达式仅能用于指定链接库。实际上，这意味着它只能出现在 LINK_LIBRARIES、INTERFACE_LINK_LIBRARIES 和 INTERFACE_LINK_LIBRARIES_DIRECT 目标属性中，并在 target_link_libraries() 和 link_libraries() 命令中指定。

  • 如果 $<LINK_LIBRARY:...> 生成器表达式出现在目标的 INTERFACE_LINK_LIBRARIES 属性中，它将被包含在由 install(EXPORT) 命令生成的导入目标中。消费此导入的环境有责任定义此表达式使用的链接特性。

  • 链接步骤中涉及的每个目标或库最多只能有一种库特性。缺少特性也与其他所有特性不兼容。例如

    add_library(lib1 ...)

# lib1 will be associated with feature1
target_link_libraries(lib2 PUBLIC "$<LINK_LIBRARY:feature1,lib1>")

# use of feature1 in the line above and would result in an error.
target_link_libraries(lib3 PRIVATE lib1 lib2)

    在无法为给定目标或库在整个构建过程中使用相同特性的情况下，可以使用 LINK_LIBRARY_OVERRIDE 和 LINK_LIBRARY_OVERRIDE_<LIBRARY> 目标属性来解决此类不兼容性。

  • $<LINK_LIBRARY:...> 生成器表达式不能保证指定的目标和库列表将保持分组在一起。要管理 GNU ld 链接器支持的 --start-group 和 --end-group 等构造，请改用 LINK_GROUP 生成器表达式。

$<LINK_GROUP:feature,library-list>¶

在 3.24 版本中添加。

指定一组要链接到目标的内容，以及一个定义该组应如何链接的 feature。例如

add_library(lib1 STATIC ...)
add_library(lib2 ...)
target_link_libraries(lib2 PRIVATE "$<LINK_GROUP:RESCAN,lib1,external>")

代码说明

  • add_library(lib1 STATIC ...) add_library(lib2 ...) target_link_libraries(lib2 PRIVATE "$<LINK_GROUP:RESCAN,lib1,external>")：调用 `add_library()` 命令，定义一个库目标。

这指定 lib2 应链接到 lib1 和 external，并且这两个库都应根据 RESCAN 特性的定义包含在链接器命令行中。

特性名称区分大小写，只能包含字母、数字和下划线。全大写的特性名称为 CMake 内置特性保留。目前，只有一个预定义的内置组特性。

RESCAN

有些链接器是单次扫描的。对于这些链接器，库之间的循环引用通常会导致未解析的符号。此特性指示链接器反复搜索指定的静态库，直到没有新的未定义引用被创建。

通常，静态库只在命令行上指定的顺序中被搜索一次。如果该库中的某个符号需要解析由命令行上稍后出现的库中的对象引用的未定义符号，链接器将无法解析该引用。通过使用 RESCAN 特性对静态库进行分组，它们将被反复搜索，直到所有可能的引用都被解析。这将使用诸如 --start-group 和 --end-group 的链接器选项，或者在 SunOS 上使用 -z rescan-start 和 -z rescan-end。

使用此功能会带来显著的性能成本。最好仅在两个或多个静态库之间存在不可避免的循环引用时才使用它。

此功能在使用针对 Linux、BSD 和 SunOS 的工具链时可用。在使用 GNU 工具链时，也可以在针对 Windows（微软桌面系统） 平台时使用。

内置和自定义组特性通过以下变量进行定义

  • CMAKE_<LANG>_LINK_GROUP_USING_<FEATURE>_SUPPORTED

  • CMAKE_<LANG>_LINK_GROUP_USING_<FEATURE>

  • CMAKE_LINK_GROUP_USING_<FEATURE>_SUPPORTED

  • CMAKE_LINK_GROUP_USING_<FEATURE>

每个这些变量的值都是在创建该目标所在的目录范围结束时所设置的值。用法如下

  1. 如果特定语言的 CMAKE_<LANG>_LINK_GROUP_USING_<FEATURE>_SUPPORTED 变量为真，则该 feature 必须由相应的 CMAKE_<LANG>_LINK_GROUP_USING_<FEATURE> 变量定义。

  2. 如果不支持特定语言的 feature，则 CMAKE_LINK_GROUP_USING_<FEATURE>_SUPPORTED 变量必须为真，且该 feature 必须由相应的 CMAKE_LINK_GROUP_USING_<FEATURE> 变量定义。

LINK_GROUP 生成器表达式与 LINK_LIBRARY 生成器表达式兼容。组中涉及的库可以使用 LINK_LIBRARY 生成器表达式来指定。

链接步骤中涉及的每个目标或外部库允许属于多个组，但前提是所有涉及的组指定相同的 feature。此类组不会在链接器命令行上合并，单独的组仍将保留。禁止为同一个目标或库混合使用不同的组特性。

add_library(lib1 ...)
add_library(lib2 ...)
add_library(lib3 ...)
add_library(lib4 ...)
add_library(lib5 ...)

target_link_libraries(lib3 PUBLIC  "$<LINK_GROUP:feature1,lib1,lib2>")
target_link_libraries(lib4 PRIVATE "$<LINK_GROUP:feature1,lib1,lib3>")
# lib4 will be linked with the groups {lib1,lib2} and {lib1,lib3}.
# Both groups specify the same feature, so this is fine.

target_link_libraries(lib5 PRIVATE "$<LINK_GROUP:feature2,lib1,lib3>")
# An error will be raised here because both lib1 and lib3 are part of two
# groups with different features.

代码说明

  • add_library(lib1 ...) add_library(lib2 ...) add_library(lib3 ...) add_library(lib4 ...) add_library(lib5 ...) target_link_libraries(lib3 PUBLIC "$<LINK_GROUP:feature1,lib1,lib2>") target_link_libraries(lib4 PRIVATE "$<LINK_GROUP:feature1,lib1,lib3>") # lib4 will be linked with the groups {lib1,lib2} and {lib1,lib3}. # Both groups specify the same feature, so this is fine. target_link_libraries(lib5 PRIVATE "$<LINK_GROUP:feature2,lib1,lib3>") # An error will be raised here because both lib1 and lib3 are part of two # groups with different features.：这一行继续补充当前命令或表达式的内容。

  • add_library(lib1 ...) add_library(lib2 ...) add_library(lib3 ...) add_library(lib4 ...) target_link_libraries(lib3 PUBLIC lib1) target_link_libraries(lib4 PRIVATE lib3 "$<LINK_GROUP:feature1,lib1,lib2>") # lib4 will only be linked with lib3 and the group {lib1,lib2}：这一行继续补充当前命令或表达式的内容。

因为 lib1 是为 lib4 定义的组的一部分，该组随后被应用于 lib3 对 lib1 的使用。最终结果将如同 lib3 的链接关系被指定为

target_link_libraries(lib3 PUBLIC "$<LINK_GROUP:feature1,lib1,lib2>")

代码说明

  • target_link_libraries(lib3 PUBLIC "$<LINK_GROUP:feature1,lib1,lib2>")：调用 `target_link_libraries()` 命令，为目标链接依赖库。

add_library(lib2A ...)
add_library(lib2B ...)
add_library(lib3 ...)

target_link_libraries(lib2B PUBLIC lib1B)

  "$<LINK_GROUP:feat,lib2A,lib2B>"
)

  • add_library(lib1A ...) add_library(lib1B ...) add_library(lib2A ...) add_library(lib2B ...) add_library(lib3 ...) # Non-group linking relationships, these are non-circular so far target_link_libraries(lib1A PUBLIC lib2A) target_link_libraries(lib2B PUBLIC lib1B) # The addition of these groups creates circular dependencies target_link_libraries(lib3 PRIVATE "$<LINK_GROUP:feat,lib1A,lib1B>" "$<LINK_GROUP:feat,lib2A,lib2B>" )：调用 `add_library()` 命令，定义一个库目标。

target_link_libraries(lib1A PUBLIC "$<LINK_GROUP:feat,lib2A,lib2B>")
target_link_libraries(lib2B PUBLIC "$<LINK_GROUP:feat,lib1A,lib1B>")

代码说明

  • target_link_libraries(lib1A PUBLIC "$<LINK_GROUP:feat,lib2A,lib2B>") target_link_libraries(lib2B PUBLIC "$<LINK_GROUP:feat,lib1A,lib1B>")：调用 `target_link_libraries()` 命令，为目标链接依赖库。

链接上下文 (Link Context)¶

$<LINK_ONLY:...>¶

版本 3.1 中新增。

... 的内容，除非在收集 传递编译属性 的使用要求时，在这种情况下它为空字符串。这旨在用于 INTERFACE_LINK_LIBRARIES 目标属性中（通常通过 target_link_libraries() 命令填充），以指定私有链接依赖项，而无需其他使用要求（例如包含目录或编译选项）。

3.24 版本新增：LINK_ONLY 也可用于 LINK_LIBRARIES 目标属性。请参阅策略 CMP0131。

$<DEVICE_LINK:list>¶

在 3.18 版本中新增。

如果是设备链接步骤，则返回列表，否则返回空列表。设备链接步骤由 CUDA_SEPARABLE_COMPILATION 和 CUDA_RESOLVE_DEVICE_SYMBOLS 属性以及策略 CMP0105 控制。此表达式仅能用于指定链接选项。

$<HOST_LINK:list>¶

在 3.18 版本中新增。

如果是正常的链接步骤，则返回列表，否则返回空列表。此表达式主要在涉及设备链接步骤时有用（参见 $<DEVICE_LINK:list> 生成器表达式）。此表达式仅能用于指定链接选项。

源相关表达式¶

以下表达式中指定的源文件在文件系统上可能不存在（即生成的文件），但必须被 CMake 所知。如果源文件是某个目标（库或可执行文件）的一部分，或者定义了源文件属性，则该源文件就会被 CMake 所知。此外，此信息仅限于声明发生的目录。

例如，对于 add_custom_command() 和 add_custom_target() 命令，这些生成器表达式可以针对源属性提供统一的行为。

function(custom_add_library target)
  unset(sources)
  foreach(source IN LISTS ARGN)
    add_custom_command(
      OUTPUT ${CMAKE_CURRENT_BINARY_DIR}/${source}.bin
      COMMAND my-compiler -o ${CMAKE_CURRENT_BINARY_DIR}/${source}.bin
                          "$<$<SOURCE_EXISTS:${source}>:$<SOURCE_PROPERTY:${source},COMPILE_OPTIONS>>"
                          ${source})
    list(APPEND sources ${CMAKE_CURRENT_BINARY_DIR}/${source}.bin)
  endforeach()
  add_custom_target(${target}
                    DEPENDS ${sources})
endfunction()

custom_add_library(my-lib file1.x file2.x file3.x)
set_property(SOURCE file1.x PROPERTY COMPILE_OPTIONS -X)
set_property(SOURCE file2.x PROPERTY COMPILE_OPTIONS -Y)

这些表达式对于某些属性具有特殊的评估规则：

目标构建规范属性

这些属性评估为一个 分号分隔列表，表示目标本身的值与目标 LINK_LIBRARIES 所命名的目标的相应 目标使用要求 的并集。

  • 对于 目标编译属性，相应使用要求的评估是在已链接目标的 INTERFACE_LINK_LIBRARIES 的闭包上进行传递的，排除受 LINK_ONLY 生成器表达式保护的条目。

  • 对于 目标链接属性，相应使用要求的评估是在已链接目标的 INTERFACE_LINK_LIBRARIES 的闭包上进行传递的，包括受 LINK_ONLY 生成器表达式保护的条目。参见策略 CMP0166。

4.1 版本变更： LINK_LIBRARIES 本身的评估现在是传递性的。参见策略 CMP0189。

目标使用要求属性

这些属性评估为一个 分号分隔列表，表示目标本身的值与目标 INTERFACE_LINK_LIBRARIES 所命名的目标的相同属性的值的并集。

  • 对于 传递编译属性，评估是在目标的 INTERFACE_LINK_LIBRARIES 的闭包上进行传递的，排除受 LINK_ONLY 生成器表达式保护的条目。

  • 对于 传递链接属性，评估是在目标的 INTERFACE_LINK_LIBRARIES 的闭包上进行传递的，包括受 LINK_ONLY 生成器表达式保护的条目。参见策略 CMP0166。

4.1 版本变更： INTERFACE_LINK_LIBRARIES 本身的评估现在是传递性的。参见策略 CMP0189。

自定义传递属性

3.30 版本新增。

这些属性在评估期间处理如下：

  • 评估 $<TARGET_PROPERTY:tgt,PROP> 时（对于某些没有 INTERFACE_ 前缀的属性 PROP），会检查目标 tgt 的 TRANSITIVE_COMPILE_PROPERTIES 和 TRANSITIVE_LINK_PROPERTIES 属性，以及目标 tgt 的 LINK_LIBRARIES 所命名的目标，以及已链接目标的 INTERFACE_LINK_LIBRARIES 的传递闭包上的属性。

    如果 PROP 被这些属性之一列出，则它评估为一个 分号分隔列表，表示目标本身的值与目标 LINK_LIBRARIES 所命名的目标的相应 INTERFACE_PROP 的值的并集。

    • 如果 PROP 由 TRANSITIVE_COMPILE_PROPERTIES 指定，则相应 INTERFACE_PROP 的评估是在已链接目标的 INTERFACE_LINK_LIBRARIES 的闭包上进行传递的，排除受 LINK_ONLY 生成器表达式保护的条目。

    • 如果 PROP 由 TRANSITIVE_LINK_PROPERTIES 指定，则相应 INTERFACE_PROP 的评估是在已链接目标的 INTERFACE_LINK_LIBRARIES 的闭包上进行传递的，包括受 LINK_ONLY 生成器表达式保护的条目。

  • 评估 $<TARGET_PROPERTY:tgt,INTERFACE_PROP> 时（对于某些带有 INTERFACE_ 前缀的属性 INTERFACE_PROP），会检查目标 tgt 的 TRANSITIVE_COMPILE_PROPERTIES 和 TRANSITIVE_LINK_PROPERTIES 属性，以及目标 tgt 的 INTERFACE_LINK_LIBRARIES 所命名的目标的传递闭包。

    如果相应的 PROP 被这些属性之一列出，则 INTERFACE_PROP 评估为一个 分号分隔列表，表示目标本身的值与目标 INTERFACE_LINK_LIBRARIES 所命名的目标的相同属性的值的并集。

    • 如果 PROP 由 TRANSITIVE_COMPILE_PROPERTIES 指定，则相应 INTERFACE_PROP 的评估是在目标 INTERFACE_LINK_LIBRARIES 的闭包上进行传递的，排除受 LINK_ONLY 生成器表达式保护的条目。

    • 如果 PROP 由 TRANSITIVE_LINK_PROPERTIES 指定，则相应 INTERFACE_PROP 的评估是在目标 INTERFACE_LINK_LIBRARIES 的闭包上进行传递的，包括受 LINK_ONLY 生成器表达式保护的条目。

如果 PROP 同时被 TRANSITIVE_COMPILE_PROPERTIES 和 TRANSITIVE_LINK_PROPERTIES 指定，则后者优先。

兼容接口属性

这些属性评估为一个从目标本身、目标 LINK_LIBRARIES 所命名的目标，以及已链接目标的 INTERFACE_LINK_LIBRARIES 的传递闭包中组合而成的单一值。来自多个目标的兼容接口属性值会根据定义它的 COMPATIBLE_INTERFACE_* 属性所要求的兼容性类型进行组合。

$<TARGET_RUNTIME_DLL（动态链接库）S:tgt>¶

3.21 版本新增。

目标在运行时所依赖的 DLL（动态链接库） 列表。这是由目标传递依赖项中所有 SHARED 目标的位置确定的。如果只需要 DLL（动态链接库） 的目录，请参阅 TARGET_RUNTIME_DLL（动态链接库）_DIRS 生成器表达式。在可执行文件、SHARED 库和 MODULE 库以外的目标上使用此生成器表达式是错误的。在非 DLL（动态链接库） 平台上，此表达式总是计算为空字符串。

此生成器表达式可用于在 POST_BUILD 自定义命令中使用 cmake -E copy -t 命令将目标依赖的所有 DLL（动态链接库） 复制到其输出目录。例如

find_package(foo CONFIG REQUIRED) # package generated by install(EXPORT)

add_executable(exe main.c)
target_link_libraries(exe PRIVATE foo::foo foo::bar)
add_custom_command(TARGET exe POST_BUILD
  COMMAND ${CMAKE_COMMAND} -E copy -t $<TARGET_FILE_DIR:exe> $<TARGET_RUNTIME_DLLS:exe>
  COMMAND_EXPAND_LISTS
)

代码说明

  • find_package(foo CONFIG REQUIRED) # package generated by install(EXPORT) add_executable(exe main.c) target_link_libraries(exe PRIVATE foo::foo foo::bar) add_custom_command(TARGET exe POST_BUILD COMMAND ${CMAKE_COMMAND} -E copy -t $<TARGET_FILE_DIR:exe> $<TARGET_RUNTIME_DLLS:exe> COMMAND_EXPAND_LISTS )：调用 `find_package()` 命令，查找并加载外部依赖包。

注意

导入的目标 仅在它们知道其 .dll 文件位置时才受支持。导入的 SHARED 库必须设置 IMPORTED_LOCATION 为其 .dll 文件。详情参见 add_library 导入库 部分。许多 查找模块 产生类型为 UNKNOWN 的导入目标，因此会被忽略。

在支持运行时路径 (RPATH) 的平台上，请参阅 INSTALL_RPATH 目标属性。在 Apple 平台上，请参阅 INSTALL_NAME_DIR 目标属性。

多级表达式求值¶

$<GENEX_EVAL:expr>¶

3.12 版本新增。

在当前上下文中作为生成器表达式求值的 expr 的内容。这允许消费其求值结果本身也是生成器表达式的生成器表达式。

$<TARGET_GENEX_EVAL:tgt,expr>¶

3.12 版本新增。

在 tgt 目标的上下文中作为生成器表达式求值的 expr 的内容。这允许消费自身包含生成器表达式的自定义目标属性。

当您想要管理支持生成器表达式的自定义属性时，拥有求值生成器表达式的能力非常有用。例如

add_library(foo ...)

set_property(TARGET foo PROPERTY
  CUSTOM_KEYS $<$<CONFIG:DEBUG>:FOO_EXTRA_THINGS>
)

add_custom_target(printFooKeys
  COMMAND ${CMAKE_COMMAND} -E echo $<TARGET_PROPERTY:foo,CUSTOM_KEYS>
)

代码说明

  • add_library(foo ...) set_property(TARGET foo PROPERTY CUSTOM_KEYS $<$<CONFIG:DEBUG>:FOO_EXTRA_THINGS> ) add_custom_target(printFooKeys COMMAND ${CMAKE_COMMAND} -E echo $<TARGET_PROPERTY:foo,CUSTOM_KEYS> )：调用 `add_library()` 命令，定义一个库目标。

这种对 printFooKeys 自定义命令的简单实现是错误的，因为 CUSTOM_KEYS 目标属性没有被求值，且内容按原样传递（即 $<$<CONFIG:DEBUG>:FOO_EXTRA_THINGS>）。

为了获得预期的结果（即如果配置为 Debug 则为 FOO_EXTRA_THINGS），需要求值 $<TARGET_PROPERTY:foo,CUSTOM_KEYS> 的输出。

add_custom_target(printFooKeys
  COMMAND ${CMAKE_COMMAND} -E
    echo $<TARGET_GENEX_EVAL:foo,$<TARGET_PROPERTY:foo,CUSTOM_KEYS>>
)

代码说明

  • add_custom_target(printFooKeys COMMAND ${CMAKE_COMMAND} -E echo $<TARGET_GENEX_EVAL:foo,$<TARGET_PROPERTY:foo,CUSTOM_KEYS>> )：调用 `add_custom_target()` 命令，定义一个没有直接输出文件的自定义目标。

转义字符¶

这些表达式求值为特定的字符串字面量。在需要防止它们产生特殊含义的地方，使用它们来代替实际的字符串字面量。

$<ANGLE-R>¶

字面量 >。例如，用于比较包含 > 的字符串。

$<COMMA>¶

字面量 ,。例如，用于比较包含 , 的字符串。

$<SEMICOLON>¶

字面量 ;。用于防止在包含 ; 的参数上进行列表扩展。

$<QUOTE>¶

3.30 版本新增。

字面量 "。用于允许在生成器表达式中使用字符串字面量引号。
