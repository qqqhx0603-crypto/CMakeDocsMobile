简介¶

本手册旨在供使用 cmake-language(7) 代码的开发人员参考，无论他们是在编写自己的模块、构建自己的构建系统，还是参与 CMake 本身的开发。

请参阅 https://cmake.com.cn/get-involved/ 以参与 CMake 上游开发。其中包含贡献指南的链接，该指南继而链接到 CMake 本身的开发人员指南。

访问 Windows 注册表¶

CMake 提供了一些在 Windows 平台上访问注册表的工具。

查询 Windows 注册表¶

在 3.24 版本中添加。

cmake_host_system_information() 命令提供了查询本地计算机注册表的可能性。有关更多信息，请参阅 cmake_host_system_information(QUERY WINDOWS_REGISTRY) 子命令。

使用 Windows 注册表查找¶

版本 3.24 更改。

find_file()、find_library()、find_path()、find_program() 和 find_package() 命令的 HINTS 和 PATHS 选项提供了在 Windows 平台上查询注册表的可能性。

注册表查询的正式语法（使用带有常规扩展的 BNF 表示法指定）如下

registry_query  ::= '[' sep_definition? root_key
                    ((key_separator sub_key)? (value_separator value_name_)?)? ']'
sep_definition  ::= '{' value_separator '}'
root_key        ::= 'HKLM' | 'HKEY_LOCAL_MACHINE' | 'HKCU' | 'HKEY_CURRENT_USER' |
                    'HKCR' | 'HKEY_CLASSES_ROOT' | 'HKCC' | 'HKEY_CURRENT_CONFIG' |
                    'HKU' | 'HKEY_USERS'
sub_key         ::= element (key_separator element)*
key_separator   ::= '/' | '\\'
value_separator ::= element | ';'
value_name      ::= element | '(default)'
element         ::= character\+
character       ::= <any character except key_separator and value_separator>

sep_definition 可选项目提供了指定用于分隔 sub_key 和 value_name 项的字符串的可能性。如果未指定，则使用字符 ;。多个 registry_query 项可以指定为路径的一部分。

# example using default separator
find_file(... PATHS "/root/[HKLM/Stuff;InstallDir]/lib[HKLM\\\\Stuff;Architecture]")

# example using different specified separators
find_library(... HINTS "/root/[{|}HKCU/Stuff|InstallDir]/lib[{@@}HKCU\\\\Stuff@@Architecture]")

如果未指定 value_name 项，或者具有特殊名称 (default)，则将返回默认值的内容（如果有）。支持的 value_name 类型为

  • REG_SZ.

  • REG_EXPAND_SZ。返回的数据会经过扩展。

  • REG_DWORD.

  • REG_QWORD.

当注册表查询失败时（通常是因为键不存在或数据类型不受支持），字符串 /REGISTRY-NOTFOUND 将替换为 [] 查询表达式。

查找模块¶

“查找模块”是一个 Find<PackageName>.cmake 文件，当为 <PackageName> 调用 find_package() 命令时加载。

查找模块的主要任务是确定包是否可用，设置 <PackageName>_FOUND 变量以反映此状态，并提供使用该包所需的任何变量、宏和导入目标。查找模块在某些上游库不提供 配置文件包 (config file package) 的情况下非常有用。

传统的方法是为所有内容（包括库和可执行文件）使用变量：请参阅下文的“标准变量名称”部分。这是大多数 CMake 提供的现有查找模块所采用的做法。

更现代的做法是尽可能像 配置文件包 一样，通过提供 导入目标 (imported target) 来运行。这样做的好处是向消费者传播 使用需求 (usage requirements)。

无论哪种情况（甚至在同时提供变量和导入目标时），查找模块都应与具有相同名称的旧版本保持向后兼容性。

FindFoo.cmake 模块通常由以下命令加载

find_package(Foo [major[.minor[.patch[.tweak]]]]
             [EXACT] [QUIET] [REQUIRED]
             [[COMPONENTS] [components...]]
             [OPTIONAL_COMPONENTS components...]
             [NO_POLICY_SCOPE])

有关查找模块设置了哪些变量的详细信息，请参阅 find_package() 文档。其中大多数通过使用 FindPackageHandleStandardArgs 来处理。

简而言之，模块应该只定位与请求版本兼容的包版本，如 Foo_FIND_VERSION 系列变量所描述的那样。如果 Foo_FIND_QUIETLY 设置为 true，它应该避免打印消息，包括任何关于未找到包的抱怨。如果 Foo_FIND_REQUIRED 设置为 true，如果找不到包，模块应该发出 FATAL_ERROR。如果两者都没有设置为 true，则在找不到包时应打印非致命消息。

查找多个半独立部分（如库包）的包应该搜索 Foo_FIND_COMPONENTS 中列出的组件（如果已设置），并且仅在对于每个未找到的搜索组件 <c>，Foo_FIND_REQUIRED_<c> 未设置为 true 时，才将 Foo_FOUND 设置为 true。find_package_handle_standard_args() 的 HANDLE_COMPONENTS 参数可用于实现此功能。

如果未设置 Foo_FIND_COMPONENTS，则搜索和要求哪些模块由查找模块决定，但应记录在案。

对于内部实现，一个普遍接受的约定是以下划线开头的变量仅供临时使用。

标准变量名称¶

对于采用设置变量方法（除了创建导入目标之外或代替创建导入目标）的 FindXxx.cmake 模块，应使用以下变量名称，以保持查找模块之间的一致性。请注意，所有变量都以 Xxx_ 开头，该前缀（除非另有说明）必须与 FindXxx.cmake 文件的名称完全匹配，包括大小写。变量名上的此前缀可确保它们不会与其他查找模块的变量发生冲突。查找模块定义的任何宏、函数和导入目标也应遵循相同的模式。

Xxx_INCLUDE_DIRS

在一个变量中列出的最终包含目录集，供客户端代码使用。这不应是缓存条目（请注意，这也意味着该变量不应作为 find_path() 命令的结果变量使用——请参阅下文的 Xxx_INCLUDE_DIR）。

Xxx_LIBRARIES

与模块一起使用的库。这些可以是 CMake 目标、库二进制文件的完整绝对路径，或者是链接器必须在其搜索路径中找到的库名称。这不应是缓存条目（请注意，这也意味着该变量不应作为 find_library() 命令的结果变量使用——请参阅下文的 Xxx_LIBRARY）。

Xxx_DEFINITIONS

编译使用该模块的代码时要使用的编译定义。这实际上不应包含诸如 -DHAS_JPEG 之类的选项，这些选项供客户端源代码文件决定是否 #include <jpeg.h>。

Xxx_EXECUTABLE

可执行文件的完整绝对路径。在这种情况下，Xxx 可能不是模块的名称，它可能是工具的名称（通常转换为全大写），前提是该工具拥有一个众所周知的名称，不太可能存在具有相同名称的其他工具。将其用作 find_program() 命令的结果变量是合适的。

Xxx_YYY_EXECUTABLE

类似于 Xxx_EXECUTABLE，但此处 Xxx 始终是模块名称，而 YYY 是工具名称（同样，通常为全大写）。如果工具名称不是非常广为人知或者有可能与另一个工具冲突，请首选这种形式。为了获得更高的一致性，如果模块提供多个可执行文件，也请首选这种形式。

Xxx_LIBRARY_DIRS

可选地，在一个变量中列出的最终库目录集，供客户端代码使用。这不应是缓存条目。

Xxx_ROOT_DIR

Xxx 安装的基本目录，如果找到了 Xxx，可以由查找模块选择性地设置。这对于大型包非常有用，因为许多文件需要相对于公共基础（或根）目录进行引用。不要与外部设置的 Xxx_ROOT 提示变量混淆，后者用于让查找模块知道在哪里查找 Xxx。

Xxx_VERSION_VV

此形式的变量指定正在提供的 Xxx 模块是否为该模块的 VV 版本。对于给定模块，不应将多个此类形式的变量设置为 true。例如，模块 Barry 可能已经发展多年并经历了多个不同的主要版本。Barry 模块的第 3 版可能会将变量 Barry_VERSION_3 设置为 true，而旧版本的模块可能会改为将 Barry_VERSION_2 设置为 true。将 Barry_VERSION_3 和 Barry_VERSION_2 同时设置为 true 将是错误的。

Xxx_WRAP_YY

当此形式的变量设置为 false 时，它表示不应使用相关的包装命令。包装命令取决于模块，它可能隐含在模块名称中，也可能由变量的 YY 部分指定。

Xxx_Yy_FOUND

对于此形式的变量，Yy 是模块组件的名称。它应该完全匹配可以传递给模块的 find_package() 命令的有效组件名称之一。如果此形式的变量设置为 false，则表示模块 Xxx 的 Yy 组件未找到或不可用。此形式的变量通常用于可选组件，以便调用者可以检查可选组件是否可用。

Xxx_FOUND

当 find_package() 命令返回给调用者时，如果模块被视为成功找到，此变量将被设置为 true。

Xxx_NOT_FOUND_MESSAGE

应由配置文件在将 Xxx_FOUND 设置为 FALSE 的情况下进行设置。包含的消息将由 find_package() 命令和 find_package_handle_standard_args() 打印，以通知用户有关问题。使用此方法代替直接调用 message() 来报告未能找到模块或包的原因。

Xxx_RUNTIME_LIBRARY_DIRS

可选地，运行链接到共享库的可执行文件时使用的运行时库搜索路径。列表应由用户代码用于创建 Windows 上的 PATH 或 UNIX 上的 LD_LIBRARY_PATH。这不应是缓存条目。

Xxx_VERSION

找到的包的完整版本字符串（如果有）。请注意，一些现有的模块也可能提供 Xxx_VERSION_STRING，因为它在当前命名约定之前被传统使用。

Xxx_VERSION_MAJOR

找到的包的主版本（如果有）。

Xxx_VERSION_MINOR

找到的包的次版本（如果有）。

Xxx_VERSION_PATCH

找到的包的补丁版本（如果有）。

以下名称通常不应在 CMakeLists.txt 文件中使用。它们旨在供查找模块使用，以指定和缓存特定文件或目录的位置。用户通常可以设置和编辑这些变量以控制查找模块的行为（例如手动输入库的路径）

Xxx_LIBRARY

库的路径。仅当模块提供单个库时才使用此形式。将其用作 find_library() 命令中的结果变量是合适的。

Xxx_Yy_LIBRARY

模块 Xxx 提供的库 Yy 的路径。当模块提供多个库，或者其他模块也可能提供同名库时，请使用此形式。将其用作 find_library() 命令中的结果变量也是合适的。

Xxx_INCLUDE_DIR

当模块仅提供单个库时，此变量可用于指定在哪里查找使用该库的头文件（或更准确地说，库的消费者应添加到其头文件搜索路径中的路径）。将其用作 find_path() 命令中的结果变量是合适的。

Xxx_Yy_INCLUDE_DIR

如果模块提供多个库，或者其他模块也可能提供同名库，则建议使用此形式来指定在哪里查找使用该模块提供的库 Yy 的头文件。同样，将其用作 find_path() 命令中的结果变量是合适的。

为了防止用户被过多的配置设置所淹没，请尽量将尽可能多的选项排除在缓存之外，至少留出一个可用于禁用模块使用或定位未找到的库的选项（例如 Xxx_ROOT_DIR）。出于同样的原因，将大多数缓存选项标记为高级。对于提供调试和发布二进制文件的包，创建带有 _LIBRARY_<CONFIG> 后缀的缓存变量是很常见的，例如 Foo_LIBRARY_RELEASE 和 Foo_LIBRARY_DEBUG。SelectLibraryConfigurations 模块在此类情况下会有所帮助。

虽然这些是标准变量名称，但应为正在替换旧版本的查找模块中先前使用的任何名称提供向后兼容性。旧变量名称应记录为已弃用，以劝阻进一步使用。

查找模块示例¶

我们将描述如何为库 Foo 创建一个简单的查找模块。

模块的顶部应以许可证声明开始，后跟一个空行，然后是 括号注释。该注释应以 .rst: 开头，以表明其内容的其余部分是 reStructuredText 格式的文档。例如

# Distributed under the OSI-approved BSD 3-Clause License.  See accompanying
# file LICENSE.rst or https://cmake.com.cn/licensing for details.

#[=======================================================================[.rst:
FindFoo
-------

Finds the Foo library.

Imported Targets
^^^^^^^^^^^^^^^^

This module provides the following imported targets, if found:

``Foo::Foo``
  The Foo library

Result Variables
^^^^^^^^^^^^^^^^

This module defines the following variables:

``Foo_FOUND``
  Boolean indicating whether (the requested version of) Foo was found.
``Foo_VERSION``
  The version of the Foo library which was found.
``Foo_INCLUDE_DIRS``
  Include directories needed to use Foo.
``Foo_LIBRARIES``
  Libraries needed to link to Foo.

Cache Variables
^^^^^^^^^^^^^^^

The following cache variables may also be set:

``Foo_INCLUDE_DIR``
  The directory containing ``foo.h``.
``Foo_LIBRARY``
  The path to the Foo library.

#]=======================================================================]

模块文档包括

  • 指定模块名称的下划线标题。

  • 模块查找内容的简单描述。某些包可能需要更多描述。如果模块的用户应该了解注意事项或其他详细信息，请在此处指定。

  • 列出模块提供的导入目标（如果有）的部分。

  • 列出模块提供的结果变量的部分。

  • 可选地，列出模块使用的缓存变量（如果有）的部分。

如果包提供了任何宏或函数，则应将它们列在附加部分中，但可以通过在定义这些宏或函数之前紧接的额外 .rst: 注释块进行记录。

查找模块实现可以从文档块下方开始。现在必须找到实际的库等。这里的代码显然会因模块而异（毕竟，处理这些正是查找模块的目的），但对于库来说，往往有一种共同的模式。

首先，我们尝试使用 pkg-config 来查找库。请注意，我们不能依赖它，因为它可能不可用，但它提供了一个很好的起点。

find_package(PkgConfig)
if(PkgConfig_FOUND)
  pkg_check_modules(PC_Foo QUIET Foo)
endif()

这应该定义一些以 PC_Foo_ 开头的变量，其中包含来自 Foo.pc 文件的信息。

现在我们需要找到库和包含文件；我们使用来自 pkg-config 的信息，在检查其他默认路径之前，为 CMake 提供有关在哪里查找的提示。

find_path(Foo_INCLUDE_DIR
  NAMES foo.h
  HINTS ${PC_Foo_INCLUDE_DIRS}
  PATH_SUFFIXES Foo
)
find_library(Foo_LIBRARY
  NAMES foo
  HINTS ${PC_Foo_LIBRARY_DIRS}
)

或者，如果库以多种配置提供，可以使用 SelectLibraryConfigurations 模块自动设置 Foo_LIBRARY 变量

find_library(Foo_LIBRARY_RELEASE
  NAMES foo
  HINTS ${PC_Foo_LIBRARY_DIRS}/Release
)
find_library(Foo_LIBRARY_DEBUG
  NAMES foo
  HINTS ${PC_Foo_LIBRARY_DIRS}/Debug
)

include(SelectLibraryConfigurations)
select_library_configurations(Foo)

如果有获取版本的好方法（例如，从头文件），则可以使用该信息来设置 Foo_VERSION。否则，尝试使用来自 pkg-config 的信息

set(Foo_VERSION ${PC_Foo_VERSION})

现在我们可以使用 FindPackageHandleStandardArgs 模块为我们处理剩余的大部分步骤

include(FindPackageHandleStandardArgs)
find_package_handle_standard_args(Foo
  REQUIRED_VARS
    Foo_LIBRARY
    Foo_INCLUDE_DIR
  VERSION_VAR Foo_VERSION
)

这将检查 REQUIRED_VARS 是否包含值（不以 -NOTFOUND 结尾）并相应地设置 Foo_FOUND。它还将缓存这些值。如果设置了 Foo_VERSION，并且向 find_package() 传递了要求的版本，它将根据 Foo_VERSION 中的版本检查请求的版本。它还将适当地打印消息；请注意，如果找到了包，它将打印第一个必需变量的内容，以指示它在哪里被找到。

此时，我们必须为查找模块的用户提供一种链接到所找到的一个或多个库的方法。如上文“查找模块”部分所述，有两种方法。传统的变量方法如下

if(Foo_FOUND)
  set(Foo_LIBRARIES ${Foo_LIBRARY})
  set(Foo_INCLUDE_DIRS ${Foo_INCLUDE_DIR})
  set(Foo_DEFINITIONS ${PC_Foo_CFLAGS_OTHER})
endif()

如果找到多个库，则所有这些库都应包含在这些变量中（有关详细信息，请参阅“标准变量名称”部分）。

提供导入目标时，应进行命名空间处理（因此有 Foo:: 前缀）；CMake 将识别出传递给 target_link_libraries() 且名称中包含 :: 的值应为导入目标（而不是仅仅是库名称），如果该目标不存在，将产生适当的诊断消息（请参阅策略 CMP0028）。

if(Foo_FOUND AND NOT TARGET Foo::Foo)
  add_library(Foo::Foo UNKNOWN IMPORTED)
  set_target_properties(Foo::Foo PROPERTIES
    IMPORTED_LOCATION "${Foo_LIBRARY}"
    INTERFACE_COMPILE_OPTIONS "${PC_Foo_CFLAGS_OTHER}"
    INTERFACE_INCLUDE_DIRECTORIES "${Foo_INCLUDE_DIR}"
  )
endif()

关于此点需要注意的一点是，INTERFACE_INCLUDE_DIRECTORIES 和类似的属性应该只包含有关目标本身的信息，而不应包含其任何依赖项的信息。相反，这些依赖项也应该是目标，并且应告知 CMake 它们是此目标的依赖项。CMake 然后会自动组合所有必要的信息。

在 add_library() 命令中创建的 IMPORTED 目标的类型始终可以指定为 UNKNOWN 类型。这简化了可能找到静态或共享变体的情况下的代码，并且 CMake 将通过检查文件来确定类型。

如果库以多种配置提供，则还应填充 IMPORTED_CONFIGURATIONS 目标属性

if(Foo_FOUND)
  if (NOT TARGET Foo::Foo)
    add_library(Foo::Foo UNKNOWN IMPORTED)
  endif()
  if (Foo_LIBRARY_RELEASE)
    set_property(TARGET Foo::Foo APPEND PROPERTY
      IMPORTED_CONFIGURATIONS RELEASE
    )
    set_target_properties(Foo::Foo PROPERTIES
      IMPORTED_LOCATION_RELEASE "${Foo_LIBRARY_RELEASE}"
    )
  endif()
  if (Foo_LIBRARY_DEBUG)
    set_property(TARGET Foo::Foo APPEND PROPERTY
      IMPORTED_CONFIGURATIONS DEBUG
    )
    set_target_properties(Foo::Foo PROPERTIES
      IMPORTED_LOCATION_DEBUG "${Foo_LIBRARY_DEBUG}"
    )
  endif()
  set_target_properties(Foo::Foo PROPERTIES
    INTERFACE_COMPILE_OPTIONS "${PC_Foo_CFLAGS_OTHER}"
    INTERFACE_INCLUDE_DIRECTORIES "${Foo_INCLUDE_DIR}"
  )
endif()

RELEASE 变体应在属性中首先列出，以便如果用户使用的配置与任何列出的 IMPORTED_CONFIGURATIONS 不完全匹配，则会选择该变体。

大多数缓存变量应标记为高级，以在 cmake-gui(1) 或 ccmake(1) 等图形界面中保持隐藏，除非用户明确选择显示和修改它们

mark_as_advanced(
  Foo_INCLUDE_DIR
  Foo_LIBRARY
)

如果此查找模块替换了在当前标准变量命名约定之前提供变量的旧模块版本，则还应提供向后兼容性变量，以尽可能减少干扰。例如

# Backward compatibility variables
set(Foo_VERSION_STRING ${Foo_VERSION})
