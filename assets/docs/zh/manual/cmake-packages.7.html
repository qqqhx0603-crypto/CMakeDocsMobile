简介¶

包为基于 CMake 的构建系统提供依赖信息。包通过 find_package() 命令进行查找。使用 find_package() 的结果要么是一组 IMPORTED 目标，要么是一组对应于构建相关信息的变量。

使用包¶

CMake 直接支持两种形式的包：配置文件包 和 查找模块包。通过 FindPkgConfig 模块也提供了对 pkg-config 包的间接支持。在所有情况下，find_package() 的基本调用形式都是相同的。

find_package(Qt4 4.7.0 REQUIRED) # CMake provides a Qt4 find-module
find_package(Qt5Core 5.1.0 REQUIRED) # Qt provides a Qt5 package config file.
find_package(LibXml2 REQUIRED) # Use pkg-config via the LibXml2 find-module

代码说明

  • find_package(Qt4 4.7.0 REQUIRED) # CMake provides a Qt4 find-module：这一行继续补充当前命令或表达式的内容。

  • find_package(Qt5Core 5.1.0 REQUIRED) # Qt provides a Qt5 package config file.：这一行继续补充当前命令或表达式的内容。

  • find_package(LibXml2 REQUIRED) # Use pkg-config via the LibXml2 find-module：这一行继续补充当前命令或表达式的内容。

在已知上游提供了包配置文件，且仅应使用该文件的情况下，可以将 CONFIG 关键字传递给 find_package()。

find_package(Qt5Core 5.1.0 CONFIG REQUIRED)
find_package(Qt5Gui 5.1.0 CONFIG)

代码说明

  • find_package(Qt5Core 5.1.0 CONFIG REQUIRED)：调用 find_package() 命令，查找并加载外部依赖包。

  • find_package(Qt5Gui 5.1.0 CONFIG)：调用 find_package() 命令，查找并加载外部依赖包。

类似地，MODULE 关键字表示仅使用查找模块。

find_package(Qt4 4.7.0 MODULE REQUIRED)

代码说明

  • find_package(Qt4 4.7.0 MODULE REQUIRED)：调用 find_package() 命令，查找并加载外部依赖包。

明确指定包类型可以改善用户在找不到包时看到的错误信息。

两种类型的包都支持指定包的组件，既可以在 REQUIRED 关键字之后

find_package(Qt5 5.1.0 CONFIG REQUIRED Widgets Xml Sql)

代码说明

  • find_package(Qt5 5.1.0 CONFIG REQUIRED Widgets Xml Sql)：调用 find_package() 命令，查找并加载外部依赖包。

或者作为单独的 COMPONENTS 列表

find_package(Qt5 5.1.0 COMPONENTS Widgets Xml Sql)

代码说明

  • find_package(Qt5 5.1.0 COMPONENTS Widgets Xml Sql)：调用 find_package() 命令，查找并加载外部依赖包。

或者作为单独的 OPTIONAL_COMPONENTS 列表

find_package(Qt5 5.1.0 COMPONENTS Widgets
                       OPTIONAL_COMPONENTS Xml Sql
)

代码说明

  • OPTIONAL_COMPONENTS Xml Sql：这一行继续补充当前命令或表达式的内容。

  • )：这一行继续补充当前命令或表达式的内容。

COMPONENTS 和 OPTIONAL_COMPONENTS 的处理由包本身定义。

通过将 CMAKE_DISABLE_FIND_PACKAGE_<PackageName> 变量设置为 TRUE，将不会搜索 <PackageName> 包，且结果始终为 NOTFOUND。同样，将 CMAKE_REQUIRE_FIND_PACKAGE_<PackageName> 设置为 TRUE 会使该包成为 REQUIRED（必需）。

配置文件包¶

配置文件包是一组由上游提供给下游使用的文件。CMake 会在多个位置搜索包配置文件，具体描述见 find_package() 文档。CMake 用户告诉 cmake(1) 在非标准路径中搜索包的最简单方法是设置 CMAKE_PREFIX_PATH 缓存变量。

配置文件包由上游供应商作为开发包的一部分提供，即它们属于头文件以及为协助下游使用该包而提供的任何其他文件。

使用配置文件包时，一组提供包状态信息的变量也会被自动设置。<PackageName>_FOUND 变量将根据是否找到包而设置为 true 或 false。<PackageName>_DIR 缓存变量会被设置为包配置文件的位置。

查找模块包¶

查找模块是一个包含一组规则的文件，用于查找依赖项的必要部分，主要是头文件和库。通常，当上游不是使用 CMake 构建的，或者不具备提供包配置文件的 CMake 感知能力时，就需要查找模块。与包配置文件不同，它不随上游发布，而是由下游通过猜测具有平台特定提示的文件位置来查找文件。

与上游提供的包配置文件不同，没有单一的参考点来标识包已找到，因此 <PackageName>_FOUND 变量不会由 find_package() 命令自动设置。不过，按照惯例，它通常会被设置，并且应该由查找模块的作者进行设置。同样，没有 <PackageName>_DIR 变量，而是每个制品（如库位置和头文件位置）都提供一个单独的缓存变量。

有关创建查找模块文件的更多信息，请参阅 cmake-developer(7) 手册。

包布局¶

配置文件包包含一个 包配置文件，以及可选的随项目分发的 包版本文件。

包配置文件¶

假设有一个项目 Foo 安装了以下文件

<prefix>/include/foo-1.2/foo.h
<prefix>/lib/foo-1.2/libfoo.a

代码说明

  • <prefix>/include/foo-1.2/foo.h：这一行给出示例文本或输出，用来说明：<prefix>/include/foo-1.2/foo.h

  • <prefix>/lib/foo-1.2/libfoo.a：这一行给出示例文本或输出，用来说明：<prefix>/lib/foo-1.2/libfoo.a

它还可以提供一个 CMake 包配置文件

<prefix>/lib/cmake/foo-1.2/FooConfig.cmake

代码说明

  • <prefix>/lib/cmake/foo-1.2/FooConfig.cmake：这一行给出示例文本或输出，用来说明：<prefix>/lib/cmake/foo-1.2/FooConfig.cmake

其内容定义了 IMPORTED 目标或定义变量，例如

# ...
# (compute PREFIX relative to file location)
# ...
set(Foo_INCLUDE_DIRS ${PREFIX}/include/foo-1.2)
set(Foo_LIBRARIES ${PREFIX}/lib/foo-1.2/libfoo.a)

代码说明

  • # ...：这是 CMake 注释。此处省略了与当前要点无关的代码。

  • # (compute PREFIX relative to file location)：这是 CMake 注释。这是代码作者保留的说明，描述当前步骤的目的、输入输出或使用限制；英文原文保留在代码中。

  • # ...：这是 CMake 注释。此处省略了与当前要点无关的代码。

  • set(Foo_INCLUDE_DIRS ${PREFIX}/include/foo-1.2)：调用 set() 命令，把变量 Foo_INCLUDE_DIRS 设为给定的值；变量说明：PREFIX：前缀相关变量，其精确取值和作用域以当前页面或脚本上下文为准；Foo_INCLUDE_DIRS：包含目录相关变量，其精确取值和作用域以当前页面或脚本上下文为准。

  • set(Foo_LIBRARIES ${PREFIX}/lib/foo-1.2/libfoo.a)：调用 set() 命令，把变量 Foo_LIBRARIES 设为给定的值；变量说明：PREFIX：前缀相关变量，其精确取值和作用域以当前页面或脚本上下文为准；Foo_LIBRARIES：当前示例使用的变量，其值由前面的命令、调用参数或运行环境提供。

如果另一个项目希望使用 Foo，它只需要定位 FooConfig.cmake 文件并加载它，即可获取有关包内容位置所需的所有信息。由于包配置文件由包安装过程提供，它已经了解所有文件的位置。

find_package() 命令可用于搜索包配置文件。此命令构建一组安装前缀，并在每个前缀下的多个位置进行搜索。给定名称 Foo，它会查找名为 FooConfig.cmake 或 foo-config.cmake 的文件。位置的全集在 find_package() 命令文档中指定。它查找的位置之一是

<prefix>/lib/cmake/Foo*/

代码说明

  • <prefix>/lib/cmake/Foo*/：这一行给出示例文本或输出，用来说明：<prefix>/lib/cmake/Foo*/

其中 Foo* 是一个不区分大小写的匹配表达式。在我们的示例中，匹配表达式将匹配 <prefix>/lib/cmake/foo-1.2，包配置文件将被找到。

一旦找到，包配置文件会立即加载。它与包版本文件一起，包含了项目使用该包所需的所有信息。

包版本文件¶

当 find_package() 命令找到候选包配置文件时，它会在其旁边查找版本文件。加载版本文件是为了测试包版本是否与请求的版本兼容。如果版本文件声明兼容，则接受配置文件。否则将其忽略。

包版本文件的名称必须与包配置文件名称匹配，但在 .cmake 扩展名之前附加了 -version 或 Version。例如，以下文件

<prefix>/lib/cmake/foo-1.3/foo-config.cmake
<prefix>/lib/cmake/foo-1.3/foo-config-version.cmake

代码说明

  • <prefix>/lib/cmake/foo-1.3/foo-config.cmake：这一行给出示例文本或输出，用来说明：<prefix>/lib/cmake/foo-1.3/foo-config.cmake

  • <prefix>/lib/cmake/foo-1.3/foo-config-version.cmake：这一行给出示例文本或输出，用来说明：<prefix>/lib/cmake/foo-1.3/foo-config-version.cmake

和

<prefix>/lib/cmake/bar-4.2/BarConfig.cmake
<prefix>/lib/cmake/bar-4.2/BarConfigVersion.cmake

代码说明

  • <prefix>/lib/cmake/bar-4.2/BarConfig.cmake：这一行给出示例文本或输出，用来说明：<prefix>/lib/cmake/bar-4.2/BarConfig.cmake

  • <prefix>/lib/cmake/bar-4.2/BarConfigVersion.cmake：这一行给出示例文本或输出，用来说明：<prefix>/lib/cmake/bar-4.2/BarConfigVersion.cmake

它们分别是包配置文件和相应的包版本文件对。

当 find_package() 命令加载版本文件时，它首先设置以下变量

PACKAGE_FIND_NAME

<PackageName>

PACKAGE_FIND_VERSION

完整请求版本字符串

PACKAGE_FIND_VERSION_MAJOR

如果已请求，则为主版本号，否则为 0

PACKAGE_FIND_VERSION_MINOR

如果已请求，则为次版本号，否则为 0

PACKAGE_FIND_VERSION_PATCH

如果已请求，则为补丁版本号，否则为 0

PACKAGE_FIND_VERSION_TWEAK

如果已请求，则为微调版本号，否则为 0

PACKAGE_FIND_VERSION_COUNT

版本组件的数量，0 到 4

版本文件必须使用这些变量来检查它是否与请求的版本兼容或完全匹配，并设置以下变量作为结果

PACKAGE_VERSION

完整提供的版本字符串

PACKAGE_VERSION_EXACT

如果版本完全匹配，则为 True

PACKAGE_VERSION_COMPATIBLE

如果版本兼容，则为 True

PACKAGE_VERSION_UNSUITABLE

如果任何版本均不适用，则为 True

版本文件在嵌套作用域中加载，因此它们可以自由设置任何变量作为其计算的一部分。当版本文件完成并检查完输出变量后，find_package 命令会清除该作用域。当版本文件声明与请求的版本兼容时，find_package 命令会为项目设置以下变量

<PackageName>_VERSION

完整提供的版本字符串

<PackageName>_VERSION_MAJOR

如果已提供，则为主版本号，否则为 0

<PackageName>_VERSION_MINOR

如果已提供，则为次版本号，否则为 0

<PackageName>_VERSION_PATCH

如果已提供，则为补丁版本号，否则为 0

<PackageName>_VERSION_TWEAK

如果已提供，则为微调版本号，否则为 0

<PackageName>_VERSION_COUNT

版本组件的数量，0 到 4

这些变量报告实际找到的包版本。其名称中的 <PackageName> 部分与传递给 find_package() 命令的参数匹配。

创建包¶

通常，上游依赖 CMake 本身，并可以使用某些 CMake 工具来创建包文件。考虑一个提供单个共享库的上游项目

project(UpstreamLib)

set(CMAKE_INCLUDE_CURRENT_DIR ON)
set(CMAKE_INCLUDE_CURRENT_DIR_IN_INTERFACE ON)

set(Upstream_VERSION 3.4.1)

include(GenerateExportHeader)

add_library(ClimbingStats SHARED climbingstats.cpp)
generate_export_header(ClimbingStats)
set_property(TARGET ClimbingStats PROPERTY VERSION ${Upstream_VERSION})
set_property(TARGET ClimbingStats PROPERTY SOVERSION 3)
set_property(TARGET ClimbingStats PROPERTY
  INTERFACE_ClimbingStats_MAJOR_VERSION 3)
set_property(TARGET ClimbingStats APPEND PROPERTY
  COMPATIBLE_INTERFACE_STRING ClimbingStats_MAJOR_VERSION
)

install(TARGETS ClimbingStats EXPORT ClimbingStatsTargets
  LIBRARY DESTINATION lib
  ARCHIVE DESTINATION lib
  RUNTIME DESTINATION bin
  INCLUDES DESTINATION include
)
install(
  FILES
    climbingstats.h
    "${CMAKE_CURRENT_BINARY_DIR}/climbingstats_export.h"
  DESTINATION
    include
  COMPONENT
    Devel
)

include(CMakePackageConfigHelpers)
write_basic_package_version_file(
  "${CMAKE_CURRENT_BINARY_DIR}/ClimbingStats/ClimbingStatsConfigVersion.cmake"
  VERSION ${Upstream_VERSION}
  COMPATIBILITY AnyNewerVersion
)

export(EXPORT ClimbingStatsTargets
  FILE "${CMAKE_CURRENT_BINARY_DIR}/ClimbingStats/ClimbingStatsTargets.cmake"
  NAMESPACE Upstream::
)
configure_file(cmake/ClimbingStatsConfig.cmake
  "${CMAKE_CURRENT_BINARY_DIR}/ClimbingStats/ClimbingStatsConfig.cmake"
  COPYONLY
)

set(ConfigPackageLocation lib/cmake/ClimbingStats)
install(EXPORT ClimbingStatsTargets
  FILE
    ClimbingStatsTargets.cmake
  NAMESPACE
    Upstream::
  DESTINATION
    ${ConfigPackageLocation}
)
install(
  FILES
    cmake/ClimbingStatsConfig.cmake
    "${CMAKE_CURRENT_BINARY_DIR}/ClimbingStats/ClimbingStatsConfigVersion.cmake"
  DESTINATION
    ${ConfigPackageLocation}
  COMPONENT
    Devel
)

CMakePackageConfigHelpers 模块提供了一个用于创建简单 ConfigVersion.cmake 文件的宏。此文件设置包的版本。当调用 find_package() 时，CMake 会读取它以确定与请求版本的兼容性，并设置一些特定于版本的变量 <PackageName>_VERSION、<PackageName>_VERSION_MAJOR、<PackageName>_VERSION_MINOR 等。install(EXPORT) 命令用于导出 ClimbingStatsTargets 导出集中的目标，该集先前由 install(TARGETS) 命令定义。此命令生成包含 IMPORTED 目标的 ClimbingStatsTargets.cmake 文件，适用于下游使用，并安排将其安装到 lib/cmake/ClimbingStats。生成的 ClimbingStatsConfigVersion.cmake 和 cmake/ClimbingStatsConfig.cmake 被安装到同一位置，从而完成包的创建。

生成的 IMPORTED 目标设置了适当的属性来定义其 使用要求，例如 INTERFACE_INCLUDE_DIRECTORIES、INTERFACE_COMPILE_DEFINITIONS 以及其他相关的内置 INTERFACE_ 属性。列在 COMPATIBLE_INTERFACE_STRING 中的用户定义属性的 INTERFACE 变体以及其他 兼容接口属性 也会传播到生成的 IMPORTED 目标。在上述情况下，ClimbingStats_MAJOR_VERSION 被定义为一个字符串，该字符串必须在任何使用者的依赖项之间兼容。通过在当前版本和下一个版本的 ClimbingStats 中设置此自定义用户属性，如果尝试将版本 3 与版本 4 一起使用，cmake(1) 将发出诊断信息。如果包的不同主版本设计为不兼容，包可以选择采用这种模式。

在导出目标以进行安装时，会指定一个带有双冒号的 NAMESPACE。这种双冒号的惯例给 CMake 一个提示，即当下游使用 target_link_libraries() 命令时，该名称是一个 IMPORTED 目标。这样，如果提供该包的项目尚未找到，CMake 就可以发出诊断信息。

在这种情况下，当使用 install(TARGETS) 时，指定了 INCLUDES DESTINATION。这导致 IMPORTED 目标的 INTERFACE_INCLUDE_DIRECTORIES 使用 CMAKE_INSTALL_PREFIX 中的 include 目录进行填充。当 IMPORTED 目标被下游使用时，它会自动消耗来自该属性的条目。

为构建树创建包配置文件¶

export(EXPORT) 命令创建一个特定于构建树且不可重定位的 IMPORTED 目标定义文件。这同样可以与合适的包配置文件和包版本文件结合使用，为构建树定义一个包，该包可以在不安装的情况下使用。构建树的使用者只需确保 CMAKE_PREFIX_PATH 包含构建目录，或者在缓存中将 ClimbingStats_DIR 设置为 <build_dir>/ClimbingStats。

创建可重定位的包¶

可重定位的包不得引用构建包的机器上绝对路径的文件，因为这些路径在安装包的机器上可能不存在。

由 install(EXPORT) 创建的包旨在实现可重定位，使用相对于包本身位置的路径。为 EXPORT 定义目标的接口时，请记住包含目录应指定为相对于 CMAKE_INSTALL_PREFIX 的相对路径。

target_include_directories(tgt INTERFACE
  # Wrong, not relocatable:
  $<INSTALL_INTERFACE:${CMAKE_INSTALL_PREFIX}/include/TgtName>
)

target_include_directories(tgt INTERFACE
  # Ok, relocatable:
  $<INSTALL_INTERFACE:include/TgtName>
)

包注册表¶

CMake 提供了两个中心位置来注册已在系统任何位置构建或安装的包

  • 用户包注册表

  • 系统包注册表

注册表对于帮助项目在非标准安装位置或直接在其自己的构建树中查找包特别有用。项目可以（使用其自身的方法，见下文）填充用户或系统注册表以引用其位置。在任一情况下，包都应在注册位置存储一个 包配置文件 (<PackageName>Config.cmake) 和可选的 包版本文件 (<PackageName>ConfigVersion.cmake)。

find_package() 命令会将搜索两个包注册表作为其文档中指定的搜索步骤的一部分。如果它拥有足够的权限，它还会删除指向不存在的目录或不包含匹配包配置文件的目录的陈旧包注册表条目。

用户包注册表¶

用户包注册表存储在每个用户的特定位置。export(PACKAGE) 命令可用于在用户包注册表中注册项目构建树。CMake 目前不提供将安装树添加到用户包注册表的界面。如果需要，安装程序必须手动教学以注册其包。

在 Windows（微软桌面系统） 上，用户包注册表存储在 HKEY_CURRENT_USER 下的一个注册表项中。

<PackageName> 可能出现在注册表项下

HKEY_CURRENT_USER\Software\Kitware\CMake\Packages\<PackageName>

系统包注册表存储在系统范围的位置。CMake 目前不提供添加到系统包注册表的界面。如果需要，安装程序必须手动教学以注册其包。

在 Windows（微软桌面系统） 上，系统包注册表存储在 HKEY_LOCAL_MACHINE 下的一个注册表项中。<PackageName> 可能出现在注册表项下

禁用包注册表¶

在某些情况下，使用包注册表并不理想。CMake 允许使用以下变量禁用它们

  • 当 CMP0090 设置为 NEW 时，export(PACKAGE) 命令不会填充用户包注册表，除非 CMAKE_EXPORT_PACKAGE_REGISTRY 变量显式启用它。当 CMP0090 没有设置为 NEW 时，export(PACKAGE) 会填充用户包注册表，除非 CMAKE_EXPORT_NO_PACKAGE_REGISTRY 变量显式禁用它。

  • CMAKE_FIND_USE_PACKAGE_REGISTRY 当设置为 FALSE 时，会在所有 find_package() 调用中禁用用户包注册表。

  • 已弃用的 CMAKE_FIND_PACKAGE_NO_PACKAGE_REGISTRY 当设置为 TRUE 时，会在所有 find_package() 调用中禁用用户包注册表。当已设置 CMAKE_FIND_USE_PACKAGE_REGISTRY 时，此变量将被忽略。

  • CMAKE_FIND_PACKAGE_NO_SYSTEM_PACKAGE_REGISTRY 在所有 find_package() 调用中禁用系统包注册表。

包注册表示例¶

命名包注册表条目的一种简单惯例是使用内容哈希。它们是确定性的且不太可能冲突（export(PACKAGE) 使用此方法）。引用特定目录的条目名称只是目录路径本身的内容哈希。

如果项目安排包注册表条目存在，例如

> reg query HKCU\Software\Kitware\CMake\Packages\MyPackage
HKEY_CURRENT_USER\Software\Kitware\CMake\Packages\MyPackage
 45e7d55f13b87179bb12f907c8de6fc4 REG_SZ c:/Users/Me/Work/lib/cmake/MyPackage
 7b4a9844f681c80ce93190d4e3185db9 REG_SZ c:/Users/Me/Work/MyPackage-build

代码说明

  • > reg query HKCU\Software\Kitware\CMake\Packages\MyPackage：这一行给出示例文本或输出，用来说明：> reg query HKCU\Software\Kitware\CMake\Packages\MyPackage；运算符：> 表示大于比较。

  • HKEY_CURRENT_USER\Software\Kitware\CMake\Packages\MyPackage：这一行给出示例文本或输出，用来说明：HKEY_CURRENT_USER\Software\Kitware\CMake\Packages\MyPackage

  • 45e7d55f13b87179bb12f907c8de6fc4 REG_SZ c:/Users/Me/Work/lib/cmake/MyPackage：这一行给出示例文本或输出，用来说明：45e7d55f13b87179bb12f907c8de6fc4 REG_SZ c:/Users/Me/Work/lib/cmake/MyPackage

  • 7b4a9844f681c80ce93190d4e3185db9 REG_SZ c:/Users/Me/Work/MyPackage-build：这一行给出示例文本或输出，用来说明：7b4a9844f681c80ce93190d4e3185db9 REG_SZ c:/Users/Me/Work/MyPackage-build

或

$ cat ~/.cmake/packages/MyPackage/7d1fb77e07ce59a81bed093bbee945bd
/home/me/work/lib/cmake/MyPackage
$ cat ~/.cmake/packages/MyPackage/f92c1db873a1937f3100706657c63e07
/home/me/work/MyPackage-build

代码说明

  • $ cat ~/.cmake/packages/MyPackage/7d1fb77e07ce59a81bed093bbee945bd：这是一条命令行输入，用来演示终端中实际执行的命令。

  • /home/me/work/lib/cmake/MyPackage：这是命令执行后的输出结果，内容为：/home/me/work/lib/cmake/MyPackage

  • $ cat ~/.cmake/packages/MyPackage/f92c1db873a1937f3100706657c63e07：这是一条命令行输入，用来演示终端中实际执行的命令。

  • /home/me/work/MyPackage-build：这是命令执行后的输出结果，内容为：/home/me/work/MyPackage-build

那么 CMakeLists.txt 代码

find_package(MyPackage)

代码说明

  • find_package(MyPackage)：调用 find_package() 命令，查找并加载外部依赖包。

将搜索已注册的位置以查找包配置文件 (MyPackageConfig.cmake)。单个包的包注册表条目之间的搜索顺序未指定，条目名称（本例中的哈希）没有意义。注册的位置可能包含包版本文件 (MyPackageConfigVersion.cmake)，以告知 find_package() 特定位置是否适合请求的版本。

包注册表所有权¶

包注册表条目由它们引用的项目安装分别拥有。包安装程序负责添加其自己的条目，相应的卸载程序负责删除它。

export(PACKAGE) 命令使用项目构建树的位置填充用户包注册表。构建树往往会被开发人员删除，并且没有可以触发其条目删除的“卸载”事件。为了保持注册表的整洁，find_package() 命令在拥有足够权限的情况下会自动删除它遇到的陈旧条目。一旦调用了 export(PACKAGE)，CMake 就不提供删除引用现有构建树的条目的界面。但是，如果项目从构建树中删除了其包配置文件，则引用该位置的条目将被视为陈旧。
