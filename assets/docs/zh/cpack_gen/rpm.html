CPack RPM 生成器¶

内置的（二进制）CPack RPM 生成器（仅限 Unix）

CPack RPM 生成器专用变量¶

CPack RPM 生成器可用于使用 CPack 创建 RPM 软件包。CPack RPM 生成器是一个 CPack 生成器，因此它使用了 CPack 所使用的 CPACK_XXX 变量。

CPack RPM 生成器具有由特定 CPACK_RPM_XXX 变量控制的特定功能。

CPACK_RPM_<COMPONENT>_XXXX 变量可用于获取组件特定的值。但请注意，<COMPONENT> 指的是以大写形式书写的分组名称。它可以是组件名称或组件组名称。通常，这些变量对应于 RPM spec 文件实体。有关 spec 文件的信息，请访问 https://rpm.org/documentation。

版本 3.6 中变更：变量中的 <COMPONENT> 部分首选使用大写形式（例如，如果组件名为 foo，则使用 CPACK_RPM_FOO_XXXX 变量名格式），这与其他 CPACK_<COMPONENT>_XXXX 变量保持一致。为了向后兼容（CMake/CPack 3.5 及更低版本），对于旧版本 CMake/CPack 中定义的变量，仍然支持区分大小写的组件名称（例如，fOo 将用作 CPACK_RPM_fOo_XXXX），但这在将来添加的变量中无法保证。出于向后兼容性考虑，在同时存在大小写形式的变量时，区分大小写的组件变量会覆盖大写版本。

以下是一些 CPack RPM 生成器的维基资源，它们保留仅出于历史原因，现已不再维护，但可能仍有参考价值：

  • https://gitlab.kitware.com/cmake/community/-/wikis/doc/cpack/Configuration

  • https://gitlab.kitware.com/cmake/community/-/wikis/doc/cpack/PackageGenerators#rpm-unix-only

CPack RPM 生成器特定变量列表

CPACK_RPM_COMPONENT_INSTALL¶

为 CPack RPM 生成器启用组件打包

强制项:

否

默认值：

关闭

如果启用（ON），则生成多个包。默认情况下，生成一个包含所有组件文件的单个包。

CPACK_RPM_<component>_PACKAGE_SUMMARY¶

RPM 软件包摘要。

强制项:

是

默认值：

CPACK_PACKAGE_DESCRIPTION_SUMMARY

3.2 版本新增：按组件划分的 CPACK_RPM_<component>_PACKAGE_SUMMARY 变量。

CPACK_RPM_PACKAGE_NAME¶
CPACK_RPM_<component>_PACKAGE_NAME¶

RPM 软件包名称。

强制项:

是

默认值：

CPACK_PACKAGE_NAME

3.5 版本新增：按组件划分的 CPACK_RPM_<component>_PACKAGE_NAME 变量。

CPACK_RPM_FILE_NAME¶
CPACK_RPM_<component>_FILE_NAME¶

3.6 版本新增。

软件包文件名。

强制项:

是

默认值：

<CPACK_PACKAGE_FILE_NAME>[-<component>].rpm，其中空格替换为 '-'

此变量可设置为：

RPM-DEFAULT

告诉 rpmbuild 自动生成软件包文件名。

<file-name>[.rpm]

使用指定的文件名。

版本 3.29 中变更：如果缺少 .rpm 后缀，将自动添加。此前必须显式添加后缀。

注意

通过使用用户提供的 spec 文件，或者使用 rpm 宏扩展（例如用于生成 debuginfo 软件包），或者简单地通过使用多个组件，可以生成多个 rpm 文件（可能源自单个 spec 文件或多个 spec 文件，每个组件执行都会生成自己的 spec 文件）。在这种情况下，由于此变量设置或 spec 文件内容结构，可能会出现重复的文件名。重复的文件会被覆盖，打包人员有责任以防止此类错误的方式设置变量。

CPACK_RPM_PACKAGE_RELEASE¶

RPM 软件包发布版本。

强制项:

是

默认值：

1

这是 RPM 软件包本身的编号，即打包的版本，而不是内容的版本（参见 CPACK_RPM_PACKAGE_VERSION）。如果之前的打包有缺陷，或者您想在此处放入特定的 Linux 发行版编号，可以更改默认值。

注意

这是放入 RPM Release: 字段的字符串。某些发行版（例如 Fedora、CentOS）需要 1%{?dist} 格式，而不仅仅是一个数字。%{?dist} 部分可以通过设置 CPACK_RPM_PACKAGE_RELEASE_DIST 来添加。

CPACK_RPM_PACKAGE_RELEASE_DIST¶

3.6 版本新增。

添加到 RPM Release: 字段的 dist 标签。

强制项:

否

默认值：

关闭

这是当前发行版报告的 %{dist} 标签，如果未设置 RPM 宏，则为空。如果设置了此变量，RPM Release: 字段值将设置为 ${CPACK_RPM_PACKAGE_RELEASE}%{?dist}。

CPACK_RPM_COMPRESSION_TYPE¶

RPM 压缩类型。

强制项:

否

默认值：

(系统默认)

可用于覆盖构建 RPM 时使用的 RPM 压缩类型。例如，某些 Linux 发行版默认使用 xz 或 zstd。使用此变量，可以指定要使用的特定压缩类型。

可能的值为

lzma

Lempel–Ziv–Markov 链算法

xz

XZ Utils LZMA2 (Lempel–Ziv–Markov 链算法，第 2 版) 压缩

bzip2

bzip2 Burrows–Wheeler 算法

gzip

GNU Gzip Deflate 压缩

zstd

在版本 3.31 中添加。

Zstandard 压缩

CPACK_RPM_PACKAGE_REQUIRES¶
CPACK_RPM_<component>_PACKAGE_REQUIRES¶

RPM spec 的 requires 字段。

强制项:

否

默认值：

可用于设置 RPM 依赖项（requires）。请注意，在设置此变量时，必须将整个值括在引号中，例如：

CPACK_RPM_PACKAGE_RELOCATABLE¶

构建一个可重定位的 RPM。

强制项:

否

默认值：

CPACK_PACKAGE_RELOCATABLE

如果此变量设置为 TRUE 或 ON，CPack RPM 生成器将尝试构建一个可重定位的 RPM 软件包。可重定位的 RPM 可以使用以下命令安装：

rpm --prefix or --relocate

为了将其安装在其他位置，请参阅 rpm(8)。请注意，如果 CPACK_SET_DESTDIR 设置为 ON，当前这可能会失败。如果设置了 CPACK_SET_DESTDIR，您会收到一条警告消息，但如果有文件以绝对路径安装，您会得到意想不到的行为。

CPACK_RPM_SPEC_INSTALL_POST¶

自 2.8.12 版本起弃用：请改用 CPACK_RPM_SPEC_MORE_DEFINE。

强制项:

否

默认值：

可用于覆盖生成的 spec 文件中的 __spec_install_post 部分。这会影响软件包创建过程中的安装步骤，而不是软件包安装期间。如需添加要在软件包安装期间执行的操作，请改用 CPACK_RPM_POST_INSTALL_SCRIPT_FILE。

CPACK_RPM_SPEC_MORE_DEFINE¶

RPM 扩展 spec 定义行。

强制项:

否

默认值：

可用于向生成的 spec 文件添加任何 %define 行。其一个用例是防止剥离可执行文件（但请注意，这也可能会禁用其他默认的安装后处理）。

set(CPACK_RPM_SPEC_MORE_DEFINE "%define __spec_install_post /bin/true")

切换 CPack RPM 生成器调试输出。

CPACK_RPM_EXCLUDE_FROM_AUTO_FILELIST¶

要排除的路径列表。

强制项:

否

默认值：
默认情况下排除以下路径：

  • /etc

  • /etc/init.d

  • /usr

  • /usr/bin

  • /usr/include

  • /usr/lib

  • /usr/libx32

  • /usr/lib64

  • /usr/share

  • /usr/share/aclocal

CPACK_RPM_RELOCATION_PATHS¶

版本 3.2 中新增。

软件包重定位路径列表。

强制项:

否

默认值：

可用于为每个可重定位的 RPM 指定一个以上的重定位路径。变量包含重定位路径列表，如果路径是相对的，则以 CPACK_RPM_<COMPONENT>_PACKAGE_PREFIX 的值或（如果未提供组件版本）CPACK_PACKAGING_INSTALL_PREFIX 的值为前缀。此变量不是基于组件的，因为其内容可用于同时为例如二进制目录和文档目录设置不同的路径前缀。只有特定组件需要的路径前缀才会被添加到该组件中——组件必须包含至少一个带有 CPACK_RPM_RELOCATION_PATHS 前缀的文件/目录/符号链接，才能添加该重定位路径。如果没有文件/目录/符号链接位于任何提供的路径前缀位置，则软件包将不包含任何重定位路径。不包含任何重定位路径，或包含位于重定位路径之外的文件/目录/符号链接的软件包，会打印出 AUTHOR_WARNING，指出 RPM 将是部分可重定位的。

符号链接的打包¶

3.3 版本中新增。

CPack RPM 生成器支持符号链接的打包。

execute_process(COMMAND ${CMAKE_COMMAND}
  -E create_symlink <relative_path_location> <symlink_name>)
install(FILES ${CMAKE_CURRENT_BINARY_DIR}/<symlink_name>
  DESTINATION <symlink_location> COMPONENT libraries)

符号链接在被添加到软件包之前将得到优化（如果可能，路径将被缩短），或者如果检测到多个重定位路径，则将生成安装后符号链接重定位脚本。

符号链接可能指向未由同一软件包打包的位置（无论是不同的组件，甚至根本没有打包），但在确定符号链接是应该创建还是存在于安装后脚本中时（取决于重定位路径），这些位置将被视为软件包的一部分。

版本 3.6 中变更：指向打包路径之外的符号链接会产生警告，并被视为不可重定位的永久符号链接。旧版本的 CMake 在此情况下会产生错误。

目前存在一些限制：

  • 对于基于组件的打包，在处理符号链接时不会检查组件间的依赖关系。指向不同组件内容的符号链接被视为与指向未打包位置的处理方式相同。

  • 指向通过一个或多个中间符号链接的位置的符号链接不会得到不同的处理——如果中间符号链接也位于可重定位路径上，则在软件包安装期间重新定位它可能会导致初始符号链接指向无效位置。

调试信息的打包¶

3.7 版本中新增。

Debuginfo 软件包包含用于调试已打包二进制文件的调试符号和源文件。

Debuginfo RPM 打包有一套自己的变量。

CPACK_RPM_DEBUGINFO_PACKAGE¶
CPACK_RPM_<component>_DEBUGINFO_PACKAGE¶

启用 debuginfo RPM 软件包的生成。

强制项:

否

默认值：

关闭

注意

二进制文件在打包前必须包含调试符号，因此对于 CMAKE_BUILD_TYPE 变量值，请使用 Debug 或 RelWithDebInfo。

此外，如果设置了 CPACK_STRIP_FILES，文件在到达 RPM 生成器之前会被剥离，因此将不包含调试符号，并且不会构建 debuginfo 软件包。请勿与 CPACK_STRIP_FILES 一起使用。

注意

从不含二进制文件的软件包、含二进制文件但无执行权限的软件包或不含调试符号的软件包生成的软件包将导致打包终止。

源码包（SRPM）的打包¶

3.7 版本中新增。

通过设置 CPACK_RPM_PACKAGE_SOURCES 变量，同时通常使用 CPACK_INSTALLED_DIRECTORIES 变量来提供包含 CMakeLists.txt 和源文件的目录，从而启用 SRPM 打包。

对于 CMake 项目，SRPM 软件包将通过执行以下命令生成：

cpack -G RPM --config ./CPackSourceConfig.cmake

注意

生成的 SRPM 软件包预期使用 cmake(1) 可执行文件构建，并使用 cpack(1) 可执行文件打包，因此 CMakeLists.txt 必须位于根源目录中，并且必须能够通过执行 cpack -G 命令生成二进制 rpm 软件包。在从生成的 SRPM 软件包生成二进制 rpm 软件包时，这两个可执行文件以及 rpmbuild 也必须存在。

一旦生成 SRPM 软件包，就可以通过创建 rpm 生成目录结构并执行 rpmbuild 工具来生成二进制软件包：

mkdir -p build_dir/{BUILD,BUILDROOT,RPMS,SOURCES,SPECS,SRPMS}
rpmbuild --define "_topdir <path_to_build_dir>" --rebuild <SRPM_file_name>

生成的软件包将位于 build_dir/RPMS 目录或其子目录中。

注意

SRPM 软件包在内部使用 CPack/RPM 生成器来生成二进制软件包，因此 CMakeScripts.txt 可以在 SRPM 到二进制 rpm 的生成步骤中决定软件包应包含的内容以及应如何打包（单体或组件）。CMake 可以通过例如在开始生成二进制 rpm 软件包的过程之前读取软件包管理器设置的环境变量来决定这一点。通过这种方式，单个 SRPM 软件包可用于根据平台的打包规则在不同平台上生成不同的二进制 rpm 软件包。
