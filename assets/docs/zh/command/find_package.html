注意

使用依赖指南 为该通用主题提供了高阶介绍。它更全面地概述了 find_package() 命令在宏观架构中的位置，包括它与 FetchContent 模块的关系。建议在深入阅读下文细节之前先阅读该指南。

查找包（通常由项目外部的某个实体提供），并加载该包的特定信息。对该命令的调用也可以被依赖提供程序拦截。

典型用法¶

大多数对 find_package() 的调用通常采用以下形式

find_package(<PackageName> [<version>] [REQUIRED] [COMPONENTS <components>...])

<PackageName> 是唯一强制性的参数。<version> 通常可以省略；如果项目没有该包就无法成功配置，则应给出 REQUIRED。一些更复杂的包支持组件，可以使用 COMPONENTS 关键字进行选择，但大多数包没有那么复杂。

以上是基本签名的简化形式。在可能的情况下，项目应优先使用此形式查找包。这可以降低复杂性，并最大限度地增加找到或提供包的方式。

理解基本签名足以应对 find_package() 的常规使用。打算提供包配置文件（package configuration file）的项目维护者应理解宏观全貌，详情请参阅完整签名以及本文档后续的所有章节。

搜索模式¶

该命令有几种搜索包的模式

模块模式 (Module mode)

在此模式下，CMake 搜索名为 Find<PackageName>.cmake 的文件，首先在 CMAKE_MODULE_PATH 列出的位置中查找，然后在 CMake 安装提供的 查找模块 (Find Modules) 中查找。如果找到了该文件，CMake 将读取并处理它。该模块负责查找包、检查版本并生成任何必要的消息。一些查找模块对版本控制的支持有限或根本不支持；请检查该查找模块的文档。

Find<PackageName>.cmake 文件通常不是由包本身提供的。相反，它通常由包外部的实体提供，例如操作系统、CMake 本身，甚至是从中调用 find_package() 命令的项目。由于是外部提供的，查找模块 往往具有启发式性质，且容易过时。它们通常搜索特定的库、文件和其他包工件。

模块模式仅由基本命令签名支持。

配置模式 (Config mode)

在此模式下，CMake 搜索匹配以下任一格式的文件

  • <PackageName>.cps

  • <lowercasePackageName>.cps

  • <lowercasePackageName>-config.cmake

  • <PackageName>Config.cmake

如果找到了后两个文件中的一个，且指定了版本详情，CMake 还会相应地查找 <lowercasePackageName>-config-version.cmake 或 <PackageName>ConfigVersion.cmake（请参阅配置模式版本选择了解这些单独版本文件的用途）。后两个选项是 CMake 脚本包描述。前两个是通用包规范 (CPS) 包描述，它们更具可移植性，并在“基础”文件中包含了版本信息。除明确注明的例外情况外，文中所有对“配置(config)文件”、“配置模式”、“包配置文件”等的引用均同样适用于 CPS 文件和 CMake 脚本文件。

注意

搜索的实现方式在大多数情况下倾向于优先选择通用包规范文件，而非 CMake 脚本配置文件。指定 CONFIGS 将禁止考虑 CPS 文件。

在配置模式下，可以为命令提供一组名称作为包名进行搜索。CMake 搜索配置和版本文件的位置比模块模式要复杂得多（请参阅配置模式搜索过程）。

配置和版本文件通常作为包的一部分安装，因此它们往往比查找模块更可靠。它们通常直接包含对包内容的了解，因此在配置或版本文件本身内部不需要进行搜索或启发式处理。

配置模式同时支持基本和完整命令签名。

FetchContent 重定向模式

3.24 版本新增: 对 find_package() 的调用可以在内部重定向到由 FetchContent 模块提供的包。对于调用者来说，其行为类似于配置模式，区别在于跳过了搜索逻辑，且不使用组件信息。详情请参阅 FetchContent_Declare() 和 FetchContent_MakeAvailable()。

当未重定向到 FetchContent 提供的包时，命令参数决定使用模块模式还是配置模式。使用基本签名时，命令首先以模块模式搜索。如果未找到包，搜索将回退到配置模式。用户可以将 CMAKE_FIND_PACKAGE_PREFER_CONFIG 变量设为 true，以反转优先级并指示 CMake 首先使用配置模式搜索，然后再回退到模块模式。基本签名也可以通过 MODULE 关键字强制仅使用模块模式。如果使用完整签名，命令将仅以配置模式搜索。

基本签名¶

find_package(<PackageName> [<version>] [EXACT] [QUIET] [MODULE]
             [REQUIRED|OPTIONAL] [[COMPONENTS] <component>...]
             [OPTIONAL_COMPONENTS <component>...]
             [REGISTRY_VIEW {64|32|64_32|32_64|HOST|TARGET|BOTH}]
             [GLOBAL]
             [NO_POLICY_SCOPE]
             [BYPASS_PROVIDER]
             [UNWIND_INCLUDE])

基本签名在模块模式和配置模式下均受支持。MODULE 关键字意味着仅可使用模块模式来查找包，不会回退到配置模式。

无论使用哪种模式，都会设置一个 <PackageName>_FOUND 变量来指示是否找到该包。当找到包时，包的特定信息可以通过该包自身记录的其他变量和 导入目标 (Imported Targets) 提供。QUIET 选项会禁用信息性消息，包括当未指定 REQUIRED 时如果未找到包的提示消息。REQUIRED 选项会在找不到包时停止处理并报错。

包所需的组件列表可以在 COMPONENTS 关键字之后列出。如果有任何组件无法满足，则该包整体被视为未找到。如果同时存在 REQUIRED 选项，则将其视为致命错误，否则程序仍继续执行。作为一种缩写形式，如果存在 REQUIRED 选项，则可以省略 COMPONENTS 关键字，直接在 REQUIRED 之后列出所需组件。

可以启用 CMAKE_FIND_REQUIRED 变量，使此调用默认变为 REQUIRED。此行为可以通过提供 OPTIONAL 关键字来覆盖。与 REQUIRED 选项一样，可以直接在 OPTIONAL 之后列出组件列表，这等同于列在 COMPONENTS 关键字之后。当给定 OPTIONAL 关键字时，找不到包时的警告输出会被抑制。

其他可选组件可以列在 OPTIONAL_COMPONENTS 之后。如果这些组件无法满足，只要满足所有必需组件，该包整体仍可被视为找到。

可用组件及其含义由目标包定义。

  • 对于 CMake 脚本包配置文件，正式上由目标包决定如何解释给定的组件信息，但应遵循上述预期。对于未指定组件的调用，没有单一的预期行为，目标包应明确定义此类情况下的结果。常见安排包括假设应查找所有组件、不查找组件或可用组件的某些明确定义的子集。

  • 通用包规范 (CPS) 包由一个根配置文件和零个或多个附录组成，每个附录提供组件并可能具有依赖关系。CMake 总是尝试加载根配置文件。仅当附录的依赖关系得到满足，且它们提供了所请求的组件（或者未请求组件）时，才会加载附录。如果提供所需组件的附录的依赖关系无法满足，则该包被视为未找到。否则，该附录将被忽略。

3.24 版本新增: REGISTRY_VIEW 关键字指定应查询哪些注册表视图。此关键字仅在 Windows 平台上有效，在所有其他平台上将被忽略。正式上，由目标包决定如何解释给定的注册表视图信息。

3.24 版本新增: 指定 GLOBAL 关键字会将所有导入目标提升到导入项目中的全局作用域。或者，可以通过设置 CMAKE_FIND_PACKAGE_TARGETS_GLOBAL 变量来启用此功能。

<version> 参数请求所找到的包应与之兼容的版本。它有两种可能的指定形式

  • 单个版本，格式为 major[.minor[.patch[.tweak]]]，其中每个组件都是数值。

  • 版本范围，格式为 versionMin...[<]versionMax，其中 versionMin 和 versionMax 具有与单个版本相同的组件为整数的格式和约束。默认情况下，两个端点都包含在内。通过指定 <，将排除上限端点。版本范围仅支持 CMake 3.19 或更高版本。

注意

除 CPS 包外，版本支持目前仅在逐个包的基础上提供。当指定了版本范围但包仅被设计为期望单个版本时，该包将忽略范围的上限端点，仅考虑范围下限处的单个版本。支持版本范围的非 CPS 包以各自包决定的方式进行。详情和重要注意事项请参阅下文的版本选择章节。

EXACT 选项请求版本精确匹配。此选项与版本范围的指定不兼容。

如果查找模块内的递归调用未给出 <version> 和/或组件列表，则相应参数会自动从外部调用转发（包括 <version> 的 EXACT 标志）。

有关 NO_POLICY_SCOPE 选项的讨论，请参阅 cmake_policy() 命令文档。

3.24 版本新增: BYPASS_PROVIDER 关键字仅允许在 find_package() 被依赖提供程序调用时使用。提供程序可以使用它来直接调用内置的 find_package() 实现，并防止该调用被重新路由回自身。CMake 的未来版本可能会检测到从非依赖提供程序位置使用此关键字的尝试并以致命错误中止。

4.2 版本新增: UNWIND_INCLUDE 关键字仅允许在父级 find_package() 调用内调用 find_package() 时使用。当 find_package(UNWIND_INCLUDE) 调用未能找到所需的包时，它将启动“回溯 (unwind)”状态。在此状态下，后续对 find_package() 和 include() 的调用均被禁止，并且所有父级 include() 命令在达到其作用域时将立即调用 return()。此“回溯”将持续到返回到父级 find_package() 为止。

UNWIND_INCLUDE 仅旨在由 install(EXPORT_PACKAGE_DEPENDENCIES) 生成的 find_package() 调用使用，但对于希望以类似方式手动管理依赖项的用户来说可能很有用。

完整签名¶

find_package(<PackageName> [version] [EXACT] [QUIET]
             [REQUIRED|OPTIONAL] [[COMPONENTS] <component>...]
             [OPTIONAL_COMPONENTS <component>...]
             [CONFIG|NO_MODULE]
             [GLOBAL]
             [NO_POLICY_SCOPE]
             [BYPASS_PROVIDER]
             [NAMES <name>...]
             [CONFIGS <config>...]
             [HINTS <path>...]
             [PATHS <path>...]
             [REGISTRY_VIEW {64|32|64_32|32_64|HOST|TARGET|BOTH}]
             [PATH_SUFFIXES <suffix>...]
             [NO_DEFAULT_PATH]
             [NO_PACKAGE_ROOT_PATH]
             [NO_CMAKE_PATH]
             [NO_CMAKE_ENVIRONMENT_PATH]
             [NO_SYSTEM_ENVIRONMENT_PATH]
             [NO_CMAKE_PACKAGE_REGISTRY]
             [NO_CMAKE_BUILDS_PATH] # Deprecated; does nothing.
             [NO_CMAKE_SYSTEM_PATH]
             [NO_CMAKE_INSTALL_PREFIX]
             [NO_CMAKE_SYSTEM_PACKAGE_REGISTRY]
             [CMAKE_FIND_ROOT_PATH_BOTH |
              ONLY_CMAKE_FIND_ROOT_PATH |
              NO_CMAKE_FIND_ROOT_PATH])

CONFIG 选项、同义词 NO_MODULE 选项，或使用未在基本签名中指定的选项，均会强制执行纯配置模式。在纯配置模式下，命令跳过模块模式搜索，直接开始配置模式搜索。

配置模式搜索尝试定位包提供的配置文件。创建一个名为 <PackageName>_DIR 的缓存条目以保存包含该文件的目录。默认情况下，该命令搜索名称为 <PackageName> 的包。如果给定了 NAMES 选项，则使用其后的名称代替 <PackageName>。在确定是否将调用重定向到由 FetchContent 提供的包时，也会考虑这些名称。

对于指定的每个包名，命令搜索其文件名匹配以下任一格式的文件

  • <PackageName>Config.cmake

  • <lowercasePackageName>-config.cmake

  • <PackageName>.cps

  • <lowercasePackageName>.cps

可以使用 CONFIGS 选项提供一组替代的配置文件名。下文指定了配置模式搜索过程。找到后，检查任何版本约束，如果满足，则由 CMake 读取并处理配置文件。由于该文件由包提供，它已经知道包内容的位置。配置文件的完整路径存储在 CMake 变量 <PackageName>_CONFIG 中。

注意

由于 CPS 文件不允许具有不匹配包名的名称，因此指定 CONFIGS 将禁止搜索 CPS 文件。

CMake 在搜索具有适当版本的包时考虑过的所有配置文件都存储在 <PackageName>_CONSIDERED_CONFIGS 变量中，关联的版本存储在 <PackageName>_CONSIDERED_VERSIONS 变量中。

如果找不到包配置文件，除非指定了 QUIET 参数，否则 CMake 将生成一个描述问题的错误。如果指定了 REQUIRED 且未找到包，则会生成致命错误并停止执行配置步骤。如果 <PackageName>_DIR 已被设置为不包含配置文件的目录，或者请求的版本与在该目录中找到的包不兼容（请参阅配置模式版本选择），CMake 将忽略它并从头开始搜索。

鼓励提供包配置文件的包维护者对其进行命名和安装，以便下文概述的配置模式搜索过程能够找到它们，而无需使用额外选项。

总是搜索。

3.24 版本新增: 可以指定 REGISTRY_VIEW 来管理作为 PATHS 和 HINTS 一部分指定的 Windows 注册表查询。

指定必须查询哪些注册表视图。此选项仅在 Windows 平台上有效，在其他平台上将被忽略。未指定时，当 CMP0134 策略为 NEW 时使用 TARGET 视图。当策略为 OLD 时，默认视图请参阅 CMP0134。

64

查询64位注册表。在32位Windows上，它始终返回字符串 /REGISTRY-NOTFOUND。

32

查询32位注册表。

64_32

查询两个视图（64 和 32）并为每个视图生成一个路径。

32_64

查询两个视图（32 和 64）并为每个视图生成一个路径。

HOST

查询与主机架构匹配的注册表：64 位 Windows 上为 64，32 位 Windows 上为 32。

TARGET

查询与 CMAKE_SIZEOF_VOID_P 变量指定的架构相匹配的注册表。如果未定义，则回退到 HOST 视图。

BOTH

查询两个视图（32 和 64）。顺序取决于以下规则：如果定义了 CMAKE_SIZEOF_VOID_P 变量，则根据此变量的内容使用以下视图

  • 8: 64_32

  • 4: 32_64

如果未定义 CMAKE_SIZEOF_VOID_P 变量，则依赖主机架构。

  • 64位: 64_32

  • 32位: 32

如果指定了 PATH_SUFFIXES，则后缀将逐一附加到每个 (W) 或 (U) 目录条目。

这组目录旨在与在安装树中提供配置文件的项目配合使用。上文中标记为 (W) 的目录旨在用于 Windows 上的安装，其中前缀可能指向应用程序安装目录的顶部。那些标记为 (U) 的目录旨在用于 UNIX 平台上前缀由多个包共享的安装。这仅是一种约定，因此所有 (W) 和 (U) 目录仍会在所有平台上搜索。标记为 (A) 的目录旨在用于 Apple 平台上的安装。CMAKE_FIND_FRAMEWORK 和 CMAKE_FIND_APPBUNDLE 变量决定了优先顺序。

警告

将 CMAKE_FIND_FRAMEWORK 或 CMAKE_FIND_APPBUNDLE 设置为非 FIRST（默认值）的值，将导致 CMake 以不同于规范中规定的顺序搜索通用包规范文件。

安装前缀集按以下步骤构建。如果指定了 NO_DEFAULT_PATH，则启用所有 NO_* 选项。

  1. 搜索特定于当前正在查找的 <PackageName> 的前缀。请参阅策略 CMP0074。

    3.12 版本新增。

    具体来说，按顺序搜索以下变量指定的前缀

    1. <PackageName>_ROOT CMake 变量，其中 <PackageName> 是大小写保留的包名。

    2. CMake 变量 <PACKAGENAME>_ROOT，其中 <PACKAGENAME> 是大写的包名称。请参阅策略 CMP0144。

      在 3.27 版本中新增。

    3. <PackageName>_ROOT 环境变量，其中 <PackageName> 是大小写保留的包名。

    4. 环境变量 <PACKAGENAME>_ROOT，其中 <PACKAGENAME> 是大写的包名称。请参阅策略 CMP0144。

      在 3.27 版本中新增。

    包根变量作为堆栈维护，因此如果从查找模块内调用，父级查找模块的根路径也会在当前包的路径之后被搜索。如果传递 NO_PACKAGE_ROOT_PATH 或将 CMAKE_FIND_USE_PACKAGE_ROOT_PATH 设置为 FALSE，则可以跳过此步骤。

  2. 搜索 CMake 特定缓存变量中指定的路径。这些旨在通过 -DVAR=VALUE 在命令行上使用。这些值被解释为分号分隔的列表。如果传递 NO_CMAKE_PATH 或将 CMAKE_FIND_USE_CMAKE_PATH 设置为 FALSE，则可以跳过此步骤

    • CMAKE_PREFIX_PATH

    • CMAKE_FRAMEWORK_PATH

    • CMAKE_APPBUNDLE_PATH

  3. 搜索 CMake 特定环境变量中指定的路径。这些旨在在用户的 shell 配置中设置，因此使用主机的原生路径分隔符（Windows 上为 ;，UNIX 上为 :）。如果传递 NO_CMAKE_ENVIRONMENT_PATH 或将 CMAKE_FIND_USE_CMAKE_ENVIRONMENT_PATH 设置为 FALSE，则可以跳过此步骤

    • <PackageName>_DIR

    • CMAKE_PREFIX_PATH

    • CMAKE_FRAMEWORK_PATH

    • CMAKE_APPBUNDLE_PATH

  4. 搜索 HINTS 选项指定的路径。这些应为通过系统自省计算出的路径，例如通过已找到的另一个项目的位置提供的提示。硬编码的猜测应使用 PATHS 选项指定。

  5. 搜索标准系统环境变量。如果传递 NO_SYSTEM_ENVIRONMENT_PATH 或将 CMAKE_FIND_USE_SYSTEM_ENVIRONMENT_PATH 设置为 FALSE，则可以跳过此步骤。以 /bin 或 /sbin 结尾的路径条目会自动转换为其父目录

    • PATH

  6. 搜索存储在 CMake 用户包注册表 中的路径。如果传递 NO_CMAKE_PACKAGE_REGISTRY 或将变量 CMAKE_FIND_USE_PACKAGE_REGISTRY 设置为 FALSE，或者将已弃用的变量 CMAKE_FIND_PACKAGE_NO_PACKAGE_REGISTRY 设置为 TRUE，则可以跳过此步骤。

    有关用户包注册表的详情，请参阅 cmake-packages(7) 手册。

  7. 搜索当前系统平台文件中定义的 CMake 变量。如果传递 NO_CMAKE_INSTALL_PREFIX 或将 CMAKE_FIND_USE_INSTALL_PREFIX 设置为 FALSE，则可以跳过 CMAKE_INSTALL_PREFIX 和 CMAKE_STAGING_PREFIX 的搜索。如果传递 NO_CMAKE_SYSTEM_PATH 或将 CMAKE_FIND_USE_CMAKE_SYSTEM_PATH 设置为 FALSE，则可以跳过所有这些位置

    • CMAKE_SYSTEM_PREFIX_PATH

    • CMAKE_SYSTEM_FRAMEWORK_PATH

    • CMAKE_SYSTEM_APPBUNDLE_PATH

    这些变量包含的平台路径通常包括已安装的软件位置。例如，UNIX 平台上的 /usr/local。

  8. 搜索存储在 CMake 系统包注册表 中的路径。如果传递 NO_CMAKE_SYSTEM_PACKAGE_REGISTRY 或将 CMAKE_FIND_USE_SYSTEM_PACKAGE_REGISTRY 变量设置为 FALSE，或者将已弃用的变量 CMAKE_FIND_PACKAGE_NO_SYSTEM_PACKAGE_REGISTRY 设置为 TRUE，则可以跳过此步骤。

    有关系统包注册表的详情，请参阅 cmake-packages(7) 手册。

  9. 搜索 PATHS 选项指定的路径。这些通常是硬编码的猜测。

CMAKE_IGNORE_PATH、CMAKE_IGNORE_PREFIX_PATH、CMAKE_SYSTEM_IGNORE_PATH 和 CMAKE_SYSTEM_IGNORE_PREFIX_PATH 变量也会导致上述部分位置被忽略。

路径按上述顺序搜索。第一个找到的可行包配置文件将被使用，即使列表中后续位置有更新版本的包。

对于包含通配符表达式 (*) 的搜索路径，默认情况下按自然降序搜索匹配通配符的目录。此行为可以通过设置 CMAKE_FIND_PACKAGE_SORT_ORDER 和 CMAKE_FIND_PACKAGE_SORT_DIRECTION 变量来覆盖。这些变量决定了 CMake 考虑通配符匹配的顺序。例如，如果文件系统包含包配置文件

<prefix>/example-1.2/example-config.cmake
<prefix>/example-1.10/example-config.cmake
<prefix>/share/example-2.0/example-config.cmake

那么（当上述变量未设置时）find_package(example) 将选取 example-1.10（假设 example-1.2 和 example-1.10 都是可行的）。但请注意，find_package *不会* 找到 example-2.0，因为前两个中的一个会被先找到。

要控制 find_package 搜索匹配通配符表达式的目录的顺序，请使用 CMAKE_FIND_PACKAGE_SORT_ORDER 和 CMAKE_FIND_PACKAGE_SORT_DIRECTION。例如，要使上述示例选择 example-1.2，可以在调用 find_package 之前设置

set(CMAKE_FIND_PACKAGE_SORT_ORDER NATURAL)
set(CMAKE_FIND_PACKAGE_SORT_DIRECTION ASC)

在调用 find_package 之前。

3.16 版本新增: 添加了 CMAKE_FIND_USE_<CATEGORY> 变量以全局禁用各种搜索位置。

4.0 版本更改: 变量 CMAKE_FIND_PACKAGE_SORT_ORDER 和 CMAKE_FIND_PACKAGE_SORT_DIRECTION 现在也控制 find_package 搜索路径 <prefix>/<name>.framework/Versions/*/Resources/ 和 <prefix>/<name>.framework/Versions/*/Resources/CMake 中匹配通配符表达式的目录的顺序。在旧版本的 CMake 中，此顺序未指定。

4.2 版本更改: 当遇到多个可行匹配时，find_package 现在默认选择版本最新的一个。在旧版本的 CMake 中，结果未指定。相应地，CMAKE_FIND_PACKAGE_SORT_ORDER 的默认值已从 NONE 更改为 NATURAL，且 CMAKE_FIND_PACKAGE_SORT_DIRECTION 现在默认为 DEC（降序）而不是 ASC（升序）。

CMake 变量 CMAKE_FIND_ROOT_PATH 指定一个或多个要添加到所有其他搜索目录之前的前缀路径。这有效地在给定位置下“重定向(re-roots)”了整个搜索。作为 CMAKE_STAGING_PREFIX 子代的路径被排除在此重定向之外，因为该变量始终是主机系统上的路径。默认情况下 CMAKE_FIND_ROOT_PATH 为空。

变量 CMAKE_SYSROOT 也可用于指定仅一个目录作为前缀。设置 CMAKE_SYSROOT 也有其他影响。有关更多信息，请参阅该变量的文档。

这些变量在交叉编译时特别有用，可以指向目标环境的根目录，CMake 也会在那里进行搜索。默认情况下首先搜索 CMAKE_FIND_ROOT_PATH 中列出的目录，然后搜索 CMAKE_SYSROOT 目录，最后搜索非重定向的目录。默认行为可以通过设置 CMAKE_FIND_ROOT_PATH_MODE_PACKAGE 来调整。此行为可以通过选项在每个调用基础上手动覆盖

CMAKE_FIND_ROOT_PATH_BOTH

按上述顺序搜索。

NO_CMAKE_FIND_ROOT_PATH

不要使用 CMAKE_FIND_ROOT_PATH 变量。

ONLY_CMAKE_FIND_ROOT_PATH

仅搜索重定向的目录以及 CMAKE_STAGING_PREFIX 下的目录。

默认搜索顺序旨在为常见用例实现从最具体到最不具体的排序。项目可以通过多次调用命令并使用 NO_* 选项来覆盖此顺序。

find_package (<PackageName> PATHS <paths>... NO_DEFAULT_PATH)
find_package (<PackageName>)

一旦其中一个调用成功，结果变量将被设置并存储在缓存中，这样就不会再次搜索。

默认情况下，结果变量中存储的值将是找到该文件的路径。CMAKE_FIND_PACKAGE_RESOLVE_SYMLINKS 变量可以在调用 find_package 之前设置为 TRUE，以解析符号链接并存储文件的实际路径。

每个非 REQUIRED 的 find_package 调用都可以被禁用或设为 REQUIRED

  • 将 CMAKE_DISABLE_FIND_PACKAGE_<PackageName> 变量设置为 TRUE 将禁用该包。这也将禁用对 FetchContent 提供的包的重定向。

  • 将 CMAKE_REQUIRE_FIND_PACKAGE_<PackageName> 变量设置为 TRUE 使该包变为 REQUIRED。

将两个变量同时设置为 TRUE 是错误的。

CMAKE_REQUIRE_FIND_PACKAGE_<PackageName> 变量在确定包是否为必需时，优先于 OPTIONAL 关键字。

  • 版本范围的上限充当了可接受版本的硬性限制。因此，虽然对 1.4.0 版本的请求可能会被版本为 1.6.0 且声明“主版本号相同”兼容性的包所满足，但如果请求的版本范围是 1.4.0...1.5.0，则同一个包将被拒绝。

通用包规范 (Common Package Specification)¶

对于通用包规范 (CPS) 包配置文件，包版本号由 CMake 根据一组公认的版本方案进行检查。目前，识别以下方案

simple

版本号是由整数组成的元组，后跟一个可选的尾随段，该段在版本比较时会被忽略。

自定义

解释版本号的机制未指定。版本字符串必须完全匹配才能使包被接受。

有关每种方案的详细说明以及如何执行比较的解释，请参考 version_schema。请注意，规范中可能包含 CMake 不支持的方案。

除了包的 version 外，CPS 还允许包选择性地指定 compat_version，即包提供兼容性的最旧版本。也就是说，包保证期望 compat_version 的使用者应该能够使用该包，即使包的实际版本较新。如果未指定，compat_version 隐含地等于包版本，即不提供向后兼容性。

当包使用识别的方案时，CMake 将根据以下规则确定包的可接受性

  • 如果指定了 EXACT，或者如果包没有提供 compat_version，则包的 version 必须等于请求的版本。

  • 否则

    • 包的 version 必须大于或等于请求的（最小）版本，且

    • 包的 compat_version 必须小于或等于请求的（最小）版本，且

    • 如果给定了请求的最高版本，则该版本必须大于（或等于，取决于最高版本是指定为包含还是排除）包的 version。

注意

选择这种范围匹配实现是为了尽可能接近 write_basic_package_version_file() 的行为，尽管排除了过于宽泛的范围导致无法匹配任何内容的情况。

对于使用 simple 版本方案的包，如果版本可接受，则会设置以下变量

<PackageName>_VERSION

完整的提供版本字符串

<PackageName>_VERSION_MAJOR

如果提供了版本，则为主版本号，否则为 0

<PackageName>_VERSION_MINOR

如果提供了版本，则为次版本号，否则为 0

<PackageName>_VERSION_PATCH

如果提供了版本，则为补丁版本号，否则为 0

<PackageName>_VERSION_TWEAK

如果提供了版本，则为修订版本号，否则为 0

<PackageName>_VERSION_COUNT

版本组件的数量，非负

无论指定的是单个版本还是版本范围，变量 <PackageName>_FIND_VERSION_COMPLETE 都会被定义，并保存所指定的完整请求版本字符串。

在模块模式 (Module mode) 下，加载的查找模块负责处理这些变量中详细说明的请求；详情请参阅查找模块。在配置模式 (Config mode) 下，find_package 会自动处理 REQUIRED、QUIET 和 <version> 选项，但将组件的处理留给包配置文件，以适合该包的方式进行。包配置文件可以将 <PackageName>_FOUND 设置为 false，以告知 find_package 组件要求未得到满足。

CPS 传递依赖要求¶

通用包规范包描述由一个或多个组件组成，这些组件可能又依赖于包内部或外部的其他组件。当需要外部组件时，所提供的包被标记为该包的包级要求。此外，所需的组件集通常会在上述外部包要求中注明。

在 CMake 脚本包描述使用 find_dependency() 命令来处理传递依赖的地方，CMake 会使用内部嵌套的 find_package 调用来为 CPS 本身处理传递依赖。此调用可以通过另一个 CPS 包或 CMake 脚本包来解析 CPS 包依赖项。处理 CPS 组件依赖项的方式受到一些注意事项的限制。

当解析传递依赖的候选者是另一个 CPS 包时，事情很简单；COMPONENTS 和 CPS “组件”可以直接比较（实际上与 CMake 的“导入目标 (imported targets)”同义）。然而，鼓励（且经常这样做）CMake 脚本包检查是否找到了所需的组件，无论该包是否描述了单独的组件。此外，即使那些确实描述了组件的包，通常也没有与 CPS 正常情况下同等的导入目标相关性。因此，将 CPS 包声明的所需组件集传递给 COMPONENTS 将导致解析依赖项出现虚假失败。

为了解决这个问题，如果解析 CPS 传递依赖的候选者是 CMake 脚本包，CMake 会将消费 CPS 包声明的所需组件作为 OPTIONAL_COMPONENTS 传递，并执行单独的内部检查，以确认候选包是否提供了所需的导入目标。这些目标必须根据 CPS 约定命名为 <PackageName>::<ComponentName>，否则检查将认为该包未找到。
