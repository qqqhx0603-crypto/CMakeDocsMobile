尝试构建一些代码。

尝试编译整个项目¶

try_compile(<compileResultVar> PROJECT <projectName>
            SOURCE_DIR <srcdir>
            [BINARY_DIR <bindir>]
            [TARGET <targetName>]
            [LOG_DESCRIPTION <text>]
            [NO_CACHE]
            [NO_LOG]
            [CMAKE_FLAGS <flags>...]
            [OUTPUT_VARIABLE <var>])

代码说明

  • try_compile(<compileResultVar> PROJECT <projectName> SOURCE_DIR <srcdir> [BINARY_DIR <bindir>] [TARGET <targetName>] [LOG_DESCRIPTION <text>] [NO_CACHE] [NO_LOG] [CMAKE_FLAGS <flags>...] [OUTPUT_VARIABLE <var>])：调用 `try_compile()` 命令，具体行为由这一行传入的参数决定。

在 3.25 版本中新增。

尝试构建一个项目。构建成功时在 <compileResultVar> 中返回 TRUE，构建失败时返回 FALSE。

在这种形式中，<srcdir> 应包含一个完整的 CMake 项目，其中包括 CMakeLists.txt 文件及所有源文件。命令运行后，<bindir> 和 <srcdir> 不会被删除。指定 <targetName> 以构建特定目标，而不是 all 或 ALL_BUILD 目标。关于其他选项的含义，请参阅下文。

版本 3.24 变更: 描述平台设置的 CMake 变量，以及 CMAKE_TRY_COMPILE_PLATFORM_VARIABLES 变量中列出的变量，会被传播到项目的构建配置中。请参阅策略 CMP0137。在此之前，这仅由 源文件 签名完成。

版本 3.26 新增: 如果不指定 NO_LOG 选项，此命令会记录一个 configure-log try_compile 事件。

版本 3.30 新增: 如果设置了 PROPAGATE_TOP_LEVEL_INCLUDES_TO_TRY_COMPILE 全局属性，则 CMAKE_PROJECT_TOP_LEVEL_INCLUDES 会被传播到项目的构建配置中。

该命令为 3.25 之前的 CMake 版本提供了一个替代签名。为了清晰起见，推荐使用上述签名。

try_compile(<compileResultVar> <bindir> <srcdir>
            <projectName> [<targetName>]
            [CMAKE_FLAGS <flags>...]
            [OUTPUT_VARIABLE <var>])

代码说明

  • try_compile(<compileResultVar> <bindir> <srcdir> <projectName> [<targetName>] [CMAKE_FLAGS <flags>...] [OUTPUT_VARIABLE <var>])：调用 `try_compile()` 命令，具体行为由这一行传入的参数决定。

尝试编译源文件¶

try_compile(<compileResultVar>
            [SOURCES_TYPE <type>]
            {SOURCES <srcfile>...                 |
             SOURCE_FROM_CONTENT <name> <content> |
             SOURCE_FROM_VAR <name> <var>         |
             SOURCE_FROM_FILE <name> <path>       }...
            [LOG_DESCRIPTION <text>]
            [NO_CACHE]
            [NO_LOG]
            [CMAKE_FLAGS <flags>...]
            [COMPILE_DEFINITIONS <defs>...]
            [LINK_OPTIONS <options>...]
            [LINK_LIBRARIES <libs>...]
            [LINKER_LANGUAGE <lang>]
            [OUTPUT_VARIABLE <var>]
            [COPY_FILE <fileName> [COPY_FILE_ERROR <var>]]
            [<LANG>_STANDARD <std>]
            [<LANG>_STANDARD_REQUIRED <bool>]
            [<LANG>_EXTENSIONS <bool>]
            )

代码说明

  • try_compile(<compileResultVar> [SOURCES_TYPE <type>] {SOURCES <srcfile>... | SOURCE_FROM_CONTENT <name> <content> | SOURCE_FROM_VAR <name> <var> | SOURCE_FROM_FILE <name> <path> }... [LOG_DESCRIPTION <text>] [NO_CACHE] [NO_LOG] [CMAKE_FLAGS <flags>...] [COMPILE_DEFINITIONS <defs>...] [LINK_OPTIONS <options>...] [LINK_LIBRARIES <libs>...] [LINKER_LANGUAGE <lang>] [OUTPUT_VARIABLE <var>] [COPY_FILE <fileName> [COPY_FILE_ERROR <var>]] [<LANG>_STANDARD <std>] [<LANG>_STANDARD_REQUIRED <bool>] [<LANG>_EXTENSIONS <bool>] )：调用 `try_compile()` 命令，具体行为由这一行传入的参数决定。

在 3.25 版本中新增。

尝试从一个或多个源文件构建可执行文件或静态库。二进制类型由变量 CMAKE_TRY_COMPILE_TARGET_TYPE 决定。构建成功时，布尔值 true 将存入 <compileResultVar>；构建失败时存入布尔值 false（除非指定了 NO_CACHE，否则该结果会被缓存）。

在这种形式中，必须提供一个或多个源文件。此外，SOURCES 和/或 SOURCE_FROM_* 中的一个必须在其他关键字之前出现。

如果 CMAKE_TRY_COMPILE_TARGET_TYPE 未设置或设置为 EXECUTABLE，则源文件必须包含 main 定义，且 CMake 将创建一个 CMakeLists.txt 文件以将源文件构建为可执行文件。如果 CMAKE_TRY_COMPILE_TARGET_TYPE 设置为 STATIC_LIBRARY，则会构建一个静态库，且不需要 main 定义。对于可执行文件，生成的 CMakeLists.txt 文件将包含类似以下内容

add_definitions(<expanded COMPILE_DEFINITIONS from caller>)
include_directories(${INCLUDE_DIRECTORIES})
link_directories(${LINK_DIRECTORIES})
add_executable(cmTryCompileExec <srcfile>...)
target_link_options(cmTryCompileExec PRIVATE <LINK_OPTIONS from caller>)
target_link_libraries(cmTryCompileExec ${LINK_LIBRARIES})

代码说明

  • add_definitions(<expanded COMPILE_DEFINITIONS from caller>) include_directories(${INCLUDE_DIRECTORIES}) link_directories(${LINK_DIRECTORIES}) add_executable(cmTryCompileExec <srcfile>...) target_link_options(cmTryCompileExec PRIVATE <LINK_OPTIONS from caller>) target_link_libraries(cmTryCompileExec ${LINK_LIBRARIES})：调用 `add_definitions()` 命令，具体行为由这一行传入的参数决定。

对于每个 try_compile 操作，CMake 都会自动在 ${CMAKE_BINARY_DIR}/CMakeFiles/CMakeScratch 下生成一个名称未指定的唯一目录。除非将 --debug-trycompile 传递给 cmake，否则这些目录会被自动清理。在任何 cmake 执行开始时，之前运行留下的此类目录也会被无条件清理。

该命令为 3.25 之前的 CMake 版本提供了一个替代签名。为了清晰起见，推荐使用上述签名。

try_compile(<compileResultVar> <bindir> {<srcfile>|SOURCES <srcfile>...}
            [CMAKE_FLAGS <flags>...]
            [COMPILE_DEFINITIONS <defs>...]
            [LINK_OPTIONS <options>...]
            [LINK_LIBRARIES <libs>...]
            [OUTPUT_VARIABLE <var>]
            [COPY_FILE <fileName> [COPY_FILE_ERROR <var>]]
            [<LANG>_STANDARD <std>]
            [<LANG>_STANDARD_REQUIRED <bool>]
            [<LANG>_EXTENSIONS <bool>]
            )

代码说明

  • try_compile(<compileResultVar> <bindir> {<srcfile>|SOURCES <srcfile>...} [CMAKE_FLAGS <flags>...] [COMPILE_DEFINITIONS <defs>...] [LINK_OPTIONS <options>...] [LINK_LIBRARIES <libs>...] [OUTPUT_VARIABLE <var>] [COPY_FILE <fileName> [COPY_FILE_ERROR <var>]] [<LANG>_STANDARD <std>] [<LANG>_STANDARD_REQUIRED <bool>] [<LANG>_EXTENSIONS <bool>] )：调用 `try_compile()` 命令，具体行为由这一行传入的参数决定。

在此版本中，try_compile 将使用 <bindir>/CMakeFiles/CMakeTmp 进行操作，所有此类文件都会被自动清理。为了调试，可以将 --debug-trycompile 传递给 cmake 以避免此清理。然而，如果多次连续的 try_compile 操作使用相同的 <bindir>，它们将重用这个单一的输出目录，这意味着您一次只能调试一个此类 try_compile 调用。建议使用较新的签名以简化对多个 try_compile 操作的调试。

选项¶

上述签名的选项包括

CMAKE_FLAGS <flags>...

指定 -DVAR:TYPE=VALUE 形式的标志，这些标志将传递给用于驱动测试构建的 cmake(1) 命令行。上述示例展示了如何使用变量 COMPILE_DEFINITIONS、INCLUDE_DIRECTORIES、LINK_DIRECTORIES、LINK_LIBRARIES 和 LINK_OPTIONS 的值。编译器选项可以通过 CMAKE_FLAGS -DCOMPILE_DEFINITIONS=-Werror 传入。

COMPILE_DEFINITIONS <defs>...

指定要传递给生成的测试项目中 add_definitions() 的 -Ddefinition 参数。

COPY_FILE <fileName>

将生成的内置可执行文件或静态库复制到给定的 <fileName>。

COPY_FILE_ERROR <var>

在 COPY_FILE 之后使用，将尝试复制文件时遇到的任何错误信息存入变量 <var> 中。

LINK_LIBRARIES <libs>...

指定在生成的项目中要链接的库。库列表可以引用系统库以及调用项目中的 导入目标 (Imported Targets)。

如果指定了此选项，则提供给 CMAKE_FLAGS 选项的任何 -DLINK_LIBRARIES=... 值都将被忽略。

版本 3.29 新增: 也支持导入库的别名目标。

LINK_OPTIONS <options>...

3.14 版新增。

指定链接步骤选项，这些选项将根据生成的项目中的 CMAKE_TRY_COMPILE_TARGET_TYPE 变量，传递给 target_link_options() 或用于设置 STATIC_LIBRARY_OPTIONS 目标属性。

LINKER_LANGUAGE <lang>

在版本 3.29 中添加。

指定生成的项目的 LINKER_LANGUAGE 目标属性。当同时使用多种语言的源文件时，请将其设置为包含程序入口点（例如 main）的源文件的语言。

LOG_DESCRIPTION <text>

3.26 版新增。

指定一个非空的文本描述，说明检查的目的。这会记录在 cmake-configure-log(7) 条目中。

NO_CACHE

在 3.25 版本中新增。

<compileResultVar> 将存储在普通变量中，而不是缓存条目中。

<compileResultVar> 通常被缓存，以便可以使用一种简单的模式来避免在随后的 CMake 执行中重复进行测试。

if(NOT DEFINED RESULTVAR)
  # ...(check-specific setup code)...
  try_compile(RESULTVAR ...)
  # ...(check-specific logging and cleanup code)...
endif()

代码说明

  • if(NOT DEFINED RESULTVAR) # ...(check-specific setup code)... try_compile(RESULTVAR ...) # ...(check-specific logging and cleanup code)... endif()：开始一个 `if()` 条件判断，根据条件结果选择是否执行后续语句。

如果防护变量和结果变量不相同（例如，测试是更大检查的一部分），则 NO_CACHE 可能有助于避免将中间结果变量泄漏到缓存中。

NO_LOG

3.26 版新增。

不要为此调用记录 cmake-configure-log(7) 条目。

OUTPUT_VARIABLE <var>

将构建过程的输出存储在给定的变量中。

SOURCE_FROM_CONTENT <name> <content>

在 3.25 版本中新增。

将 <content> 写入操作目录中名为 <name> 的文件。这可以用于在文件内容动态指定时，绕过单独编写源文件的需要。指定的 <name> 不允许包含路径组件。

SOURCE_FROM_CONTENT 可以被多次指定。

SOURCE_FROM_FILE <name> <path>

在 3.25 版本中新增。

将 <path> 复制到操作目录中名为 <name> 的文件。这可用于将文件整合到操作目录中，如果某个已经存在的源文件（即作为项目源代码库中的独立文件）需要引用由 SOURCE_FROM_* 创建的其他文件，这会很有用。（否则，SOURCES 通常更方便。）指定的 <name> 不允许包含路径组件。

SOURCE_FROM_VAR <name> <var>

在 3.25 版本中新增。

将 <var> 的内容写入操作目录中名为 <name> 的文件。这与 SOURCE_FROM_CONTENT 相同，但从指定的 CMake 变量中获取内容，而不是直接从文本获取，这在通过包装 try_compile 的函数传递参数时很有用。指定的 <name> 不允许包含路径组件。

SOURCE_FROM_VAR 可以被多次指定。

SOURCES_TYPE <type>

版本 3.28 新增。

源文件可以使用 SOURCES_TYPE 参数进行分类。一旦指定，所有后续指定的源文件将被视为该类型，直到给定另一个 SOURCES_TYPE 为止。可用类型包括

NORMAL

源文件不会被添加到生成的项目中的任何 FILE_SET 中。

CXX_MODULE

版本 3.28 新增。

源文件被添加到生成的项目中类型为 CXX_MODULES 的 FILE_SET 中。

源文件的默认类型是 NORMAL。

<LANG>_STANDARD <std>

版本 3.8 新增。

指定生成的项目的 C_STANDARD、CXX_STANDARD、OBJC_STANDARD、OBJCXX_STANDARD 或 CUDA_STANDARD 目标属性。

<LANG>_STANDARD_REQUIRED <bool>

版本 3.8 新增。

指定生成的项目的 C_STANDARD_REQUIRED、CXX_STANDARD_REQUIRED、OBJC_STANDARD_REQUIRED、OBJCXX_STANDARD_REQUIRED 或 CUDA_STANDARD_REQUIRED 目标属性。

<LANG>_EXTENSIONS <bool>

版本 3.8 新增。

指定生成的项目的 C_EXTENSIONS、CXX_EXTENSIONS、OBJC_EXTENSIONS、OBJCXX_EXTENSIONS 或 CUDA_EXTENSIONS 目标属性。

其他行为设置¶

版本 3.4 新增: 如果设置了，以下变量将传递到生成的 try_compile CMakeLists.txt 中，以使用默认值初始化编译目标属性

  • CMAKE_CUDA_RUNTIME_LIBRARY

  • CMAKE_ENABLE_EXPORTS

  • CMAKE_EXECUTABLE_ENABLE_EXPORTS

  • CMAKE_SHARED_LIBRARY_ENABLE_EXPORTS

  • CMAKE_EXE_LINKER_FLAGS，除非使用 4.0 之前的 CMake 版本且未将策略 CMP0056 设置为 NEW

  • CMAKE_<LANG>_LINK_FLAGS，除非使用 4.3 之前的 CMake 版本且未将策略 CMP0210 设置为 NEW

  • CMAKE_LINK_SEARCH_START_STATIC

  • CMAKE_LINK_SEARCH_END_STATIC

  • CMAKE_MSVC_RUNTIME_LIBRARY

  • CMAKE_POSITION_INDEPENDENT_CODE

  • CMAKE_WATCOM_RUNTIME_LIBRARY

版本 3.14 变更: 如果 CMP0083 设置为 NEW，为了在链接时获得正确的行为，必须在调用 try_compile 命令之前调用 CheckPIESupported 模块中的 check_pie_supported() 命令。

一些策略会根据需要自动在生成的测试项目中设置，以尊重调用项目的状态

  • CMP0065 (在 4.0 之前的 CMake 版本中)

  • CMP0083

  • CMP0091

  • CMP0104

  • CMP0123

  • CMP0126

  • CMP0128

  • CMP0136

  • CMP0141

  • CMP0155

  • CMP0157

  • CMP0181

  • CMP0184

  • CMP0210

设置变量 CMAKE_TRY_COMPILE_CONFIGURATION 以选择构建配置

  • 对于多配置生成器，这会选择要构建的配置。

  • 对于单配置生成器，这会在测试项目中设置 CMAKE_BUILD_TYPE。

版本 3.6 新增: 设置 CMAKE_TRY_COMPILE_TARGET_TYPE 变量以指定用于源文件签名的目标类型。

版本 3.6 新增: 设置 CMAKE_TRY_COMPILE_PLATFORM_VARIABLES 变量以指定必须传播到测试项目中的变量。此变量旨在仅在工具链文件中使用，并且仅由 try_compile() 命令用于源文件形式时生效，而不是当给定整个项目时。

版本 3.8 变更: 如果 CMP0067 设置为 NEW，或者使用了 <LANG>_STANDARD、<LANG>_STANDARD_REQUIRED 或 <LANG>_EXTENSIONS 选项中的任何一个，则语言标准变量会被尊重

  • CMAKE_C_STANDARD

  • CMAKE_C_STANDARD_REQUIRED

  • CMAKE_C_EXTENSIONS

  • CMAKE_CXX_STANDARD

  • CMAKE_CXX_STANDARD_REQUIRED

  • CMAKE_CXX_EXTENSIONS

  • CMAKE_OBJC_STANDARD

  • CMAKE_OBJC_STANDARD_REQUIRED

  • CMAKE_OBJC_EXTENSIONS

  • CMAKE_OBJCXX_STANDARD

  • CMAKE_OBJCXX_STANDARD_REQUIRED

  • CMAKE_OBJCXX_EXTENSIONS

  • CMAKE_CUDA_STANDARD

  • CMAKE_CUDA_STANDARD_REQUIRED

  • CMAKE_CUDA_EXTENSIONS

它们的值用于在生成的项目中设置相应的目标属性（除非被明确的选项覆盖）。

版本 3.14 变更: 对于 Green Hills MULTI 生成器，GHS 工具集和目标系统定制缓存变量也会被传播到测试项目中。

版本 3.24 新增: 可以设置 CMAKE_TRY_COMPILE_NO_PLATFORM_VARIABLES 变量以禁止将平台变量传递到测试项目中。

版本 3.25 新增: 如果 CMP0141 设置为 NEW，可以使用 CMAKE_MSVC_DEBUG_INFORMATION_FORMAT 来指定 MSVC 调试信息格式。

版本 3.30 新增: 如果设置了 PROPAGATE_TOP_LEVEL_INCLUDES_TO_TRY_COMPILE 全局属性，则在使用 整个项目签名 时，CMAKE_PROJECT_TOP_LEVEL_INCLUDES 会被传播到测试项目的构建配置中。

版本 4.0 新增: 如果 CMP0184 设置为 NEW，可以使用 CMAKE_MSVC_RUNTIME_CHECKS 来指定启用的 MSVC 运行时检查。

版本 4.3 新增: 如果 CMP0210 设置为 NEW，CMAKE_<LANG>_LINK_FLAGS 和 CMAKE_<LANG>_LINK_FLAGS_<CONFIG> 会被传播到测试项目的构建配置中。
