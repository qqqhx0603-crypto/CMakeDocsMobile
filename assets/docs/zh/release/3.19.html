自 CMake 3.18 以来的变更包括：

新特性¶

预设 (Presets)¶

  • cmake(1) 和 cmake-gui(1) 现在可以识别 CMakePresets.json 和 CMakeUserPresets.json 文件（参见 cmake-presets(7)）。

生成器 (Generators)¶

  • Xcode 生成器现在在为 Xcode 12.0 或更高版本生成项目时，使用 Xcode 的“新构建系统”。参见 CMAKE_XCODE_BUILD_SYSTEM 变量。可以使用 -T buildsystem=1 切换回传统构建系统。

  • Xcode 生成器增加了通过 Link Binaries With Libraries（链接二进制文件与库）构建阶段来链接库和框架的支持，而不是总是直接嵌入链接器标志。此行为由新的 XCODE_LINK_BUILD_PHASE_MODE 目标属性控制，该属性由新的 CMAKE_XCODE_LINK_BUILD_PHASE_MODE 变量初始化。

  • 用于 VS 2015 及更高版本的 Visual Studio 生成器 增加了对 Visual Studio Android 工具的支持。现在可以将 CMAKE_SYSTEM_NAME 设置为 Android，从而为 Android 工具生成 .vcxproj 文件。

语言¶

  • CMake 现在支持 ISPC 作为一等语言，可以通过 project() 和 enable_language() 命令启用。ISPC 目前在 Linux、macOS 和 Windows 上通过使用 Intel ISPC 编译器的 Makefile 生成器 和 Ninja 生成器提供支持。

  • Clang 的 CUDA 语言支持现在包括：

    • 可分离编译（CUDA_SEPARABLE_COMPILATION），以及

    • 交叉编译时查找分散的工具包安装。

  • CUDA 语言支持现在可以在 QNX 上工作。

平台¶

  • 现在支持 Apple Silicon（自 CMake 3.19.2 起）

    • CMAKE_HOST_SYSTEM_PROCESSOR 是使用 uname -m 选择的。由于这可能因 CMake 自身的架构和调用进程树的架构而异，因此可以设置 CMAKE_APPLE_SILICON_PROCESSOR 变量或 CMAKE_APPLE_SILICON_PROCESSOR 环境变量来明确指定主机架构。

    • 如果未设置 CMAKE_OSX_ARCHITECTURES，CMake 会添加明确的标志来告诉编译器为 CMAKE_HOST_SYSTEM_PROCESSOR 构建，这样工具链就不必根据进程树的架构进行猜测。

基于文件的 API¶

  • cmake-file-api(7) 的 “codemodel” 第 2 版 version 字段已更新为 2.2。

  • cmake-file-api(7) 的 “codemodel” 第 2 版 “target” 对象在 compileGroups 对象中增加了一个新的 languageStandard 字段。

命令行¶

  • cmake(1) 命令行工具的 --install 模式增加了一个 --default-directory-permissions 选项。

  • cmake(1) 增加了一个 -E create_hardlink 命令行工具，可用于在文件之间创建硬链接。

GUI¶

  • CMake GUI 现在拥有一个环境变量编辑器。

命令¶

  • add_test() 命令现在（正式）支持在其创建的测试名称中使用空格和其他特殊字符。参见策略 CMP0110。

  • cmake_language() 命令增加了 DEFER 模式，用于调度在目录处理结束时执行的命令调用。

  • configure_file() 命令增加了 NO_SOURCE_PERMISSIONS 选项，用于禁止将输入文件的权限复制到输出文件。

  • execute_process() 命令增加了 COMMAND_ERROR_IS_FATAL 选项以指定致命错误。

  • file(ARCHIVE_CREATE) 命令增加了 COMPRESSION_LEVEL 选项以指定压缩级别。

  • file(CHMOD) 和 file(CHMOD_RECURSE) 子命令已添加，用于设置文件和目录的权限。

  • file(DOWNLOAD) 命令的 <file> 参数现在是可选的。如果未指定，则文件不会被保存。

  • file(GENERATE) 命令增加了一个新的 TARGET 关键字，以支持解析依赖于目标的生成器表达式。

  • file() 命令增加了一个新的 REAL_PATH 子命令，用于计算解析了符号链接后的路径。

  • find_package() 命令现在可以处理版本范围。

  • separate_arguments() 命令增加了一个新的 PROGRAM 选项。它允许将参数视为程序调用，如果能找到可执行文件，则会将可执行文件解析为完整路径。

  • set_property()、get_property() 和 get_directory_property() 命令的 DIRECTORY 选项现在接受二进制目录路径的引用，例如 CMAKE_CURRENT_BINARY_DIR 的值。

  • string() 命令增加了一组新的 JSON 子命令，提供了 JSON 解析功能。

变量¶

  • 添加了 CMAKE_CLANG_VFS_OVERLAY 变量，用于告知 Clang 在从区分大小写文件系统的主机进行交叉编译时，使用 VFS 覆盖层来支持 Windows SDK。

  • CMAKE_MFC_FLAG 变量现在支持生成器表达式。

  • 添加了 CMAKE_OPTIMIZE_DEPENDENCIES 变量，用于初始化新的 OPTIMIZE_DEPENDENCIES 目标属性，并避免为静态库不必要地构建依赖项。

  • 添加了 CMAKE_PCH_INSTANTIATE_TEMPLATES 变量，用于初始化新的 PCH_INSTANTIATE_TEMPLATES 目标属性。

  • 添加了 CMAKE_VS_WINDOWS_TARGET_PLATFORM_VERSION_MAXIMUM 变量，用于告知 Visual Studio 生成器 选择 Windows SDK 的最大版本。

属性¶

  • EXCLUDE_FROM_ALL 目标属性现在支持 生成器表达式。

  • OPTIMIZE_DEPENDENCIES 目标属性已添加，用于避免为静态库不必要地构建依赖项。

  • PCH_INSTANTIATE_TEMPLATES 目标属性已添加，用于在预编译头文件中启用模板实例化。这默认是启用的，并且可以显著提高编译时间。目前仅支持 Clang（版本 11 或更高）。

  • WIN32_EXECUTABLE 目标属性现在支持 生成器表达式。

模块¶

  • CheckCompilerFlag 模块已添加，用于将 CheckCCompilerFlag 和 CheckCXXCompilerFlag 推广到更多语言。它还支持 CUDA 和 ISPC 语言。

  • CheckLinkerFlag 模块现在支持 CUDA 语言。

  • CheckSourceCompiles 模块已添加，用于将 CheckCSourceCompiles 和 CheckCXXSourceCompiles 推广到更多语言。它还支持 CUDA 和 ISPC 语言。

  • CheckSourceRuns 模块已添加，用于将 CheckCSourceRuns 和 CheckCXXSourceRuns 推广到更多语言。它还支持 CUDA 语言。

  • CMakePackageConfigHelpers 模块增加了对版本范围的支持。

  • FindCUDAToolkit 模块增加了对查找不包含 nvcc 的 CUDA 工具包的支持，以及对交叉编译时查找分散的工具包安装的支持。

  • FindPackageHandleStandardArgs 模块学会了处理版本范围。它还增加了 find_package_check_version() 命令，用于根据 find_package() 命令的版本相关参数检查版本的有效性。

  • FindPython3、FindPython2 和 FindPython 模块获得了处理版本范围的能力。

  • FindPython3、FindPython2 和 FindPython 模块分别提供了变量 Python3_LINK_OPTIONS、Python2_LINK_OPTIONS 和 Python_LINK_OPTIONS 以用于链接选项。

  • FindSDL 模块现在提供：

    • 一个导入目标 SDL::SDL。

    • 结果变量 SDL_LIBRARIES 和 SDL_INCLUDE_DIRS。

    • 版本变量 SDL_VERSION、SDL_VERSION_MAJOR、SDL_VERSION_MINOR 和 SDL_VERSION_PATCH。

  • FindSWIG 模块获得了处理版本范围的能力。

  • FindTIFF 模块增加了一个 CXX 组件，用于查找包含 C++ 绑定的 tiffxx 库。

  • FindVulkan 模块现在提供了一个 Vulkan::glslc 导入目标和相关的 Vulkan_GLSLC_EXECUTABLE 变量，其中包含 GLSL SPIR-V 编译器的路径。

  • UseSWIG 模块增加了对新源文件属性 OUTPUT_DIR 和 OUTFILE_DIR 的支持，以便按源文件管理输出目录。

CTest¶

  • ctest(1) 现在支持 CUDA compute-sanitizer 检查器（以前称为 cuda-memcheck）作为 CTEST_MEMORYCHECK_COMMAND。可以通过向 CTEST_MEMORYCHECK_COMMAND_OPTIONS 变量添加适当的标志来选择 compute-sanitizer 支持的不同工具（memcheck、racecheck、synccheck 和 initcheck）。默认标志是 --tool memcheck --leak-check full。

CPack¶

  • CPack 增加了 CPACK_PRE_BUILD_SCRIPTS、CPACK_POST_BUILD_SCRIPTS 和 CPACK_PACKAGE_FILES 变量。

  • CPack External Generator 增加了 CPACK_EXTERNAL_BUILT_PACKAGES 变量。

  • CPack WIX Generator 增加了 CPACK_WIX_CUSTOM_XMLNS 选项，用于指定自定义 XML 命名空间。

其他¶

  • 接口库 (Interface Libraries) 现在可以通过 add_library() 或 target_sources() 添加源文件。那些带有源文件的库将作为构建系统的一部分进行生成。

废弃和已移除的特性¶

  • 与早于 2.8.12 的 CMake 版本兼容性现在已被废弃，并将在未来的版本中移除。将策略版本设置为旧值的 cmake_minimum_required() 或 cmake_policy() 调用现在会发出废弃警告。

  • 为策略 CMP0071（CMP0071 及以下版本已被废弃）添加了明确的废弃诊断。cmake-policies(7) 手册解释说，所有策略的旧 (OLD) 行为均已被废弃，项目应迁移到新 (NEW) 行为。

  • 不再支持早于 10.5 的 macOS SDK。

  • cmake-gui(1) 现在需要 Qt5。已移除对使用 Qt4 进行编译的支持。

  • cmake(1) 命令行选项 --warn-unused-vars 已被移除，现在被静默忽略。该选项自 CMake 3.3 起就无法正常工作。

文档¶

添加了以下指南：

  • IDE 集成指南

  • 导入和导出指南

其他变更¶

  • 为 macOS 构建现在将使用系统上可用的最新 SDK，除非用户已使用 CMAKE_OSX_SYSROOT 明确选择了 SDK。部署目标或系统 macOS 版本不会影响 SDK 的选择。

  • CMAKE_<LANG>_COMPILER 变量现在可用于存储像 CC 和其他环境变量那样的“强制”编译器标志。

  • CMAKE_<LANG>_FLAGS_INIT 变量现在会在编译器识别检查期间被考虑，前提是未设置 CMAKE_<LANG>_FLAGS 或 CFLAGS 等其他来源。

  • find_program() 命令现在需要执行权限，而不是读取文件的权限。参见策略 CMP0109。

  • 如果丢失了 location 属性的导入目标，在使用该 location 时会在生成过程中失败。参见策略 CMP0111。

  • 以下查询目录或文件名组件的基于目标的生成器表达式，不再添加对被评估目标的依赖。参见策略 CMP0112。

    • TARGET_FILE_DIR

    • TARGET_LINKER_FILE_BASE_NAME

    • TARGET_LINKER_FILE_NAME

    • TARGET_LINKER_FILE_DIR

    • TARGET_SONAME_FILE_NAME

    • TARGET_SONAME_FILE_DIR

    • TARGET_PDB_FILE_NAME

    • TARGET_PDB_FILE_DIR

    • TARGET_BUNDLE_DIR

    • TARGET_BUNDLE_CONTENT_DIR

  • Makefile 生成器 不再重复目标依赖项中的自定义命令。参见策略 CMP0113。

  • ExternalProject 模块对步骤目标依赖项的处理已修订。参见策略 CMP0114。

  • OSX_ARCHITECTURES 目标属性现在对于 ASM 语言也是尊重的。

  • 如果 CUDA 编译器检测在用户指定 CMAKE_CUDA_ARCHITECTURES 或 CMAKE_CUDA_HOST_COMPILER 的情况下失败，则会引发错误。

更新¶

自 CMake 3.19.0 以来的变更包括：

3.19.1¶

  • CMake 3.19.0 通过传递显式的语言标志（如 -x c）来编译带有 LANGUAGE 属性的源文件。这与该属性的文档含义一致，即源文件是用指定的语言编写的。然而，这可能会破坏那些仅使用该属性来确保使用指定语言编译器的项目。此更改已被撤销，以恢复 CMake 3.18 及以下版本的行为。

  • Clang 的 CUDA 11.1 支持。

3.19.2¶

  • cmake.org 上提供的预编译 macOS 二进制文件现在是具有 x86_64 和 arm64 架构的通用二进制文件。它需要 macOS 10.10 或更新版本。包文件命名模式已从 cmake-$ver-Darwin-x86_64 更改为 cmake-$ver-macos-universal。

  • 更新了 Apple Silicon 主机架构选择支持。CMake 3.19.0 和 3.19.1 总是选择 arm64 作为主机架构。CMake 3.19.2 回归到使用 uname -m，正如 CMake 3.18 及以下版本所做的那样。由于这可能因 CMake 自身的架构和调用进程树的架构而异，因此可以设置 CMAKE_APPLE_SILICON_PROCESSOR 变量或 CMAKE_APPLE_SILICON_PROCESSOR 环境变量来明确指定主机架构。

  • 添加了 CMAKE_ISPC_HEADER_SUFFIX 变量和相应的 ISPC_HEADER_SUFFIX 目标属性，用于控制 ISPC 编译器生成的头文件所使用的头后缀。

3.19.3¶

  • cmake.org 现在提供预编译的 Linux aarch64 二进制文件。

  • cmake.org 现在提供两个预编译的 macOS 二进制文件：

    • 命名模式 cmake-$ver-macos-universal 是一个具有 x86_64 和 arm64 架构的通用二进制文件。它需要 macOS 10.13 或更新版本。

    • 命名模式 cmake-$ver-macos10.10-universal 是一个具有 x86_64 和 arm64 架构的通用二进制文件。它需要 macOS 10.10 或更新版本。

3.19.4¶

  • 3.19.0 中引入的 CMAKE_VS_WINDOWS_TARGET_PLATFORM_VERSION_MAXIMUM 变量此前仅适用于 Visual Studio 14 2015 生成器。现在它已修复，也可适用于后续 VS 版本的 Visual Studio 生成器。

3.19.5¶

  • 当启用 IOS_INSTALL_COMBINED 并使用 Xcode 生成器时，现在可以通过在命令行运行 cmake --install 或 cpack 来发起安装或包创建。使用 Xcode 新构建系统时，由于 Xcode 的限制，这些是仅支持的方法。仅在使用传统构建系统时，才支持通过在 Xcode 中构建 install 或 package 目标来发起这些操作。

  • 作为支持 Xcode Link Binaries With Libraries 构建阶段的一部分，在 3.19.0 中引入的框架处理破坏了在不重新配置的情况下切换设备构建和模拟器构建的能力。该能力现已恢复。

3.19.6¶

  • cmake-presets(7) 特性不再允许在 CMakePresets.json 或 CMakeUserPresets.json 文件中使用注释。在 CMake 3.19.0 到 3.19.5 的实现中，这是被错误允许的，并且未被记录。

3.19.7¶

  • 对于 VS 2017 及更高版本的 Visual Studio 生成器，CMAKE_GENERATOR_TOOLSET 字段的 version= 现在接受三部分组成的 MSVC 工具集版本，例如 14.28.29910。参见 CMAKE_VS_PLATFORM_TOOLSET_VERSION 变量。
