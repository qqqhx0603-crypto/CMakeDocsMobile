自 CMake 3.0 以来的变更内容如下。

文档变更¶

  • 新增了 cmake-compile-features(7) 手册。

新功能¶

生成器¶

  • 新增了 Visual Studio 14 2015 生成器。

Windows Phone 和 Windows（微软桌面系统） Store¶

  • Visual Studio 11（微软集成开发环境）（微软集成开发环境） (2012) 及以上版本的生成器现已支持为 Windows（微软桌面系统） Phone 和 Windows（微软桌面系统） Store 生成项目。可以通过在 cmake(1) 命令行中或 CMAKE_TOOLCHAIN_FILE 中将 CMAKE_SYSTEM_NAME 变量设置为 WindowsPhone 或 WindowsStore 来激活这些平台。同时，将 CMAKE_SYSTEM_VERSION 设置为 8.0 或 8.1 以指定目标 Windows（微软桌面系统） 版本。

NVIDIA Nsight Tegra（NVIDIA Tegra 调试扩展）¶

  • Visual Studio 10（微软集成开发环境）（微软集成开发环境） (2010) 及以上版本的生成器现已支持为 NVIDIA Nsight Tegra（NVIDIA Tegra 调试扩展） Visual Studio（集成开发环境） Edition（NVIDIA Tegra 的 Visual Studio（集成开发环境） 扩展） 生成项目。可以通过在 cmake(1) 命令行中或 CMAKE_TOOLCHAIN_FILE 中将 CMAKE_SYSTEM_NAME 变量设置为 Android 来激活此平台。

语法¶

  • cmake-language(7) 中关于 变量引用 和 转义序列 的语法已被简化，以实现更快的处理速度。参见策略 CMP0053。

  • if() 命令不再自动对引号或括号参数中的变量进行解引用。参见策略 CMP0054。

命令¶

  • add_custom_command() 命令现在可以解析 DEPENDS 参数中的 cmake-generator-expressions(7)。

  • export(PACKAGE) 命令现在会检查 CMAKE_EXPORT_NO_PACKAGE_REGISTRY 变量以跳过导出包。

  • file(STRINGS) 命令新增了 ENCODING 选项，以支持提取 UTF-8 字符串。

  • find_package() 命令现在会检查 CMAKE_FIND_PACKAGE_NO_PACKAGE_REGISTRY 和 CMAKE_FIND_PACKAGE_NO_SYSTEM_PACKAGE_REGISTRY 变量以跳过搜索包注册表。

  • get_property() 命令新增了用于属性的 INSTALL 作用域。

  • install() 命令新增了 MESSAGE_NEVER 选项，以避免在安装过程中输出信息。

  • set_property() 命令新增了用于属性的 INSTALL 作用域。

  • string() 命令新增了 GENEX_STRIP 子命令，用于移除 生成器表达式。

  • string() 命令新增了 UUID 子命令，用于生成通用唯一识别码 (UUID)。

  • 新增了 target_compile_features() 命令，允许像填充其他构建变量一样填充 COMPILE_FEATURES 目标属性。

  • 新增了 target_sources() 命令，用于向 SOURCES 目标属性中添加内容。

变量¶

  • Visual Studio 8（微软集成开发环境）（微软集成开发环境） (2005) 及以上版本的生成器现在可以从新的 CMAKE_GENERATOR_PLATFORM 变量中读取目标平台名称（如果该名称未在生成器名称中指定）。平台名称可以在 cmake(1) 命令行中使用 -A 选项指定，例如 -G "Visual Studio 12 2013" -A x64。

  • CMAKE_GENERATOR_TOOLSET 变量现在可以在由 CMAKE_TOOLCHAIN_FILE 变量指定的工具链文件中进行初始化。这在结合 Xcode 或 Visual Studio（集成开发环境） 生成器进行交叉编译时非常有用。

  • 引入了 CMAKE_INSTALL_MESSAGE 变量，以选择性地减少安装输出信息。

属性¶

  • 新增了 CXX_STANDARD 和 CXX_EXTENSIONS 目标属性，可用于指定 CMake 计算所需编译选项（如 -std=c++11 或 -std=gnu++11）时使用的值。可以通过设置 CMAKE_CXX_STANDARD 和 CMAKE_CXX_EXTENSIONS 变量来初始化这些目标属性。

  • 新增了 C_STANDARD 和 C_EXTENSIONS 目标属性，可用于指定 CMake 计算所需编译选项（如 -std=c11 或 -std=gnu11）时使用的值。可以通过设置 CMAKE_C_STANDARD 和 CMAKE_C_EXTENSIONS 变量来初始化这些目标属性。

  • 新增的 COMPILE_FEATURES 目标属性可以包含构建目标所需的功能列表。CMake 利用此信息确保当前编译器能够构建该目标，并添加支持这些语言特性所需的必要编译标志。

  • 引入了新的 COMPILE_PDB_NAME 和 COMPILE_PDB_OUTPUT_DIRECTORY 目标属性，用于指定 MSVC 编译器程序数据库文件的位置（cl /Fd）。这补充了现有的用于指定链接器程序数据库文件位置（link /pdb）的 PDB_NAME 和 PDB_OUTPUT_DIRECTORY 目标属性。

  • INTERFACE_LINK_LIBRARIES 目标属性现在支持 $<LINK_ONLY:...> 生成器表达式。

  • 引入了新的 INTERFACE_SOURCES 目标属性。该属性由依赖目标使用，依赖目标会编译并链接所列出的源文件。

  • SOURCES 目标属性在配置时读取时，如果策略 CMP0051 设置为 NEW，则现在可以包含如 TARGET_OBJECTS 等 生成器表达式。

  • SOURCES 目标属性现在普遍支持 生成器表达式。这些生成器表达式可在 add_library() 和 add_executable() 命令中使用。

  • 现在可以写入并追加到 SOURCES 目标属性。可以使用 CMAKE_DEBUG_TARGET_PROPERTIES 变量来追踪源文件的来源。

  • 新增了 VS_DEPLOYMENT_CONTENT 源文件属性，用于告知 Visual Studio（集成开发环境） 生成器将内容标记为 Windows（微软桌面系统） Phone 和 Windows（微软桌面系统） Store 项目的部署内容。

  • 新增了 VS_DEPLOYMENT_LOCATION 源文件属性，用于告知 Visual Studio（集成开发环境） 生成器在 Windows（微软桌面系统） Phone 和 Windows（微软桌面系统） Store 项目中标记为部署的内容的相对位置。

  • 创建了 VS_WINRT_COMPONENT 目标属性，用于告知 Visual Studio（集成开发环境） 生成器将共享库编译为 Windows（微软桌面系统） Runtime (WinRT) 组件。

  • Xcode 生成器现在会检查源文件属性 XCODE_EXPLICIT_FILE_TYPE 和 XCODE_LAST_KNOWN_FILE_TYPE 以获取自定义 Xcode 文件引用类型。

模块¶

  • BundleUtilities 模块现在可以解析并替换 OS X 上的 @rpath 占位符，从而正确打包使用它们的应用程序。

  • CMakePackageConfigHelpers 模块的 configure_package_config_file() 命令新增了 INSTALL_PREFIX 选项，用于生成非 CMAKE_INSTALL_PREFIX 前缀路径的包配置文件。

  • 新增了 CheckFortranSourceCompiles 模块，提供 check_fortran_source_compiles() 命令。

  • ExternalData 模块现在会容忍对缺失源文件的 DATA{} 引用，并产生警告，而不是直接报错。这有助于开发者在测试参考输出尚未创建时，预先编写新的 DATA{} 引用。

  • ExternalProject 模块现已支持使用 .7z、.tar.xz 和 .txz 扩展名的 lzma 压缩源代码压缩包。

  • ExternalProject 模块的 ExternalProject_Add 命令新增了 BUILD_ALWAYS 选项，用于使外部项目构建步骤在宿主项目每次构建时都运行。

  • ExternalProject 模块的 ExternalProject_Add 命令新增了 EXCLUDE_FROM_ALL 选项，用于为外部项目目标设置 EXCLUDE_FROM_ALL 目标属性。

  • ExternalProject 模块的 ExternalProject_Add_Step 命令新增了 EXCLUDE_FROM_MAIN 选项，用于使该步骤不成为主外部项目目标的直接依赖。

  • ExternalProject 模块的 ExternalProject_Add 命令新增了 DOWNLOAD_NO_PROGRESS 选项，用于在下载源代码压缩包时禁用进度输出。

  • FeatureSummary 模块的 feature_summary API 现在接受多个 WHAT 选项的值，并进行适当的组合。

  • FindCUDA 模块现已支持 fatbin 和 cubin 模块。

  • FindGTest 模块的 gtest_add_tests 宏新增了 AUTO 选项，用于自动读取测试可执行文件的 SOURCES 目标属性并扫描源文件以添加测试。

  • FindGLEW 模块现在提供导入目标。

  • FindGLUT 模块现在提供导入目标。

  • FindHg 模块新增了 Hg_WC_INFO 命令，用于帮助运行 hg 命令以提取 Mercurial 工作副本的相关信息。

  • 引入了 FindOpenCL 模块。

  • FindOpenMP 模块现已支持 Fortran。

  • FindPkgConfig 模块现在会使用 PKG_CONFIG 环境变量的值作为 pkg-config 可执行文件（如果已设置）。

  • 引入了 FindXercesC 模块。

  • FindZLIB 模块现在提供导入目标。

  • GenerateExportHeader 模块的 generate_export_header 命令现在允许用于 对象库（Object Libraries）。

  • InstallRequiredSystemLibraries 模块新增了 CMAKE_INSTALL_OPENMP_LIBRARIES 选项，用于安装 MSVC OpenMP 运行时库。

  • UseSWIG 模块现在可以尽可能从 .i 源文件中检测模块名称，从而避免需要显式设置 SWIG_MODULE_NAME 源文件属性。

  • 新增了 WriteCompilerDetectionHeader 模块，允许创建用于编译器可选功能检测的可移植头文件。

生成器表达式¶

  • 新增的 COMPILE_FEATURES 生成器表达式 允许根据可用的编译器特性来设置构建属性。

CTest¶

  • ctest_coverage() 命令现在会读取变量 CTEST_COVERAGE_EXTRA_FLAGS 以设置 CoverageExtraFlags。

  • ctest_coverage() 命令现在支持通过 codecov 工具处理 Intel 覆盖率文件。

  • ctest_memcheck() 命令现已支持 sanitizer 模式，包括 AddressSanitizer、MemorySanitizer、ThreadSanitizer 和 UndefinedBehaviorSanitizer。选项可以使用新的 CTEST_MEMORYCHECK_SANITIZER_OPTIONS 变量进行设置。

CPack¶

  • cpack(1) 新增了 IFW 生成器，用于使用 Qt Framework Installer 工具进行打包。参见 CPack IFW 生成器。

  • cpack(1) 新增了 7Z 和 TXZ 生成器，支持 lzma 压缩存档。

  • CPack DEB 生成器 新增了 CPACK_DEBIAN_COMPRESSION_TYPE 变量，用于设置 tarball 的压缩类型。

  • CPack WIX 生成器 现已支持 CPACK_WIX_ACL 安装文件属性，用于指定访问控制列表 (ACL)。

其他¶

  • cmake(1) 的 -E 选项新增了 env 命令。

  • cmake(1) 的 -E tar 命令现已支持 lzma 压缩文件。

  • 对象库（Object Libraries） 现在可以包含不编译为目标文件的额外源文件，只要它们不影响常规库的链接即可（例如 .dat 文件没问题，但 .def 文件不行）。

  • VS 8 及更高版本的 Visual Studio（集成开发环境） 生成器现已支持 ASM_MASM 语言。

  • Visual Studio（集成开发环境） 生成器现在会将 .hlsl 源文件视为高级着色器语言 (HLSL) 源文件（在 .vcxproj 文件中使用 FXCompile）。新增了源文件属性 VS_SHADER_TYPE、VS_SHADER_MODEL 和 VS_SHADER_ENTRYPOINT，用于指定着色器类型、模型和入口点名称。

新的诊断信息¶

  • 引入了策略 CMP0052，用于控制导出目标中 INTERFACE_INCLUDE_DIRECTORIES 的目录。

已弃用和移除的功能¶

  • 在 CMake 3.0 中，target_link_libraries() 命令意外地开始允许非引号参数使用其中包含列表（由 ; 分隔）的 生成器表达式。例如

    set(libs B C)
target_link_libraries(A PUBLIC $<BUILD_INTERFACE:${libs}>)

    代码说明

    • set(libs B C)：调用 set() 命令，把变量 libs 设为给定的值；变量说明：libs：当前示例使用的变量，其值由前面的命令、调用参数或运行环境提供。

    • target_link_libraries(A PUBLIC $<BUILD_INTERFACE:${libs}>)：调用 target_link_libraries() 命令，为目标链接依赖库；变量说明：libs：当前示例使用的变量，其值由前面的命令、调用参数或运行环境提供；生成器表达式：$<BUILD_INTERFACE:...> 在生成构建系统时按当前目标和配置计算结果。

    这等同于编写

    target_link_libraries(A PUBLIC $<BUILD_INTERFACE:B C>)

    代码说明

    • target_link_libraries(A PUBLIC $<BUILD_INTERFACE:B C>)：调用 target_link_libraries() 命令，为目标链接依赖库；生成器表达式：$<BUILD_INTERFACE:...> 在生成构建系统时按当前目标和配置计算结果。

    这从来都不是预期的工作方式。在 CMake 2.8.12 中它并不起作用。此类生成器表达式应放在带引号的参数中

    set(libs B C)
target_link_libraries(A PUBLIC "$<BUILD_INTERFACE:${libs}>")

    代码说明

    • set(libs B C)：调用 set() 命令，把变量 libs 设为给定的值；变量说明：libs：当前示例使用的变量，其值由前面的命令、调用参数或运行环境提供。

    • target_link_libraries(A PUBLIC "$<BUILD_INTERFACE:${libs}>")：调用 target_link_libraries() 命令，为目标链接依赖库；变量说明：libs：当前示例使用的变量，其值由前面的命令、调用参数或运行环境提供；生成器表达式：$<BUILD_INTERFACE:...> 在生成构建系统时按当前目标和配置计算结果。

    CMake 3.1 再次要求使用引号以确保其正确工作。

  • 在 CMake 3.1 之前，Makefile（Make 构建脚本） 生成器在生成的 makefile 中不能正确转义 make 变量赋值中的 #，导致它们被视为注释。这使得类似

    add_compile_options(-Wno-#pragma-messages)

    代码说明

    • add_compile_options(-Wno-#pragma-messages)：调用 add_compile_options() 命令，为当前目录或目标追加编译选项。

    的代码在 Makefile（Make 构建脚本） 生成器中无法工作，但在其他生成器中可以。现在它已被正确转义，使得不同生成器之间的行为保持一致。然而，一些项目可能尝试使用类似

    set(CMAKE_C_FLAGS "${CMAKE_C_FLAGS} -Wno-\\#pragma-messages")

    代码说明

    • set(CMAKE_C_FLAGS "${CMAKE_C_FLAGS} -Wno-\\#pragma-messages")：调用 set() 命令，把变量 CMAKE_C_FLAGS 设为给定的值；变量说明：CMAKE_C_FLAGS：CMake 选项相关变量，其精确取值和作用域以当前页面或脚本上下文为准。

    的代码来绕过原始错误。这为 Makefile（Make 构建脚本） 生成器添加了必要的转义，但也导致其他生成器将 -Wno-\#pragma-messages 传递给 shell（命令解释器），这仅在 POSIX shell（命令解释器） 中有效。不幸的是，这个转义修复无法以兼容的方式进行，因此这种特定于平台和生成器的变通方法不再有效。项目代码可以通过测试 CMAKE_VERSION 变量值来使变通方法也具有版本针对性。

  • 当设置了未文档化的 CMAKE_ALLOW_UNKNOWN_VARIABLE_READ_ACCESS 变量时，由 variable_watch() 命令建立的回调函数将不再接收 ALLOWED_UNKNOWN_READ_ACCESS 访问类型。未初始化的变量访问将始终报告为 UNKNOWN_READ_ACCESS。

  • CMakeDetermineVSServicePack 模块现在会发出已弃用的警告，不应再使用。请改用 CMAKE_<LANG>_COMPILER_VERSION 变量。

  • FindITK 模块已被完全移除。它只是 find_package(ITK ... NO_MODULE) 的一个简单封装。当未找到 ITK 时，这会产生更清晰的错误消息。

  • FindVTK 模块已被完全移除。它只是 find_package(VTK ... NO_MODULE) 的一个简单封装。当未找到 VTK 时，这会产生更清晰的错误消息。

    该模块还提供了对查找 VTK 4.0 的兼容性支持。此功能已被放弃。

其他更改¶

  • cmake-gui(1) 现在可以捕获由 execute_process() 命令启动的子进程输出，并将其显示在输出窗口中。

  • cmake-language(7) 中生成器表达式和列表扩展解析器的内部实现已得到优化，在大型项目上表现出显著的速度提升。

  • Makefile（Make 构建脚本） 生成器现在可以在 Windows（微软桌面系统） 上结合 GNU 工具使用响应文件，以便将库目录和名称传递给链接器。

  • 在生成链接器命令行时，CMake 现在避免重复与共享库 (SHARED library) 目标对应的项。

  • 对 Open Watcom 编译器的支持进行了全面重构。CMAKE_<LANG>_COMPILER_ID 现在为 OpenWatcom，且 CMAKE_<LANG>_COMPILER_VERSION 现在使用 Open Watcom 的外部版本编号。外部版本号比内部版本号小 11。

  • Emacs 的 cmake-mode.el 主要编辑模式不再将 _ 视为单词的一部分，使其与其他主要模式保持一致。
