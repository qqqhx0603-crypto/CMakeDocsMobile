自 CMake 2.8.12 以来的变更包括以下内容。

文档变更¶

  • CMake 文档已转换为 reStructuredText 格式，现在通过 Sphinx (https://sphinx-doc.cn) 转换为 man 和 html 页面。这使得文档能够被正确索引并包含交叉引用。

    从旧的内部文档格式的转换是通过自动化过程完成的，因此某些文档可能仍包含伪影。它们将随时间逐步更新。

    已实现一个基础的 reStructuredText 处理器，以支持 cmake --help-command 及类似的命令行选项。

  • 新增了手册

    • cmake-buildsystem(7)

    • cmake-commands(7)，替换了 cmakecommands(1) 和 cmakecompat(1)

    • cmake-developer(7)

    • cmake-generator-expressions(7)

    • cmake-generators(7)

    • cmake-language(7)

    • cmake-modules(7)，替换了 cmakemodules(1)

    • cmake-packages(7)

    • cmake-policies(7)，替换了 cmakepolicies(1)

    • cmake-properties(7)，替换了 cmakeprops(1)

    • cmake-qt(7)

    • cmake-toolchains(7)

    • cmake-variables(7)，替换了 cmakevars(1)

  • CMake 3.0.0 及以上版本的发行说明现在将包含在 html 文档中。

新功能¶

语法¶

  • CMake 语言已通过 方括号参数 (Bracket Argument) 和 方括号注释 (Bracket Comment) 语法进行了扩展，这些语法灵感源自 Lua 的长括号语法。

    set(x [===[bracket argument]===] #[[bracket comment]])

    等长开括号和闭括号之间的内容将按字面解释，不进行变量替换。

    警告

    此语法变更无法以完全兼容的方式实现。无法使用策略，因为语法解析发生在设置策略的机会之前。现有的使用以开括号开头的无引号参数的代码将会在没有任何诊断信息的情况下被以不同方式解析。幸运的是，该语法非常生僻，实际中几乎不会遇到这个问题。

生成器¶

  • 新增了一个 CodeLite 额外生成器，可供 Makefile 或 Ninja 生成器使用。

  • 新增了一个 Kate 额外生成器，可供 Makefile 或 Ninja 生成器使用。

  • Ninja 生成器学会了使用 ninja 作业池，通过新的 JOB_POOLS 全局属性指定。

命令¶

  • add_library() 命令新增了一个 INTERFACE 库类型。接口库没有构建规则，但可以拥有定义 使用需求 (usage requirements) 的属性，并且可以被安装、导出和导入。这对于创建仅包含头文件的库非常有用，这些库对其他库具有具体的链接依赖关系。

  • export() 命令新增了一种 EXPORT 模式，用于从由 install(TARGETS) 命令的 EXPORT 选项配置的导出集中检索目标列表。这使得从构建树导出与从安装树导出的相同目标变得简单。

  • export() 命令学会了与多个相关的导出集协同工作，从而允许从单个树中构建和导出多个包。该功能要求 CMake 等到生成步骤才能写入输出文件。这意味着不应在稍后的项目配置期间 include() 生成的目标文件，因为它将不可用。请改用 别名目标 (Alias Targets)。参见策略 CMP0024。

  • install(FILES) 命令学会了在文件列表中支持 生成器表达式 (generator expressions)。

  • project() 命令学会了将某些版本变量设置为由新的 VERSION 选项指定的值，或设置为空字符串。参见策略 CMP0048。

  • string() 命令新增了一个 CONCAT 模式。它与新的 方括号参数 语法结合使用特别有用。

  • unset() 命令新增了一个 PARENT_SCOPE 选项，与 set() 命令的选项匹配。

  • include_external_msproject() 命令学会了处理非 C++ 项目，如 .vbproj 或 .csproj。

  • ctest_update() 命令学会了更新由 Perforce (p4) 版本控制工具管理的工作树。

  • message() 命令新增了一个 DEPRECATION 模式。默认情况下不会发出此类消息，但如果启用了 CMAKE_WARN_DEPRECATED，它们可以作为警告发出；如果启用了 CMAKE_ERROR_DEPRECATED，则作为错误发出。

  • target_link_libraries() 命令现在允许重复使用 LINK_PUBLIC 和 LINK_PRIVATE 关键字。

变量¶

  • 引入了变量 CMAKE_FIND_NO_INSTALL_PREFIX，以告知 CMake 默认不要将 CMAKE_INSTALL_PREFIX 的值添加到 CMAKE_SYSTEM_PREFIX_PATH 变量中。这在构建一个安装了其自身某些依赖项的项目时很有用，以避免找到它即将替换的文件。

  • 引入了变量 CMAKE_STAGING_PREFIX，用于在交叉编译时指定宿主机系统上的安装前缀，该前缀与目标系统的 CMAKE_INSTALL_PREFIX 值不同。

  • 引入了变量 CMAKE_SYSROOT，用于指定工具链 SDK 的安装前缀，通常用于交叉编译。它用于向编译器传递 --sysroot 选项，并作为 find_* 命令搜索的前缀。

  • 引入了变量 CMAKE_<LANG>_COMPILER_TARGET，用于在交叉编译时，通过 CMAKE_TOOLCHAIN_FILE 变量指定的 工具链文件 中指定目标平台。这用于向某些交叉编译器驱动程序传递类似 --target=<triple> 的选项。

  • 引入了变量 CMAKE_MAP_IMPORTED_CONFIG_<CONFIG>，以可选地初始化 MAP_IMPORTED_CONFIG_<CONFIG> 目标属性。

属性¶

  • ADDITIONAL_MAKE_CLEAN_FILES 目录属性学会了支持 生成器表达式。

  • 引入了一个新的目录属性 CMAKE_CONFIGURE_DEPENDS，允许项目指定配置过程所依赖的额外文件。当这些文件中的任何一个被修改时，CMake 将在构建时重新运行。此前，这只能通过将此类文件指定为 configure_file() 命令的输入来实现。

  • 新增了一个 AUTORCC 功能，允许将 .qrc 文件列为目标源文件，从而无需调用 qt4_add_resources()。

  • 新增了一个 AUTOUIC 功能，取代了对 qt4_wrap_ui() 的调用需求。

  • 测试属性学会了支持 生成器表达式。这对于为测试属性（如 REQUIRED_FILES 和 WORKING_DIRECTORY）指定每个配置的值非常有用。

  • 引入了一个新的测试属性 SKIP_RETURN_CODE，以告知 ctest(1) 将特定的测试返回代码视为未运行该测试。这对于测试驱动程序报告某些测试需求不可用非常有用。

  • 引入了新型的 兼容接口属性 (Compatible Interface Properties)，即 COMPATIBLE_INTERFACE_NUMBER_MAX 和 COMPATIBLE_INTERFACE_NUMBER_MIN，分别用于计算数值的最大值和最小值。

模块¶

  • CheckTypeSize 模块的 check_type_size() 命令和 CheckStructHasMember 模块的 check_struct_has_member() 命令新增了一个 LANGUAGE 选项，用于可选地检查 C++ 类型。

  • ExternalData 模块学会了在有本地存储可用时，无需 URL 模板即可工作。

  • ExternalProject 函数 ExternalProject_Add 新增了一个 GIT_SUBMODULES 选项，用于指定检出可用子模块的子集。

  • 新增了一个 FindBacktrace 模块，以支持 find_package(Backtrace) 调用。

  • 新增了一个 FindLua 模块，以支持 find_package(Lua) 调用。

  • FindBoost 模块新增了一个 Boost_NAMESPACE 选项，用于更改库名称上的 boost 前缀。

  • FindBoost 模块学会了使用新的 Boost_USE_DEBUG_RUNTIME 选项来控制是否搜索带有 g 标记的库（针对 MS 调试运行时）。默认为 ON 以保留现有行为。

  • FindJava 和 FindJNI 模块学会了使用 JAVA_HOME CMake 变量或环境变量，并随后在 OS X 上尝试使用 /usr/libexec/java_home。

  • UseJava 模块的 add_jar 函数新增了一个 MANIFEST 选项，用于向 jar 传递 -m 选项。

  • 引入了一个新的 CMakeFindDependencyMacro 模块，包含一个 find_dependency 宏，用于在 包配置文件 中查找传递依赖项。此类依赖项将被 FeatureSummary 模块的列表所省略。

  • FindQt4 模块学会了为 Qt 可执行文件创建 导入目标 (Imported Targets)。这有助于在使用相同的 Qt 版本 构建系统时消除歧义。

  • FindRuby 模块学会了搜索 Ruby 2.0 和 2.1。

生成器表达式¶

  • 新增了 $<PLATFORM_ID> 和 $<PLATFORM_ID:...> 生成器表达式。

  • $<CONFIG> 生成器表达式 现在有一种不带参数的变体。这等同于 $<CONFIGURATION> 表达式。

  • 新增了 $<UPPER_CASE:...> 和 $<LOWER_CASE:...> 生成器表达式。

  • 新增了一个 $<MAKE_C_IDENTIFIER:...> 生成器表达式。

其他 (Other)¶

  • cmake(1) 的 -E 选项新增了一个 sleep 命令。

  • ccmake(1) 对话框学会了遵循 STRINGS 缓存条目属性，以循环遍历可能的枚举值列表。

  • cmake-gui(1) 对话框学会了在会话间记住窗口设置。

  • cmake-gui(1) 对话框学会了记住缓存条目的类型，以便在 添加条目 (Add Entry) 对话框中进行自动补全。

新诊断信息¶

  • 在被 生成器表达式 有条件链接的导入目标中，其 INTERFACE_INCLUDE_DIRECTORIES 目标属性中命名的目录此前未检查是否存在。现在会检查了。参见策略 CMP0027。

  • 构建目标名称现在必须符合有效性模式，且不得与 CMake 定义的目标冲突。参见策略 CMP0037。

  • 将自身指定为链接依赖项的构建目标此前会被静默接受，但现在会进行诊断。参见 CMP0038。

  • target_link_libraries() 命令过去会静默忽略将由 add_custom_target() 创建的构建目标作为其第一个参数的调用，但现在会对该错误进行诊断。参见策略 CMP0039。

  • add_custom_command() 命令过去会静默忽略指定带有不存在目标的 TARGET 选项的调用，但现在会对该错误进行诊断。参见策略 CMP0040。

  • INTERFACE_INCLUDE_DIRECTORIES 目标属性中的相对路径如果包含 生成器表达式，过去会被静默接受，但现在会被拒绝。参见策略 CMP0041。

  • get_target_property() 命令学会了拒绝指定不存在目标的调用。参见策略 CMP0045。

  • add_dependencies() 命令学会了拒绝指定依赖于不存在目标的调用。参见策略 CMP0046。

  • 链接依赖分析学会了假设包含 :: 的名称是指 别名目标 或 导入目标。如果缺少此类链接目标，现在将产生错误。在此前的情况下，CMake 会生成一个在构建时失败的链接行。参见策略 CMP0028。

  • 当 project() 或 enable_language() 命令初始化对某种语言的支持时，如果找不到编译器的完整路径并将其存储在相应的 CMAKE_<LANG>_COMPILER 变量中，现在会报错。这会提前产生更好的错误消息，并在已知没有可用编译器时停止处理。

  • 通过 add_library() 或 add_executable() 命令指定的目标源文件现在会拒绝那些需要未记录的额外变量扩展层的项。参见策略 CMP0049。

  • 使用 add_custom_command() 的未记录 SOURCE 签名现在会导致错误。参见策略 CMP0050。

已弃用和移除的功能¶

  • 已删除了支持为 CMake 2.4 之前版本编写的代码的兼容性选项。

  • 一些长期过时的命令，即不应再被调用的命令，已通过策略在代码中被禁止：

    • 策略 CMP0029 禁止 subdir_depends()

    • 策略 CMP0030 禁止 use_mangled_mesa()

    • 策略 CMP0031 禁止 load_command()

    • 策略 CMP0032 禁止 output_required_files()

    • 策略 CMP0033 禁止 export_library_dependencies()

    • 策略 CMP0034 禁止 utility_source()

    • 策略 CMP0035 禁止 variable_requires()

    • 策略 CMP0036 禁止 build_name()

  • cmake(1) 的 -i 向导模式已被删除。请改用交互式对话框，如 ccmake(1)，或使用 -D 选项从命令行设置缓存值。

  • 支持诸如 --help-man 和 --help-html 等命令行选项的内置文档格式化程序已被删除，取而代之的是上述新的文档系统。这些以及其他过去用于生成 man 和 html 格式页面的命令行选项不再起作用。cmake(1) 的 --help-custom-modules 选项现在会在运行时发出警告，并生成一个报告该限制的最小化文档。

  • COMPILE_DEFINITIONS_<CONFIG> 目录属性和 COMPILE_DEFINITIONS_<CONFIG> 目标属性已被弃用。请改用相应的 COMPILE_DEFINITIONS 目录属性或 COMPILE_DEFINITIONS 目标属性，并使用 生成器表达式（如 $<CONFIG:...>）来指定每个配置的定义。参见策略 CMP0043。

  • LOCATION 目标属性不应再从非 IMPORTED 目标读取。在多配置生成器中，这毫无意义，因为在配置项目时不知道构建配置。它已被 $<TARGET_FILE> 生成器表达式所取代。参见策略 CMP0026。

  • COMPILE_FLAGS 目标属性现在被记录为已弃用，尽管不会发出警告。请改为使用 COMPILE_OPTIONS 目标属性或 target_compile_options() 命令。

  • GenerateExportHeader 模块的 add_compiler_export_flags 命令现已弃用。它已被 <LANG>_VISIBILITY_PRESET 和 VISIBILITY_INLINES_HIDDEN 目标属性所取代。

其他更改¶

  • 版本方案已更改为仅使用两个组件来表示功能级别，而不是三个。第三个组件现在将用于错误修复版本或开发版本的日期。详细信息请参阅 CMAKE_VERSION 变量文档。

  • CMake 本身在 Windows 和 OS X 上的默认安装位置不再包含 CMake 版本号。这允许轻松替换，而无需手动重新生成本地构建树。

  • Visual Studio 10 (2010) 及更高版本的生成器已被重命名，以包含产品年份，就像旧版 VS 的生成器一样：

    • Visual Studio 10 -> Visual Studio 10 2010

    • Visual Studio 11 -> Visual Studio 11 2012

    • Visual Studio 12 -> Visual Studio 12 2013

    这明确了每个生成器对应哪个 Visual Studio 版本。旧名称为兼容性起见予以保留。

  • Apple 提供的 Clang 的 CMAKE_<LANG>_COMPILER_ID 值现在是 AppleClang。它必须与上游 Clang 区分开，因为版本号不同。参见策略 CMP0025。

  • QNX 上 qcc 的 CMAKE_<LANG>_COMPILER_ID 值现在是 QCC。它必须与 GNU 区分开，因为命令行选项不同。参见策略 CMP0047。

  • 在 64 位 OS X 上，CMAKE_HOST_SYSTEM_PROCESSOR 值现在被正确检测为 x86_64 而不是 i386。

  • 在 OS X 上，CMake 学会了默认启用 MACOSX_RPATH 目标属性指定的行为。这激活了对运行时共享库搜索使用 @rpath。参见策略 CMP0042。

  • build_command() 命令现在返回 cmake(1) --build 命令行，而不是直接调用本机构建工具。使用 Visual Studio 生成器时，CMake 和 CTest 不再需要预先定位 CMAKE_MAKE_PROGRAM。对适当 msbuild 或 devenv 工具的选择现在会在解决方案 (.sln) 文件可用时尽可能晚地执行，以便它可以依赖于项目内容。

  • cmake(1) --build 命令现在默认与本机构建工具共享其自身的 stdout 和 stderr 管道。曾经激活此功能的 --use-stderr 选项现在被忽略。

  • $<C_COMPILER_ID:...> 和 $<CXX_COMPILER_ID:...> 生成器表达式 过去执行不区分大小写的比较，但现在已更正为执行区分大小写的比较。参见策略 CMP0044。

  • 内置的 edit_cache 目标在没有交互式终端可用时（例如使用 Ninja 或 IDE 生成器时）将不再默认选择 ccmake(1)。相反，如果可用，将首选 cmake-gui(1)。

  • ExternalProject 下载步骤学会了在某些情况下重试下载，以增强对临时网络故障的鲁棒性。

  • FeatureSummary 不再列出传递依赖项，因为它们不是由当前项目直接请求的。

  • cmake-mode.el 主要 Emacs 编辑模式已在多个方面进行了清理和增强。

  • 在 导入目标 的 INTERFACE_INCLUDE_DIRECTORIES 中指定的包含目录在作为 使用需求 处理时，默认被视为 SYSTEM 包含目录。
