自 CMake 3.14 以来的变更包括以下内容。

新功能¶

生成器¶

  • Xcode 生成器现在支持每个目标（per-target）的方案（schemes）。请参阅 CMAKE_XCODE_GENERATE_SCHEME 变量和 XCODE_GENERATE_SCHEME 目标属性。

  • Green Hills MULTI 生成器已更新

    • 它现在支持 add_custom_command() 和 add_custom_target() 命令。

    • 它现在可在 Linux 上使用。

语言¶

  • Ninja 生成器增加了对 Swift 语言的初步支持。

    • 使用 SWIFTC 环境变量来指定编译器。

    • 添加了 Swift_DEPENDENCIES_FILE 目标属性和 Swift_DEPENDENCIES_FILE 源文件属性，用于自定义依赖文件。

    • 添加了 Swift_MODULE_NAME 目标属性以自定义 Swift 模块名称。

    • 添加了 Swift_DIAGNOSTICS_FILE 源属性，用于指示序列化 Swift 诊断信息的写入位置。

    Swift 支持目前处于实验阶段，不被视为稳定，未来 CMake 版本中可能会发生变化。

编译器¶

  • 现在支持 Windows 上针对 MSVC ABI 但具有 GNU 风格命令行界面的 Clang 编译器变体。

  • 增加了对基于 Clang 的 ARM 编译器的支持，编译器 ID 为 ARMClang。

  • 增加了对 IAR 编译器架构 Renesas RX、RL78、RH850 和 Texas Instruments MSP430 的支持。

  • 增加了对为 Linux 构建的 IAR 编译器（IAR BuildLx）的支持。

命令行¶

  • 添加了 CMAKE_GENERATOR 环境变量，用于在不使用 -G 选项运行 cmake(1) 时指定默认生成器。此外，还创建了 CMAKE_GENERATOR_PLATFORM、CMAKE_GENERATOR_TOOLSET 和 CMAKE_GENERATOR_INSTANCE 环境变量来配置生成器。

  • cmake(1) --build 工具的 --target 参数增加了对多个目标的支持，例如 cmake --build . --target Library1 Library2。现在它还有一个简短形式的 -t 别名，例如 cmake --build . -t Library1 Library2。

  • cmake(1) 命令增加了一个新的 --install 选项。这可以在构建项目后使用，无需使用生成的构建系统或原生构建工具即可运行安装。

  • cmake(1) 命令学习了一个新的 CLI 选项 --loglevel。

  • cmake(1) -E remove_directory 命令行工具增加了对删除多个目录的支持。

  • cmake(1) -E tar 工具已得到改进：

    • 它现在即使在某些文件不可读的情况下，也会继续向归档文件中添加文件。这种行为与经典的 tar 工具更加一致。

    • 它现在解析所有标志，如果提供了无效标志，则会发出警告。

    • 如果未指定操作标志，它现在会显示错误，并附带可能的操作列表：t（列出）、c（创建）或 x（提取）。

    • 它现在支持仅提取（-x）或列出（-t）特定文件或目录。

    • 它现在支持通过 --zstd 选项使用 Zstandard 压缩。Zstandard 的设计旨在提供与 DEFLATE (zip) 算法相当的压缩比，但速度更快，特别是在解压方面。

命令¶

  • add_custom_command() 和 add_custom_target() 命令增加了一个新的 JOB_POOL 选项，该选项可与 Ninja 生成器一起工作，在构建声明中设置 pool 变量。

  • add_library() 命令的 ALIAS 选项现在支持 UNKNOWN 类型的导入库。

  • cmake_parse_arguments() 命令增加了一个额外的 <prefix>_KEYWORDS_MISSING_VALUES 输出变量，用于报告调用者提供的没有值的关键字参数。

  • execute_process() 命令增加了一个 COMMAND_ECHO 选项，并支持 CMAKE_EXECUTE_PROCESS_COMMAND_ECHO 变量，以实现在执行前回显命令行字符串。

  • file(INSTALL) 命令学习了一个新参数 FOLLOW_SYMLINK_CHAIN，可用于递归解析并安装符号链接。

  • list() 学习了新的子命令：PREPEND、POP_FRONT 和 POP_BACK。

  • message() 命令学习了新的类型：NOTICE、VERBOSE、DEBUG 和 TRACE。

  • string() 学习了一个新的子命令 REPEAT。

变量¶

  • CMAKE_CROSSCOMPILING_EMULATOR 变量和相应的 CROSSCOMPILING_EMULATOR 目标属性现在支持向模拟器传递参数。

  • 添加了 CMAKE_FIND_PACKAGE_PREFER_CONFIG 变量，用于告知 find_package() 调用即使在有查找模块可用时，也优先查找包配置文件。

  • 添加了 CMAKE_FRAMEWORK 变量，用于初始化所有目标上的 FRAMEWORK 属性。

  • 添加了 CMAKE_VS_JUST_MY_CODE_DEBUGGING 变量和 VS_JUST_MY_CODE_DEBUGGING 目标属性，以在编译时使用 MSVC cl 19.05 及更高版本时启用 Visual Studio 调试器的“仅我的代码”（Just My Code）功能。

  • 引入了 CMAKE_MSVC_RUNTIME_LIBRARY 变量和 MSVC_RUNTIME_LIBRARY 目标属性，用于选择针对 MSVC ABI 的编译器所使用的运行时库。请参阅策略 CMP0091。

  • 添加了 CMAKE_PROJECT_INCLUDE 和 CMAKE_PROJECT_INCLUDE_BEFORE 变量，允许在 project() 调用处注入自定义代码，而无需预先知道项目名称。

属性¶

  • 添加了 ADDITIONAL_CLEAN_FILES 目标属性和 ADDITIONAL_CLEAN_FILES 目录属性。它们允许注册在清理阶段应该被删除的其他文件。

  • PUBLIC_HEADER 和 PRIVATE_HEADER 属性现在可以在 接口库（Interface Libraries） 上设置。可以通过传递 PUBLIC_HEADER 和 PRIVATE_HEADER 参数，使用 install(TARGETS) 命令安装这些属性指定的头文件。

  • 添加了 VS_PACKAGE_REFERENCES 目标属性，用于告知 Visual Studio 生成器 添加对 nuget 包的引用。

  • 添加了 VS_PROJECT_IMPORT 目标属性，以允许托管的 Visual Studio 项目文件导入外部 .props 文件。

  • 添加了 VS_NO_SOLUTION_DEPLOY 目标属性，用于告知 Visual Studio 生成器 是否将构件部署到 WinCE 或 Windows Phone 目标设备。

模块¶

  • FindBoost 模块经过重构，旨在为其“配置”模式和“模块”模式以及其他查找模块提供更一致的用户体验。

    • 现在定义了一个新的导入目标 Boost::headers（与 Boost::boost 相同）。

    • 添加了新的输出变量 Boost_VERSION_MACRO、Boost_VERSION_MAJOR、Boost_VERSION_MINOR、Boost_VERSION_PATCH 和 Boost_VERSION_COUNT。

    • 传递给 find_package() 的 QUIET 参数在配置模式下不再被忽略。请注意，Boost 1.70.0 附带的 CMake 包忽略了传递给 find_package() 的 QUIET 参数。此问题已在下一个 Boost 版本中修复。

    • 删除了输入开关 Boost_DETAILED_FAILURE_MSG。

    • Boost_VERSION 在模块模式下现在以 x.y.z 格式报告版本。请参阅策略 CMP0093。

  • FindCups 模块现在提供导入目标。

  • 添加了 FindEnvModules 模块，用于在 CTest 脚本 中使用基于 Lua 和 TCL 的环境模块。

  • FindGLEW 模块现在提供了一个与上游 GLEW 在其自己的 CMake 包文件中所提供的更加一致的接口。

  • FindPkgConfig 现在使用其他（非库）链接器标志填充导入目标的 INTERFACE_LINK_OPTIONS 属性。

  • FindPostgreSQL 模块学会了分别查找调试版和发布版变体。

  • FindPython3、FindPython2 和 FindPython 模块增加了额外的查找策略、控件以及一个新的默认值。请参阅策略 CMP0094。

  • FindPython、FindPython2 和 FindPython3 模块获得了一个新目标（分别为 Python::Module、Python2::Module 和 Python3::Module），可用于开发 Python 模块。

  • FindPython3、FindPython2 和 FindPython 模块获得了控制虚拟环境处理方式的能力。

  • UseSWIG 模块学会了通过向 SWIG 编译器传递 python 语言的 -interface <library_name> 或 CSharp 语言的 -dllimport <library_name> 来管理备用库名称。

生成器表达式¶

  • 生成器表达式 $<C_COMPILER_ID>、$<CXX_COMPILER_ID>、$<CUDA_COMPILER_ID>、$<Fortran_COMPILER_ID>、$<COMPILE_LANGUAGE>、$<COMPILE_LANG_AND_ID> 和 $<PLATFORM_ID> 学会了支持从逗号分隔列表中匹配一个值。

  • 添加了生成器表达式 $<CUDA_COMPILER_ID:...> 和 $<CUDA_COMPILER_VERSION:...>。

  • 引入了生成器表达式 $<COMPILE_LANG_AND_ID:...>，允许根据每个源文件的 CMAKE_<LANG>_COMPILER_ID 和 LANGUAGE 为目标文件指定编译选项。

  • 添加了一个生成器表达式 $<FILTER:list,INCLUDE|EXCLUDE,regex>。

  • 添加了一个生成器表达式 $<REMOVE_DUPLICATES:list>。

  • 生成器表达式 $<SHELL_PATH:...> 增加了对路径列表的支持。

  • 添加了新的 $<TARGET_FILE*> 生成器表达式，用于检索各种构件文件名的前缀、基名和后缀：

    • $<TARGET_FILE_PREFIX:...>

    • $<TARGET_FILE_BASE_NAME:...>

    • $<TARGET_FILE_SUFFIX:...>

    • $<TARGET_LINKER_FILE_PREFIX:...>

    • $<TARGET_LINKER_FILE_BASE_NAME:...>

    • $<TARGET_LINKER_FILE_SUFFIX:...>

    • $<TARGET_PDB_FILE_BASE_NAME:...>

  • 生成器表达式 $<TARGET_OBJECTS:...> 现在在 SHARED、STATIC、MODULE 库和可执行文件上得到支持。

CTest¶

  • ctest_submit() 命令增加了一个新选项：BUILD_ID。它可用于将 CDash 分配给此构建的 ID 存储到变量中。

  • ctest_update() 命令学会了使用一个新变量：CTEST_UPDATE_VERSION_OVERRIDE。这可用于指定源代码树的当前版本，而不是使用更新命令来发现当前检出的版本。

CPack¶

  • CPack IFW 生成器 增加了一个新的 CPACK_IFW_PACKAGE_STYLE_SHEET 变量，用于自定义安装程序样式表。

已弃用和已移除的功能¶

  • cmake-server(7) 模式已被弃用，并将在未来版本的 CMake 中删除。请将客户端迁移为改用 cmake-file-api(7)。

  • ADDITIONAL_MAKE_CLEAN_FILES 目录属性现已弃用。请改为使用 ADDITIONAL_CLEAN_FILES 目录属性。

  • 变量 CMAKE_AUTOMOC_RELAXED_MODE 被视为已弃用。支持仍然存在，但在未来版本中将被删除。

  • export(PACKAGE) 命令现在除非通过 CMAKE_EXPORT_PACKAGE_REGISTRY 启用，否则不执行任何操作。请参阅策略 CMP0090。

  • Xcode 生成器现在至少需要 Xcode 5。

  • 为策略 CMP0066 添加了明确的弃用诊断（CMP0065 及以下版本已被弃用）。cmake-policies(7) 手册解释了所有策略的旧（OLD）行为都已被弃用，项目应迁移到新（NEW）行为。

其他更改¶

  • 如果 target_compile_features() 指定的特性在编译器的默认标准级别中可用，CMake 3.14 及以下版本错误地添加了不必要的 -std= 标志，这可能会降低标准级别。此错误已在 CMake 3.15 中修复。这种行为更改可能会暴露依赖于未记录实现细节的现有项目中的错误。指定编译特性仅确保编译器在具有这些特性的模式下运行，而不是使用特定的标准级别或传递显式的 -std= 标志。

  • CMake 学会了如何使用 IBM AIX XL 编译器和 SunPro 编译器编译 C++14，以及使用 AppleClang 编译器编译 C++20。

  • 对于类 MSVC 编译器，CMAKE_<LANG>_FLAGS 的值默认不再包含像 /W3 这样的警告标志。请参阅策略 CMP0092。

  • 定义 __ibmxl__ 的基于 IBM Clang 的 XL 编译器现在使用编译器 ID XLClang 而不是 XL。请参阅策略 CMP0089。

  • file(REMOVE) 和 file(REMOVE_RECURSE) 命令已更改为通过警告忽略空参数，而不是将其视为相对路径并删除当前目录的内容。

更新¶

自 CMake 3.15.0 以来的变更包括以下内容。

3.15.1¶

  • 在 CMake 3.15.0 中，针对 MSVC ABI 的 GNU 风格 Clang 编译器支持通过相应的 -std= 标志实现了 CMAKE_CXX_STANDARD 的 98 和 11 值。然而，这些模式无法与 MSVC 标准库一起使用。因此，CMake 3.15.1 即便对于 C++98 和 C++11 也传递 C++14 标准标志。这与 MSVC 本身一致，即 MSVC 始终在感知 C++14 的模式下运行。

  • 3.15.0 中添加的初步 Swift 支持已得到更新。

3.15.2¶

  • 在 CMake 3.15.0 和 3.15.1 中，CMAKE_FIND_PACKAGE_PREFER_CONFIG 变量导致 find_package() 命令即使在没有 REQUIRED 选项的情况下，也会因为找不到包而失败。此问题已修复。

3.15.3¶

  • CrayPrgEnv 编译器包装器支持已针对 Cray 编程环境 19.06 版本进行更新，对于该版本，XC Cray 系统上的默认链接模式现在是动态链接，而不是静态链接。

3.15.4¶

  • 在 CMake 3.15.0 到 3.15.3 中，EXCLUDE_FROM_ALL 目录属性从 3.14 之前的行为发生了退化，导致目录中的目标甚至从其自己的“all”中被排除。此问题已修复。该错误也存在于 3.14.0 到 3.14.6 中，已在 3.14.7 中修复。
