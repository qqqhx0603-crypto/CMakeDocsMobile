自 CMake 3.1 以来的变更包括以下内容。

新特性¶

语法¶

  • CMake 现在支持 Windows（微软桌面系统） 上 以 UTF-8 编码的 Unicode 字符。在此之前，已支持那些系统 API 接受 UTF-8 编码字符串的平台。Unicode 字符现在可用于 CMake 代码、源文件路径、配置文件（如 .h.in 文件）以及其他由 CMake 读取和写入的文件中。请注意，由于 CMake 与许多其他工具互操作，在使用某些 Unicode 字符时可能仍存在一些限制。

命令 (Commands)¶

  • add_custom_command() 和 add_custom_target() 命令引入了一个新的 BYPRODUCTS 选项，用于指定作为自定义命令副作用而产生的文件。这些文件不被视为“输出”（outputs），因为它们并不总是需要比输入文件更新。

  • add_custom_command() 和 add_custom_target() 命令引入了一个新的 USES_TERMINAL 选项，用于请求在可能的情况下使该命令能够直接访问终端。Ninja 生成器会将此类命令置于 console pool 中。由 CMake 提供、旨在进行个人交互式使用的构建目标（例如 install）现在也会被放置在此池中。

  • 新增了 continue() 命令，该命令可以在循环上下文内调用，以结束当前迭代并从循环块的顶部开始下一次迭代。

  • 创建了 file(LOCK) 子命令，允许 CMake 进程通过文件和目录锁进行同步。

  • file(STRINGS) 现在支持将 UTF-16LE、UTF-16BE、UTF-32LE、UTF-32BE 作为 ENCODING 选项。

  • install(EXPORT) 命令现在即使在导出集中的目标安装位置或 使用要求 是相对于安装前缀指定的情况下，也能与绝对路径的 DESTINATION 一起正常工作。CMAKE_INSTALL_PREFIX 变量的值作为相对引用的基准，被硬编码到已安装的导出文件中。

  • try_compile() 命令的源文件签名现在会在生成的测试项目中遵循链接标志（例如 CMAKE_EXE_LINKER_FLAGS）。请参阅策略 CMP0056。

  • try_run() 命令现在像 try_compile() 一样，支持 LINK_LIBRARIES 选项。

  • file(GENERATE) 命令现在生成的输出文件如果已设置权限，将与输入文件具有相同的权限。

  • file(GENERATE) 命令现在可以生成用作构建系统目标源文件的文件。生成的文件会自动将其 GENERATED 属性设置为 TRUE。

变量 (Variables)¶

  • 引入了 CMAKE_MATCH_COUNT 变量，用于记录在 if() 命令或 string() 命令中最近一次正则表达式匹配所进行的匹配次数。

属性 (Properties)¶

  • 引入了 ANDROID_API_MIN 目标属性，用于指定工具链所针对的最低版本。

  • 添加了 VS_SHADER_FLAGS 源文件属性，用于为 Visual Studio（集成开发环境） 生成器指定 .hlsl 文件的额外着色器标志。

模块 (Modules)¶

  • ExternalData 模块支持了 自定义获取脚本。这允许项目指定自定义的 .cmake 脚本，以便在构建期间获取数据对象。

  • ExternalProject 模块学习了创建独立外部项目步骤目标的选项，这些目标不依赖于内置步骤。

  • ExternalProject 模块的 ExternalProject_Add() 命令引入了一个新的 CMAKE_CACHE_DEFAULT_ARGS 选项，用于初始化外部项目中的缓存值，而不必在后续构建中设置它们。

  • ExternalProject 模块的 ExternalProject_Add() 命令引入了一个新的 TEST_EXCLUDE_FROM_MAIN 选项，用于从主构建中排除测试。

  • ExternalProject 模块的 ExternalProject_Add() 命令引入了一个新的 UPDATE_DISCONNECTED 选项，以避免自动从版本控制系统更新源树结账。

  • FindCUDA 模块增加了对 CUDA 7.0 中 cusolver 库的支持。

  • FindGit 模块学会了寻找安装在用户主目录中的 GitHub for Windows（微软桌面系统） 自带的 git 命令行工具。

  • 引入了 FindGSL 模块，用于查找 GNU 科学计算库（GNU Scientific Library）。

  • 引入了 FindIntl 模块，用于查找 Gettext 的 libintl 库。

  • FindLATEX 模块增加了对组件（components）的支持。

  • FindMPI 模块增加了对 Windows（微软桌面系统） 上 MS-MPI 的支持。

  • FindOpenSSL 模块现在分别在 OPENSSL_CRYPTO_LIBRARY 和 OPENSSL_SSL_LIBRARY 中独立报告 crypto 和 ssl 库，从而允许应用程序只链接其中之一。

  • WriteCompilerDetectionHeader 模块学习了为 cxx_thread_local 特性的可移植性创建一个定义。该定义会根据情况扩展为 C++11 的 thread_local 关键字，或预标准化之前的编译器特定等效项。

  • WriteCompilerDetectionHeader 模块学习了为每个编译器和每种语言创建多个输出文件，而不是创建一个大的文件。

CTest¶

  • ctest_coverage() 命令增加了对 Delphi 代码覆盖率的支持。

  • ctest_coverage() 命令增加了对 Javascript 代码覆盖率的支持。

  • 引入了 CTestCoverageCollectGCOV 模块，作为收集 gcov 结果以提交给 CDash 的 ctest_coverage() 命令的替代方案。

CPack¶

  • CPack RPM 生成器增加了设置各组件描述和摘要的选项。请参阅 CPACK_RPM_<component>_PACKAGE_DESCRIPTION 和 CPACK_RPM_<component>_PACKAGE_SUMMARY 变量。

  • CPack RPM 生成器增加了指定安装前和安装后脚本需求的选项。请参阅 CPACK_RPM_PACKAGE_REQUIRES_PRE 和 CPACK_RPM_PACKAGE_REQUIRES_POST 变量。

  • CPack RPM 生成器增加了指定卸载前和卸载后脚本需求的选项。请参阅 CPACK_RPM_PACKAGE_REQUIRES_PREUN 和 CPACK_RPM_PACKAGE_REQUIRES_POSTUN 变量。

  • CPack RPM 生成器引入了一个新的 CPACK_RPM_<COMPONENT>_PACKAGE_PREFIX 变量，用于指定组件特定的值，以代替 CPACK_PACKAGING_INSTALL_PREFIX。

  • CPack RPM 生成器引入了一个新的 CPACK_RPM_RELOCATION_PATHS 变量，用于为单个 RPM 包指定多个重定位前缀。

其他¶

  • cmake(1) 的 -E tar 命令现在支持使用 J 标志创建 .xz 压缩归档文件。

  • cmake(1) 的 -E tar 命令引入了一个新的 --files-from=<file> 选项，用于从文件中读取文件名，以克服命令行长度限制。

  • cmake(1) 的 -E tar 命令引入了一个新的 --mtime=<date> 选项，用于指定在 tarball 条目中记录的修改时间。

  • 编译特性功能现在能识别更多编译器支持的特性，包括：

    • Xcode 版本 4.4 到 6.1 的 Apple Clang (AppleClang)。

    • UNIX 和 Apple 上 GCC 版本 4.4 到 5.0 (GNU)。

    • 2010 到 2015 版本的 Microsoft Visual Studio（微软集成开发环境） (MSVC)。

    • Oracle SolarisStudio (SunPro) 12.4 版本。

  • AUTORCC 特性现在将 .qrc 文件中列出的文件作为依赖项进行跟踪。如果 rcc 工具的输入文件发生更改，该工具会自动重新运行。

新的诊断信息¶

  • break() 命令现在会拒绝在循环上下文之外的调用，或者向该命令传递参数。请参阅策略 CMP0055。

已弃用和移除的功能 (Deprecated and Removed Features)¶

  • 以 cmake-language(7) 编写的文件，例如 CMakeLists.txt 或 *.cmake 文件，现在被期望以 UTF-8 编码。如果文件已经是 ASCII 编码，它们将是兼容的。如果文件是其他编码（包括 Latin 1），则需要进行转换。

  • FindOpenGL 模块不再通过 FindX11 模块显式搜索对 X11 库的任何依赖。此类依赖不应该显式化。直接使用 X11 API 的应用程序应该显式查找并链接到 X11 库。

  • CMake 的实现现在依赖于某些旧编译器不支持的 C++ 编译器特性。因此，这些旧编译器无法再用于构建 CMake 本身。不过，CMake 仍然能够为这些旧编译器的用户生成 Makefile（Make 构建脚本） 和项目文件。已知无法再构建 CMake 的编译器包括：

    • Visual Studio 6（微软集成开发环境）（微软集成开发环境） 和 7.0 —— 已被 VisualStudio 7.1 及更高版本取代。

    • GCC 2.95 —— 已被 GCC 3 及更高版本的编译器取代。

    • Borland 编译器 —— 已被其他 Windows（微软桌面系统） 编译器取代。

    • Compaq 编译器 —— 已被其他编译器取代。

    • SGI 编译器 —— IRIX 不再作为主机平台支持。

其他更改 (Other Changes)¶

  • 在 Windows（微软桌面系统） 和 OS X 上，支持通过 https 进行网络通信的命令（例如 file(DOWNLOAD)、file(UPLOAD) 和 ctest_submit()）现在即使在未针对 OpenSSL 构建 CMake 的情况下也支持 SSL/TLS。默认使用 Windows（微软桌面系统） 或 OS X 原生的 SSL/TLS 实现。操作系统配置的证书颁发机构将被自动信任。

    在其他平台上，当 CMake 使用 OpenSSL 构建时，这些命令现在会在一些 /etc 路径中搜索操作系统配置的证书颁发机构以自动信任。

  • 在 OS X 上使用 Makefile（Make 构建脚本） 和 Ninja（构建工具） 生成器时，如果在 /usr/bin 中找到编译器，它现在会映射到 Xcode 应用程序文件夹内的相应编译器（如果存在）。这使得这些构建树在 xcode-select 切换到不同的 Xcode 安装时，仍能继续使用其原始编译器。

  • Visual Studio（集成开发环境） 生成器现在以 UTF-8 而不是 Windows（微软桌面系统）-1252 写入解决方案和项目文件。Windows（微软桌面系统）-1252 支持北美、南美和西欧等地的 Latin 1 语言。使用 UTF-8，现在支持更多的语言。

  • Xcode 生成器不再要求预先确定 CMAKE_MAKE_PROGRAM 变量的值。它现在会在构建时根据需要定位 xcodebuild。

  • 在使用 SolarisStudio 12 构建 CMake 本身时，默认的 libCStd 标准库不足以构建 CMake。SolarisStudio 发行版支持使用 STLPort4 或 libstdc++ 的编译器选项。现在，在使用 SolarisStudio 编译器构建 CMake 时，会自动添加用于选择标准库的适当选项。
