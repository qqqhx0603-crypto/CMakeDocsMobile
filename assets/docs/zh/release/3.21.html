自 CMake 3.20 以来的主要变更如下。

新特性¶

预设 (Presets)¶

  • cmake-presets(7) 增加了在配置预设中指定安装前缀 (install prefix) 的支持。

  • cmake-presets(7) 增加了对预设进行条件启用的支持。

  • cmake-presets(7) 增加了对 ${hostSystemName} 宏的支持。

  • cmake-presets(7) 增加了对省略 generator 和 binaryDir 字段的支持。

生成器¶

  • 添加了 Visual Studio 17 2022 生成器。

  • Makefile（Make 构建脚本） 生成器 和 Ninja 生成器现在支持为 C、CXX、OBJC 和 OBJCXX 语言添加链接器启动器工具 (linker launcher tools)。详情请参阅 CMAKE_<LANG>_LINKER_LAUNCHER 变量和 <LANG>_LINKER_LAUNCHER 目标属性。

语言¶

  • CMake 现在支持将 HIP 作为一等语言，可通过 project() 和 enable_language() 命令启用。

  • C_STANDARD、OBJC_STANDARD 以及 编译特性 (Compile Features) 功能增加了对 C17 和 C23 的支持。

  • 源文件扩展名 .ixx 和 .cppm 现在被视为 C++ 文件。

命令行¶

  • cmake(1) 增加了 --install-prefix 命令行选项，用于指定安装前缀的位置。

  • cmake(1) 增加了 --toolchain 命令行选项，用于指定工具链文件。

  • cmake(1) 的 -E capabilities 输出，对于某些生成器，现在可能包含一个 supportedPlatforms 字段，列出了在 CMAKE_GENERATOR_PLATFORM 中已知的支持平台。

  • 打印到终端的消息现在可以根据消息类型进行着色。

编译器¶

  • 现在支持富士通 (Fujitsu) 编译器，使用编译器 ID Fujitsu（传统 Trad 模式）和编译器 ID FujitsuClang（Clang 模式）。

平台¶

  • CMake 现在支持 MSYS 运行时环境，类似于 CYGWIN。

基于文件的 API¶

  • cmake-file-api(7) 的 "codemodel" 2.3 版本已更新。

  • cmake-file-api(7) 的 "codemodel" 版本 2 增加了一个新的 "directory" 对象，包含目录级别的信息。这包括由 install() 命令生成的安装程序列表。

命令¶

  • add_custom_command() 命令的 DEPFILE 选项

    • 现在可以使用 生成器表达式，

    • 现在被 VS 2012 及以上版本的 Visual Studio（集成开发环境） 生成器 支持，并且

    • 现在被 Xcode 生成器支持。

  • add_custom_command(TARGET) 命令增加了对解析目标依赖生成器表达式的支持。

  • build_command() 命令增加了一个 PARALLEL_LEVEL 选项。

  • 添加了 file(COPY_FILE) 命令，用于复制单个文件。

  • file(GET_RUNTIME_DEPENDENCIES) 命令增加了新的 POST_INCLUDE_FILES 和 POST_EXCLUDE_FILES 参数。

  • file(REAL_PATH) 命令增加了 EXPAND_TILDE 选项，用于将开头的波浪号替换为用户主目录的路径。

  • file(RENAME) 命令现在可以选择将失败信息捕获到结果变量中。它还增加了 NO_REPLACE 选项，如果目标已存在则失败。

  • install() 命令增加了一个新的 IMPORTED_RUNTIME_ARTIFACTS 模式，可用于安装导入目标的运行时构件。

  • install() 命令增加了一个新的 RUNTIME_DEPENDENCY_SET 模式，可使用 file(GET_RUNTIME_DEPENDENCIES) 安装运行时依赖。

  • install(TARGETS) 命令增加了新的 RUNTIME_DEPENDENCIES 和 RUNTIME_DEPENDENCY_SET 参数，可使用 file(GET_RUNTIME_DEPENDENCIES) 安装运行时依赖。

  • install(SCRIPT|CODE) 命令支持一个新的 ALL_COMPONENTS 选项，允许相应代码在每个组件的安装过程中为每个组件运行。

  • project() 命令现在设置变量 PROJECT_IS_TOP_LEVEL 和 <PROJECT-NAME>_IS_TOP_LEVEL，以指示它是否在顶层 CMakeLists.txt 文件中被调用。

变量¶

  • CMAKE_TOOLCHAIN_FILE 环境变量已添加，用于为 CMAKE_TOOLCHAIN_FILE 变量提供默认值。

属性¶

  • IMPORTED_TARGETS 目录属性已添加，用于获取在当前目录中创建的 导入目标 (Imported Targets) 列表。

  • XCODE_EMBED_APP_EXTENSIONS 目标属性已添加，用于指示 Xcode 生成器嵌入应用扩展（如 iMessage 贴纸包）。嵌入的各个方面可以使用 XCODE_EMBED_APP_EXTENSIONS_PATH、XCODE_EMBED_APP_EXTENSIONS_CODE_SIGN_ON_COPY 和 XCODE_EMBED_APP_EXTENSIONS_REMOVE_HEADERS_ON_COPY 属性进行自定义。

模块¶

  • FindBLAS 和 FindLAPACK 模块增加了对串行 Fujitsu_SSL2 和并行 Fujitsu_SSL2BLAMP 库的支持。

  • FindDevIL 模块现在提供导入目标。

  • FindIconv 模块现在具有版本支持。

  • FindIntl 模块现在具有版本支持。

  • FindMPI 模块增加了对主机和交叉编译模式下 Fujitsu 和 FujitsuClang 的支持。

  • FindMsys 模块已添加，用于查找 MSYS 安装。像 FindCygwin 一样，它被其他一些查找模块自动用于定位 Windows（微软桌面系统） 上的 UNIX 风格工具。

  • FindOpenMP 模块增加了对 Fujitsu 和 FujitsuClang 的支持。

  • FindVulkan 模块增加了导入目标 Vulkan::Headers 和 Vulkan::glslangValidator。

  • UseJava 模块命令 add_jar 增加了 RESOURCES 选项，允许显式命名具有非可选命名空间的资源。

  • UseSWIG 模块现在使用标准的库命名约定来处理 CSharp 语言。请参阅策略 CMP0122。

  • UseSWIG 模块现在支持使用 swig 工具为 Xcode 生成器生成隐式依赖。

生成器表达式¶

  • 添加了新的 TARGET_RUNTIME_DLLS 生成器表达式。

CTest¶

  • ctest(1) 增加了关于捕获 额外测试测量数据 的文档。

  • ctest(1) 现在能够识别在运行时附加到测试的文件。之前只能通过 ATTACHED_FILES 或 ATTACHED_FILES_ON_FAIL 测试属性在配置时附加文件。详情请参阅 额外测试测量数据。

  • ctest(1) 增加了 --output-junit 选项，用于将测试结果写入 JUnit XML 文件。

  • ctest_build() 命令增加了一个 PARALLEL_LEVEL 选项。

CPack¶

  • CPack DragNDrop 生成器 增加了 CPACK_DMG_FILESYSTEM 选项，用于控制 .dmg 文件系统。

  • CPack IFW 生成器 现在支持在传递给 cpack_ifw_configure_component() 或 cpack_ifw_configure_component_group() 作为 DEPENDS 或 DEPENDENCIES 参数的名称中使用连字符。这需要 QtIFW 3.1 或更高版本。

  • CPack NSIS 生成器 增加了新的 CPACK_NSIS_EXECUTABLE 变量，用于指定要使用的 makensis 可执行文件，而不是默认的一个。

  • CPACK_CUSTOM_INSTALL_VARIABLES 变量已添加，用于在 CPack 执行的 cmake_install.cmake 脚本调用中设置变量。

弃用和已移除的特性¶

  • 无文档记录的 CMAKE_SYSTEM_NAME 版本剥离行为已被彻底移除。如果它由 -D 标志或 工具链文件 设置，它将保持不变，即使它仍然包含版本号。类似的、同样未记录的 CMAKE_HOST_SYSTEM_NAME 版本剥离行为已被提前，在调用 project() 或 enable_language() 之前执行。

  • ARMClang cpu/arch 编译和链接标志不再根据 CMAKE_SYSTEM_PROCESSOR 变量或未记录的 CMAKE_SYSTEM_ARCH 变量自动添加。它们必须显式指定。请参阅策略 CMP0123。

其他更改¶

  • find_file()、find_path()、find_program() 和 find_library() 命令以相同的方式处理缓存变量，无论它们是如何定义的。详情请参阅策略 CMP0125。

  • find_file()、find_path()、find_program() 和 find_library() 命令增加了 NO_CACHE 选项，用于将查找结果存储在普通变量中。

  • foreach() 命令现在将循环变量隔离在循环作用域中。详情请参阅策略 CMP0124。

  • list() 命令的 GET、INSERT、SUBLIST 和 REMOVE_AT 子命令现在在为其任何索引参数提供无效（即非整数）值时会报错，具体取决于策略 CMP0121 的设置。

  • set(CACHE) 命令不再删除同名的普通变量（如果存在）。请参阅策略 CMP0126。

  • target_link_libraries() 调用通过 TARGET_OBJECTS 生成器表达式引用对象库时，现在将目标文件置于链接行上的所有库之前，而不管其指定的顺序如何。详情请参阅关于 通过 $<TARGET_OBJECTS> 链接对象库 的文档。

  • Ninja（构建工具） 生成器 现在使用绝对路径将源文件和包含目录传递给编译器。这使得诊断消息和调试符号更加一致，并与 Makefile（Make 构建脚本） 生成器 匹配。

  • NMake Makefiles 生成器现在在使用 VS 9 或更高版本的 nmake 时，将生成的 makefile 编码为带 BOM 的 UTF-8。

  • VS 2010 及以上版本的 Visual Studio（集成开发环境） 生成器 现在将每个源文件的预处理器定义放置在目标范围的预处理器定义之后。这使得 VS 与 Ninja（构建工具） 生成器 和 Makefile（Make 构建脚本） 生成器 保持一致。

  • cmake.org 上提供的预编译二进制文件现在支持 liblzma 多线程。请参阅 CPACK_THREADS 和 CPACK_ARCHIVE_THREADS 变量。

更新¶

自 CMake 3.21.0 以来的更改包括以下内容。

3.21.1¶

  • Visual Studio 17 2022 生成器现在基于 "Visual Studio 2022（微软集成开发环境）（微软集成开发环境） Preview 2"。之前它是基于 "Preview 1.1" 的。

3.21.2¶

  • 启用了 CUDA_SEPARABLE_COMPILATION 的 CUDA 目标现在可以在非根目录中正确生成。

  • Visual Studio 17 2022 生成器现在基于 "Visual Studio 2022（微软集成开发环境）（微软集成开发环境） Preview 3.1"。之前它是基于 "Preview 2" 的。

3.21.3¶

  • Visual Studio 17 2022 生成器现在基于 "Visual Studio 2022（微软集成开发环境）（微软集成开发环境） Preview 4"。之前它是基于 "Preview 3.1" 的。

  • AMD ROCm 平台 hipcc 编译器在 CMake 3.21.0 到 3.21.2 中被识别为具有 ID ROCMClang 的独立编译器。这已被移除，因为它导致了回归问题。相反

    • hipcc 可能不再用作 HIP 编译器，因为它干扰了 CMake 需要传递给 Clang 的标志。请直接使用 Clang。

    • hipcc 可以再次用作 CXX 编译器，并被视为它在底层选择的任何编译器，就像 CMake 3.20 及以下版本一样。

3.21.4¶

  • Visual Studio 17 2022 生成器现在基于 "Visual Studio 2022（微软集成开发环境）（微软集成开发环境）" 发布候选版。之前它是基于预览版本的。

3.21.5, 3.21.6, 3.21.7¶

这些版本未对已记录的功能或接口进行任何更改。为了支持生态系统更改和/或修复回归，进行了一些实现更新。
