自 CMake 3.22 以来的主要变更如下。

新功能¶

预设¶

  • cmake-presets(7) 文件现在支持架构版本 4。

  • cmake-presets(7) 文件现在具有一个可选的 include 字段，允许这些文件包含其他文件。

  • cmake-presets(7) 文件现在支持 ${fileDir} 宏，其中包含预设文件所在的目录。

  • cmake-presets(7) 增加了对在构建预设中指定 resolvePackageReferences 命令行选项的支持，以控制从外部包管理器恢复包引用的行为。目前仅 Visual Studio（集成开发环境） 生成器支持此选项，以支持从 NuGet 恢复包。其他生成器将忽略此选项。

生成器¶

  • 适用于 VS 2019 及更高版本的 Visual Studio（集成开发环境） 生成器 现在支持 C# 项目的 .NET SDK 风格项目文件 (.csproj)。请参阅 DOTNET_SDK 目标属性和相应的 CMAKE_DOTNET_SDK 变量。add_custom_command() 尚不支持 .NET SDK 风格的项目。

  • 适用于 VS 2017 及更高版本的 Visual Studio（集成开发环境） 生成器 现在可以支持使用未被 VS 安装程序识别的 Visual Studio（集成开发环境） 便携式实例。请参阅 CMAKE_GENERATOR_INSTANCE 变量。

命令行¶

  • 当 cmake(1) 的 --build 命令与设置了 VS_PACKAGE_REFERENCES 目标属性的项目上的 Visual Studio（集成开发环境） 生成器 一起使用时，现在会自动从 NuGet 恢复包引用。可以设置缓存变量 CMAKE_VS_NUGET_PACKAGE_RESTORE 来在构建树中切换此行为。使用 --resolve-package-references 命令行选项可在单次调用中控制此行为。

  • cmake(1) 命令行工具增加了 --debug-find-pkg 选项，以在特定的 find_package() 调用下启用调试消息。

  • cmake(1) 命令行工具增加了 --debug-find-var 选项，以对使用特定结果变量的 find_* 调用启用调试消息。

编译器¶

  • 基于 LLVM 的 IBM Open XL C/C++ 编译器现在得到支持，编译器 ID 为 IBMClang。

  • MCST LCC 编译器现在得到支持，编译器 ID 为 LCC。请参阅策略 CMP0129。

基于文件的 API¶

  • cmake-file-api(7) “codemodel” 版本 2 的 version 字段已更新为 2.4。

  • cmake-file-api(7) “codemodel” 版本 2 的 directory 对象 installers 字段增加了一个新的 fileSet 安装程序类型，以及相关的 fileSetName、fileSetType、fileSetDirectories 和 fileSetTarget 字段。

命令¶

  • define_property() 命令增加了一个新的 INITIALIZE_FROM_VARIABLE 选项，用于在创建目标时从变量初始化目标属性。

  • install(TARGETS) 命令增加了一个新的 FILE_SET 参数，可用于安装与目标关联的头文件集。

  • string(TIMESTAMP) 和 file(TIMESTAMP) 命令现在支持用于微秒的 %f 说明符。

  • target_sources() 命令增加了一个新的 FILE_SET 模式，可用于将头文件添加为目标的仅头文件源文件。

变量¶

  • CMAKE_CUDA_ARCHITECTURES 变量和关联的 CUDA_ARCHITECTURES 目标属性现在支持 CUDA 工具包 7.0+ 的 all 和 all-major 值。

  • 增加了 CMAKE_IGNORE_PREFIX_PATH 和 CMAKE_SYSTEM_IGNORE_PREFIX_PATH 变量，用于指示 find_package()、find_program()、find_library()、find_path() 和 find_file() 命令忽略指定的路径前缀。

  • 增加了 CMAKE_LINK_LIBRARIES_ONLY_TARGETS 变量和相应的 LINK_LIBRARIES_ONLY_TARGETS 目标属性，用于可选地要求所有可能是目标名称的链接项必须是现有目标的名称。

属性¶

  • 增加了 HEADER_SETS 和 INTERFACE_HEADER_SETS 只读目标属性，用于列出与目标关联的头文件集。

  • 增加了 HEADER_SET 和 HEADER_SET_<NAME> 目标属性，分别用于列出默认头文件集和命名头文件集中的文件。

  • 增加了 HEADER_DIRS 和 HEADER_DIRS_<NAME> 目标属性，分别用于指定默认头文件集和命名头文件集的基目录。

  • 增加了 IMPORTED_NO_SYSTEM 目标属性，用于指定 导入目标 (Imported Target) 不应被视为系统库（即其包含目录不会被自动设为 SYSTEM）。

  • 增加了 XCODE_EMBED_PLUGINS 目标属性，用于告诉 Xcode 生成器将哪些目标放入 Embed PlugIns 构建阶段。

  • 增加了 XCODE_SCHEME_ENABLE_GPU_FRAME_CAPTURE_MODE 目标属性和对应的 CMAKE_XCODE_SCHEME_ENABLE_GPU_FRAME_CAPTURE_MODE 变量，用于告诉 Xcode 生成器以允许指定 Xcode 方案选项属性 GPU Frame Capture。

模块¶

  • CheckPIESupported 模块现在支持 OBJC、OBJCXX、CUDA 和 HIP 语言。它现在还遵循 CMAKE_SYSROOT 和 CMAKE_OSX_SYSROOT 变量。

  • ExternalProject 模块的 ExternalProject_Add() 命令增加了对 USES_TERMINAL_PATCH 选项的支持，以便为补丁步骤提供独占的终端访问权限。

  • FindCUDAToolkit 模块现在提供了一个用于 libcufft_static_nocallback 的目标（如果找到）。

  • FindGLUT 模块现在提供 GLUT_INCLUDE_DIRS 结果变量，以符合 cmake-developer(7) 手册中记载的命名约定。这取代了旧版的 GLUT_INCLUDE_DIR 变量。

  • FindGTest 模块现在提供了一个用于 GMock 的目标（如果找到）。

  • FindVulkan 模块现在提供一个 Vulkan_VERSION 结果变量，用于报告版本号。

CTest¶

  • ctest(1) 增加了一个新的 CTEST_SUBMIT_INACTIVITY_TIMEOUT 变量，可用于指定提交不活跃的超时时间。

CPack¶

  • CPack productbuild 生成器 增加了新的 CPACK_PRODUCTBUILD_DOMAINS、CPACK_PRODUCTBUILD_DOMAINS_ANYWHERE、CPACK_PRODUCTBUILD_DOMAINS_USER 和 CPACK_PRODUCTBUILD_DOMAINS_ROOT 变量，用于将 domains 元素添加到 Distribution XML 中。使用这些变量，现在可以安装产品到用户主目录，而无需管理员权限。

  • CPack productbuild 生成器 增加了一个新变量 CPACK_PRODUCTBUILD_IDENTIFIER，用于自定义与产品关联的唯一产品标识符。

  • CPack productbuild 生成器 和 CPack PackageMaker 生成器使用的 CPack.distribution.dist.in 模板已更新，使用一个新的 CPACK_APPLE_PKG_INSTALLER_CONTENT 变量作为其主要内容。这取代了此前未记录且现已弃用的 CPACK_PACKAGEMAKER_CHOICES 变量。

  • CPack IFW 生成器 增加了新的 CPACK_IFW_ARCHIVE_FORMAT 和 CPACK_IFW_ARCHIVE_COMPRESSION 变量，用于设置打包新组件数据存档时使用的格式以及选择使用的压缩级别。这些功能适用于 QtIFW 4.2 及更高版本。

  • CPack IFW 生成器 增加了新的 CPACK_IFW_PACKAGE_DISABLE_COMMAND_LINE_INTERFACE 变量，以防止用户向安装程序传递任何消费者命令，例如 install、update 和 remove。此功能适用于 QtIFW 4.0 及更高版本。

  • CPack IFW 生成器 增加了新的 CPACK_IFW_PACKAGE_PRODUCT_IMAGES 变量，用于添加要在 PerformInstallationPage 上显示的图像列表。此功能适用于 QtIFW 4.0 及更高版本。

  • CPack IFW 生成器 增加了新的 CPACK_IFW_PACKAGE_RUN_PROGRAM、CPACK_IFW_PACKAGE_RUN_PROGRAM_ARGUMENTS 和 CPACK_IFW_PACKAGE_RUN_PROGRAM_DESCRIPTION 变量，用于在安装程序完成后（如果用户接受该操作）执行命令。此功能适用于 QtIFW 4.0 及更高版本。

  • CPack IFW 生成器 增加了新的 CPACK_IFW_PACKAGE_SIGNING_IDENTITY 变量，用于指定用于签署生成的应用程序包的代码签名标识。此功能仅在 macOS（苹果桌面系统） 上可用，且适用于 QtIFW 3.0 及更高版本。

  • CPack WIX 生成器 增加了一个新变量 CPACK_WIX_SKIP_WIX_UI_EXTENSION，以跳过包含 WixUIExtensions。

已弃用和移除的功能¶

  • cmake(1) 现在会在指定多个源路径时发出警告，例如 cmake -S src1 src2。这从未被正式记录或支持，但旧版本意外地接受多个源路径并使用最后指定的路径。请更新脚本以避免传递多个源路径参数。

  • cpack(1) 中未记录的 OSXX11 生成器已被移除。

  • 此前在 CPack.distribution.dist.in 模板中使用的未记录 CPACK_PACKAGEMAKER_CHOICES 变量已被新的 CPACK_APPLE_PKG_INSTALLER_CONTENT 变量替换。这仅影响那些提供了自己的自定义 CPack.distribution.dist.in 模板文件，但仍然依赖于 CPACK_PACKAGEMAKER_CHOICES 被设置的项目。这些自定义模板文件应更新为改用 CPACK_APPLE_PKG_INSTALLER_CONTENT，或者完整定义模板文件的所有内容而不依赖于任一变量的替换。

其他更改¶

  • CPack DragNDrop 生成器 不再将 CPACK_RESOURCE_FILE_LICENSE 作为生成的 .dmg 中的许可协议附件，除非通过 CPACK_DMG_SLA_USE_RESOURCE_FILE_LICENSE 选项明确激活。在 CMake 项目中，CPack 模块默认启用此选项以保持兼容性。

  • CUDA 目标现在可以同时启用 CUDA_SEPARABLE_COMPILATION 和 CUDA_PTX_COMPILATION。

  • CUDA 编译器检测现在：

    • 在所有无法计算默认架构（如果需要）的情况下发出错误（请参阅 CMP0104），

    • 在 Clang 上处理 CMAKE_CUDA_ARCHITECTURES 的 OFF 设置，

    • 支持存在多个默认架构的理论情况，以及

    • 尝试检测无效架构并发出错误。

  • 带有 Clang 的 CUDA 现在实现了策略 CMP0105 以及 $<DEVICE_LINK:...> 和 $<HOST_LINK:...> 生成器表达式。

  • define_property() 命令的 BRIEF_DOCS 和 FULL_DOCS 参数现在是可选的。

  • ccmake(1) 现在可以在从源码构建 CMake 时在 Windows（微软桌面系统） 上启用。这是实验性的，因此不包含在官方发行版中。

更新¶

自 CMake 3.23.0 以来的变更如下。

3.23.1¶

  • CMake 3.23.0 中增加的 target_sources() FILE_SET 功能尚不能在 Apple FRAMEWORK 目标中正确放置头文件。在等待未来版本进一步开发的同时，目前在 Apple 平台上向此类目标添加类型为 HEADERS 的 FILE_SET 会报错。

  • CMake 3.23.0 中增加的 HEADER_SETS 和 INTERFACE_HEADER_SETS 目标属性现在是 target_sources() 命令所创建头文件集的只读记录。

3.23.2¶

  • CPack.distribution.dist.in 模板文件中使用的 CPACK_PACKAGEMAKER_CHOICES 变量在 CMake 3.23.0 中被新的 CPACK_APPLE_PKG_INSTALLER_CONTENT 变量替换。这破坏了那些提供了自己的模板文件但仍期望定义 CPACK_PACKAGEMAKER_CHOICES 变量的项目。现在旧的 CPACK_PACKAGEMAKER_CHOICES 变量也被设置为与以前相同的内容，但已被正式弃用。

3.23.3, 3.23.4, 3.23.5¶

  • 这些版本未对已记录的功能或接口进行任何更改。为了支持生态系统更改和/或修复回归，进行了一些实现更新。
